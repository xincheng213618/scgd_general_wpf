# 设备服务概览

本页作为设备章节入口，优先回答“有哪些设备页可看、通常怎么配置、遇到问题先查哪里”。

## 设备服务是什么

在 ColorVision 里，设备通常以“服务”的形式被管理。主程序会维护一个设备服务列表，用户在设备窗口中查看、配置、启用和操作这些服务。

设备相关实现主要位于：

- `Engine/ColorVision.Engine/Services/`
- `Engine/ColorVision.Engine/Services/Devices/`

当前代码目录中可以看到的典型设备分类包括：

- Camera
- Calibration
- Motor
- FileServer
- FlowDevice
- Sensor
- SMU
- Spectrum

## 常见使用顺序

1. 先看 [添加与配置设备](./configuration.md)，了解新增和保存设备的基本流程。
2. 再进入具体设备页，确认该设备有哪些参数和操作。
3. 如果涉及相机，继续看 [相机管理](./camera-management.md) 和 [相机参数配置](./camera-configuration.md)。
4. 如果需要让设备参与自动化流程，再看 [工作流程概览](../workflow/README.md)。

## 使用时通常会遇到什么

- 一个设备服务可能绑定真实硬件，也可能只是某类通信或文件型服务。
- 设备列表顺序、启用状态和配置内容通常会影响后续窗口和流程里的可见性。
- 某些设备除了基础配置，还会有独立的物理设备管理、标定或参数配置页。

## 设备交接要记录什么

现场交付设备时，不要只写“设备已配置”。后续接手人员至少需要知道这台设备怎么识别、怎么连通、怎么证明它能被流程使用。

| 记录项 | 要写清楚什么 | 用途 |
| --- | --- | --- |
| 设备身份 | 名称、Code、设备类型、对应真实硬件或服务 | 防止流程绑定到错误设备 |
| 通信配置 | IP、端口、串口、波特率、MQTT 主题、文件路径等 | 复现现场连接条件 |
| 最小动作 | 相机拍图、电机回零、SMU 点测、文件服务列目录等 | 证明设备不是只出现在列表里 |
| 流程绑定 | 被哪条流程、哪个节点或哪个项目窗口引用 | 排查“手动能用、流程失败” |
| 日志证据 | 成功连接时间点、失败错误、超时或权限信息 | 给远程排查留下定位线索 |
| 回退方式 | 上一版配置、驱动、固件或项目包位置 | 升级后异常时能退回 |

## 现场最小验收

| 步骤 | 操作 | 通过标准 |
| --- | --- | --- |
| 1 | 打开设备列表并刷新 | 目标设备可见，名称和 Code 与交接记录一致 |
| 2 | 打开设备详情或专用窗口 | 关键参数可见，保存状态明确 |
| 3 | 执行一个不会破坏现场状态的最小动作 | 有返回、有日志、状态变化可解释 |
| 4 | 打开依赖它的流程或项目窗口 | 能选到同一台设备，而不是相似名称的旧设备 |
| 5 | 跑一条最小流程或模拟流程 | 设备节点能执行到，并能定位结果或失败原因 |

如果第 3 步失败，先查设备层；如果第 3 步通过但第 5 步失败，优先查流程绑定、模板参数和项目映射。

## 设备问题分层定位

| 现象 | 先查 | 再查 |
| --- | --- | --- |
| 列表里没有设备 | 是否创建并保存、类型是否正确 | 插件/项目包是否提供该设备入口 |
| 设备有但打不开 | 硬件在线、驱动、端口/IP、权限 | 日志里的连接异常和超时 |
| 手动动作成功但流程失败 | 流程节点引用的设备 Code | 节点参数、模板版本、项目窗口选择 |
| 结果有但图像或数据不对 | 设备参数和采集条件 | 后续模板、导出字段或数据库记录 |
| 升级后才异常 | 配置是否被覆盖 | 旧包、旧配置和新 DLL 是否混用 |

## 排查建议

### 设备没有出现在列表里

- 先确认是否已经在设备配置窗口中创建并保存。
- 再确认对应设备依赖是否已经满足，例如物理硬件、驱动或通信环境。

### 设备出现了，但无法工作

- 优先检查该设备专题页里的参数说明。
- 再检查日志和连接状态。
- 若是流程里调用失败，再联动查看 [流程执行与调试](../workflow/execution.md)。
