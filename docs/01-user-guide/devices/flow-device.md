# 流程设备服务

流程设备服务适合放在“流程会调用它，但你平时不会频繁手动操作它”的位置来理解。当前代码里它更像一个给流程系统使用的设备入口，重点在于配置、连接和被流程正确引用，而不是复杂的独立操作面板。

## 什么时候应该看这一页

- 你要在设备列表里新增一个流程设备服务
- 你要让某个流程节点引用这类设备
- 你在排查“设备已经建了，但流程里调不到”这类问题
- 你想确认这类服务是否只是配置入口，而不是独立业务窗口

## 先怎么理解它

在当前实现里，流程设备服务主要提供两件事：

- 作为一个标准设备服务实例被系统管理
- 作为流程执行时可以引用的设备对象被 MQTT 服务层识别

也就是说，先保证它被正确创建、保存并能被流程引用，比研究更多内部消息细节更重要。

## 常见使用顺序

1. 先在设备列表里创建并保存流程设备服务。
2. 确认设备代码、主题或相关连接参数已经配置正确。
3. 回到 [工作流程概览](../workflow/README.md) 或具体流程窗口里引用这台设备。
4. 执行流程时观察是否真的命中了这台设备，而不是只确认它出现在列表里。

## 使用时要特别注意什么

### 设备存在，不代表流程已经在用它

设备能显示在列表里，只说明它作为资源被创建了；真正要看的是流程节点里是否引用了正确的设备实例。

### 先查配置，再查流程节点

当流程里调不到设备时，优先先确认：

- 当前设备代码是否正确
- 流程里引用的是不是这台设备
- 相关通信配置是否匹配

### 不要假设它有很多独立手动动作

从当前实现来看，这类设备页更偏配置和接入层。排查时不要先去找复杂的独立控制按钮，先把设备配置和流程引用关系查清。

## 常见问题

### 设备已经创建，但流程里找不到

- 先确认设备是否已经保存成功
- 再确认流程节点引用的是不是正确的设备对象
- 如果流程里按代码或名称筛选，检查是否存在不匹配

### 流程执行时设备没有响应

- 先确认设备服务本身的连接配置
- 再确认流程执行到的是否真的是对应节点
- 同时结合 [日志查看器](../interface/log-viewer.md) 看是不是消息层没有命中这台设备

### 不确定问题在设备层还是流程层

- 先把问题拆成两段：设备是否存在、流程是否引用
- 如果设备配置本身就不完整，先修设备
- 如果设备没问题，再回到 [流程执行与调试](../workflow/execution.md) 缩小范围
