# 流程执行与调试

本页只保留当前代码里能证实的执行入口和排查路径。对于大多数用户，流程执行最关键的不是“高级调试术语”，而是先确认选对了流程、服务是否在线、起始节点是否存在，以及失败时怎么缩小范围。

## 当前可直接使用的执行入口

从当前实现看，流程执行至少有这两类明确入口：

- 开始执行：`F6`
- 停止执行：`F7`

如果你看到旧文档里写的是 `F5`、`F10`、断点或单步执行，请以当前界面和代码绑定为准，不要继续按旧快捷键排查。

## 执行前先确认什么

### 已选中有效流程模板

如果当前没有选中有效模板，执行不会真正开始。先确认下拉框里选中的就是你要跑的那条流程，而不是只打开了流程窗口。

### 流程里存在起始节点

当前执行前会检查起始节点。没有起始节点时，流程会直接失败，不会进入真正的执行阶段。

### 注册中心和服务令牌可用

当前实现会先检查注册中心连接和可用服务列表。服务令牌为空时，界面会先提示刷新服务后再重试。

### 预处理通过

流程正式开始前还有一层预处理。预处理失败时，流程会被取消，不会继续往下跑。

## 常见执行顺序

1. 先在 [流程设计](./design.md) 中确认当前流程内容和起始节点。
2. 在执行窗口选中正确的流程模板。
3. 确认相关设备服务已经在线。
4. 按 `F6` 或点击运行入口开始执行。
5. 执行过程中观察日志区、当前运行节点和进度变化。
6. 需要中止时按 `F7` 或使用停止入口。

## 执行时你能看到什么

### 当前运行节点

执行过程中，界面会持续显示当前正在运行的节点名称。如果一条流程看起来“卡住了”，先看停在哪个节点，而不是立刻怀疑整条流程。

### 执行进度和耗时

当前实现会记录流程耗时、上次执行时间，并根据上次耗时估算当前进度。这个估算值适合快速判断大致阶段，但不应该当成严格的业务完成度。

### 结果和状态

流程完成后，会写入这次批次的状态、耗时和结果摘要；停止执行时，会把状态记为取消。

## 调试时应该怎么缩小范围

### 先找第一个失败节点

比起“整条流程失败了”这种判断，更有价值的是先找出第一个变红或没有继续推进的节点，再回到对应设备、模板或输入数据去查。

### 先分清是没启动，还是中途失败

常见两类问题：

- 执行前就被拦住：模板未选中、起始节点缺失、注册中心未连接、服务令牌为空、预处理失败
- 已经开始执行，但中途失败：某个节点返回错误、超时或消息不匹配

### 日志区优先于猜测

当前执行窗口会持续更新日志文本。看到失败时，先看最后停在哪个节点、前后是否出现了预处理失败、取消执行或状态消息，再决定回哪一层排查。

## 流程运行交接要记录什么

流程交接不是只交一份流程文件。接手人员需要知道它依赖哪些设备、模板、输入、结果和外部系统。

| 记录项 | 要写清楚什么 | 用途 |
| --- | --- | --- |
| 流程模板 | 模板名称、版本、导入来源、最后修改人 | 防止跑错旧流程 |
| 起始条件 | 起始节点、SN/批次输入、项目窗口或外部触发方式 | 判断流程为什么没有开始 |
| 设备依赖 | 相机、电机、SMU、文件服务等节点绑定 | 排查设备层失败 |
| 模板依赖 | 图像处理模板、校准模板、判定阈值 | 排查结果偏差 |
| 数据去向 | 数据库表、导出文件、图片目录、Socket/MES 返回 | 确认流程完成后结果在哪里 |
| 失败证据 | 第一个失败节点、日志时间点、错误信息 | 让后续人员能复现而不是重猜 |

## 最小复测脚本

现场复测或升级后，不要一开始就跑完整生产链路。先用下面顺序确认流程能稳定进入和退出。

1. 打开流程设计页，确认流程模板和起始节点存在。
2. 打开执行页，选择同一条流程模板。
3. 确认相关设备服务在线，必要时先做一次设备最小动作。
4. 准备一组不会影响正式产线的 SN、图片或测试输入。
5. 按 `F6` 执行，记录开始时间、当前节点、最终状态和耗时。
6. 查询同一轮结果：日志、图像、数据库、导出文件或外部响应至少要能对应上。
7. 如需停止，按 `F7` 后确认状态被记录为取消，而不是停在未知状态。

## 失败分叉表

| 失败位置 | 典型表现 | 优先排查 |
| --- | --- | --- |
| 执行前 | 按运行没有反应、提示刷新服务、起始节点缺失 | 注册中心、服务列表、流程模板、起始节点 |
| 预处理 | 一开始就取消或提示预处理失败 | 输入参数、模板合法性、项目窗口上下文 |
| 设备节点 | 等待超时、设备无响应、返回码异常 | 对应设备页、硬件连接、设备 Code、MQTT/串口/IP |
| 模板节点 | 执行完成但结果不对 | 模板版本、阈值、图片来源、校准数据 |
| 数据节点 | 流程完成但看不到结果 | 数据库写入、批次/SN、导出对象、权限 |
| 外部系统 | ColorVision 完成但 MES/Socket 没收到 | 协议配置、端口、项目 handler、响应字段 |

## 常见问题

### 按了运行，但流程没有开始

- 先确认注册中心是否已连接
- 再确认服务列表是否已经刷新出来
- 检查是否选中了有效流程模板
- 检查流程里是否真的存在起始节点

### 流程开始了，但很快停止

- 先看是不是预处理失败
- 再看日志里最后一个节点和对应状态
- 如果是设备相关节点，回到对应设备页确认连接和配置

### 进度不动，看起来像卡住

- 先看当前运行节点名称
- 再判断是不是某个节点正在等待设备或消息返回
- 必要时按 `F7` 停止后，先单独验证那个节点依赖的设备服务

### 手动执行成功，流程里却失败

- 先确认流程引用的是不是同一套设备和模板
- 再确认执行前的环境是否和手动测试一致
- 需要时联动看 [设备服务概览](../devices/overview.md) 和 [日志查看器](../interface/log-viewer.md)
