# Templates 架構設計

本頁只描述當前倉庫裡 Templates 系統的實際設計，不再維護基於檔案數量、目錄數量和理想分層模型展開的舊稿。

## 先把它看成什麼

`Engine/ColorVision.Engine/Templates/` 不是單一“演算法模板目錄”，而是一套混合系統，裡面同時包含：

- 模板抽象基類
- 模板註冊與初始化
- 模板管理和編輯視窗
- 各業務域模板實現
- 流程模板與流程編輯器接入
- 節點配置面板裡對模板的嵌入式呼叫

這決定了它的設計重點不是純計算，而是“模板如何被註冊、編輯、持久化並被其他功能消費”。

## 核心物件怎麼分

### `ITemplate`

`ITemplate` 是所有模板的共同入口，但它不是一個很薄的介面層，而是帶有明顯執行時職責的基類。

它當前承擔的內容包括：

- 模板名稱、標題、程式碼等後設資料
- 列表型訪問入口
- 新建、刪除、儲存、匯入、匯出等操作入口
- 預覽雙擊行為
- 建立視窗和自訂 `UserControl` 支援

還有一個容易被忽略的實現細節：模板物件在構造時會透過 UI 執行緒把自己註冊到 `TemplateControl`。這說明模板註冊並不是一個純靜態配置過程，而是和模板例項化直接繫結。

### `ITemplate<T>`

`ITemplate<T>` 是面向 `ParamModBase` 派生參數模型的泛型基類。它把當前系統裡最常見的模板形態統一成：

- `ObservableCollection<TemplateModel<T>>`
- 基於索引和名稱的訪問
- 預設模板建立
- 匯入內容暫存

換句話說，當前模板設計更像“帶 UI 和持久化語義的列表模板基類”，而不是隻描述參數結構的純 DTO 層。

### `TemplateControl`

`TemplateControl` 是當前模板系統的註冊中心。它的關鍵職責是：

- 在資料庫連線可用後觸發初始化
- 掃描已載入程式集中的 `IITemplateLoad`
- 呼叫各實現的 `Load()`
- 維護全域性模板字典 `ITemplateNames`
- 處理模板名衝突檢查

這套設計說明當前模板發現機制依賴兩個前提：

- 程式集已經裝載
- 資料庫連線已經就緒

如果模板沒有出現，優先看這條鏈，而不是先懷疑編輯視窗。

## 初始化鏈怎麼走

當前模板系統的主鏈路大致是：

1. 主程式和外掛把相關程式集載入到程序裡。
2. `TemplateInitializer` 在依賴 MySQL 初始化後觸發 `TemplateControl.GetInstance()`。
3. `TemplateControl` 掃描已載入程式集中的 `IITemplateLoad` 實現。
4. 各模板型別在 `Load()` 中把自己當前可用的模板項裝進記憶體集合。
5. 模板管理視窗、流程節點配置器、模板編輯視窗再消費這些已註冊例項。

這裡沒有獨立的 DI 容器或模板清單檔案來統一宣告模板，而是“程式集掃描 + 模板物件自注冊 + 具體模板自行載入資料”的組合。

## UI 是怎麼接上來的

### 模板管理視窗

`TemplateManagerWindow` 當前並不是一個簡單列表彈窗。它會：

- 從 `TemplateControl.ITemplateNames` 讀取已註冊模板
- 按名稱空間分組
- 提供搜尋和篩選
- 開啟對應的 `TemplateEditorWindow`

這說明模板在 UI 層的組織方式，當前主要依賴“執行時註冊結果 + 名稱空間分組”，而不是一張獨立維護的模板分類表。

### 模板編輯視窗

`TemplateEditorWindow` 是多數模板的通用操作面。不同模板型別透過各自的 `ITemplate` 實現，把編輯、匯入匯出、雙擊預覽、自訂編輯面板等行為接入這裡。

### 節點配置面板裡的模板入口

模板系統不只存在於“模板管理”選單中。流程節點屬性面板裡的 `NodePanelBuilder` 也會直接開啟 `TemplateEditorWindow`，讓節點配置在原地跳到模板編輯。

這意味著 Templates 實際上是跨視窗複用的共享子系統，而不是孤立模組。

## 資料和持久化現在是什麼風格

當前模板系統並沒有把持久化徹底隔離成一個統一倉儲層。更常見的現狀是：

- 具體模板直接使用 SqlSugar
- 直接讀寫 `ModMasterModel`、`ModDetailModel`
- 有時再聯動 `SysResourceModel` 或其他資源表

因此它更接近“模板物件自己知道如何落庫和取數”的設計，而不是嚴格分層的資料訪問架構。

這也是為什麼很多模板頁如果寫成標準三層架構說明，最終都會失真。

## Flow 模板為什麼要單獨看

流程模板是 Templates 裡最特殊的一支，因為它不僅是模板資料，還直接連到流程編輯和執行能力。

當前 `TemplateFlow` 的幾個關鍵點是：

- 它實現了 `IITemplateLoad`
- 模板程式碼固定為 `flow`
- 雙擊預覽會直接開啟 `FlowEngineToolWindow`
- 支援 `.stn` 和 `.cvflow` 的匯入
- 匯入流程包時會聯動更新流程中的模板引用

這說明 Flow 既是模板，也是流程圖載體，不能和普通參數模板完全等同看待。

## 這個設計當前最明顯的特徵

### 強執行時導向

模板能否出現、能否編輯、能否在節點裡被選中，都和執行時載入鏈強相關。

### UI 與模板邏輯貼得很近

模板抽象、編輯視窗、建立視窗、節點配置器並沒有被刻意隔離到很遠的層次。

### 業務族群是並存的，不是統一模型下的變體

`ARVR/`、`POI/`、`Jsons/`、`Flow/` 等目錄明顯帶有各自演進歷史。當前更適合把它們理解為共用一套模板基礎設施的業務模板族，而不是單一整齊的產品線。

## 讀程式碼時推薦順序

如果要快速建立真實認知，推薦順序是：

1. 先看 `ITemplate.cs`。
2. 再看 `TemplateContorl.cs` 和 `TemplateInitializer`。
3. 然後看 `TemplateManagerWindow.xaml.cs`。
4. 再進入 `TemplateFlow.cs` 或某個具體業務模板。
5. 需要理解流程節點怎麼調模板時，再看 `NodeConfigurator/NodePanelBuilder.cs`。

## 這頁不再做什麼

本頁不再繼續維護這些高風險內容：

- 基於檔案數量和程式碼量的靜態統計
- 看起來完整、但無法和現有實現逐項對應的三層架構圖
- 泛化到所有模板族的統一行為承諾

如果後續要寫重建置議，應單獨開設計提案頁，而不是混在這裡。
