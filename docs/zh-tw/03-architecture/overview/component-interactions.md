# 元件互動

本頁不再維護一張覆蓋全系統、但容易失真的“模組互動矩陣”。當前更有價值的做法，是隻說明幾條在程式碼裡真實存在、而且最常影響改動方向的互動主鏈。

## 先看這幾條主鏈

### 1. 主程式入口鏈

主程式入口在 `ColorVision/`。

它先負責：

- 基礎 DLL 預載入
- 配置、日誌、主題、語言初始化
- 檔案處理命令列分支
- 外掛是否載入的決策
- 啟動向導或啟動視窗

這一層更像“宿主”和“入口排程器”，不是具體業務本身。

### 2. UI 工作臺鏈

進入主工作區後，`UI/` 下的視窗、選單、面板和編輯器負責承接使用者操作，再把動作轉給引擎層。

這裡最常見的互動關係是：

- 主視窗和桌面選單提供入口
- 屬性編輯器、影像編輯器、資料庫瀏覽器等模組負責各自的互動檢視
- 真正的業務動作繼續往 `Engine/` 下發

### 3. 服務樹與裝置鏈

`ServiceManager` 是執行時裡最關鍵的協調物件之一。它會：

- 從資料庫讀取資源和服務定義
- 生成終端服務和裝置服務物件
- 組織分組資源
- 把流程顯示區和裝置顯示控制元件統一接入顯示管理器

因此“裝置為什麼出現在樹裡”“為什麼顯示區能開啟”“為什麼某臺裝置沒進入執行時”這幾件事，通常先看 `ServiceManager`，而不是直接看某個單獨裝置類。

### 4. 模板註冊與編輯鏈

模板系統的關鍵互動不是一張獨立表，而是：

- 相關程式集被主程式或外掛裝載
- `TemplateControl` 掃描其中實現 `IITemplateLoad` 的型別
- 這些型別在 `Load()` 中把模板註冊進全域性模板字典
- 模板編輯器、流程視窗和具體業務功能再去使用這些模板

所以“模板為什麼沒出現”這類問題，通常是載入鏈、資料庫狀態或程式集裝載鏈的問題，而不只是編輯視窗本身的問題。

### 5. 流程執行鏈

流程執行時的主鏈大致是：

1. `DisplayFlow` 選擇並重新整理當前流程模板。
2. `FlowControl` 啟動或停止當前流程。
3. `FlowEngineLib` 執行節點圖。
4. `MQTTRCService` 提供服務令牌和狀態更新。
5. 裝置節點或演算法節點返回結果。
6. 執行狀態、批次記錄、日誌文字和節點訊息被更新回介面。

這個鏈路說明一個常見事實：流程不是孤立模組，它天然依賴模板、裝置服務和註冊中心狀態。

### 6. 外掛擴充套件鏈

外掛裝載後，並不是只多出一個 DLL，而是會把新的程式集帶進整套執行時：

- 新選單或視窗入口
- 新服務或提供者實現
- 新模板、結果檢視或擴充套件點實現

因此外掛與主程式的互動更像“把能力併入現有執行時”，而不是在外部平行執行。

## 改程式碼時通常怎麼用這頁

### 改主視窗或使用者入口

優先順著：

- 主程式入口
- UI 選單/面板
- 對應引擎服務

### 改裝置行為

優先順著：

- `ServiceManager`
- 具體 `DeviceService`
- 對應 MQTT 或執行時狀態物件

### 改模板或流程

優先順著：

- `TemplateControl`
- 模板編輯器
- `DisplayFlow` / `FlowControl` / `FlowEngineLib`

### 改外掛擴充套件

優先順著：

- `PluginLoader`
- 外掛程式集
- 被外掛實現的選單、服務、模板或檢視介面

## 這頁不再做什麼

本頁不再繼續維護這些高風險內容：

- 看似完整、但和當前實現脫節的模組矩陣表
- 大量未核實的事件名稱和介面清單
- 脫離實際程式碼目錄的理論依賴圖

如果某個專題需要更深的設計分析，應放回對應專題頁，而不是在這裡鋪開一張過度承諾的總表。
