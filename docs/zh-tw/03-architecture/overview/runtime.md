# 架構執行時

本頁只描述當前程式碼裡能看見的主程式執行時鏈路，不再繼續維護英文 draft 後設資料和通用啟動時序圖。

## 先怎麼理解執行時

當前桌面程式的執行時不是“一次性把所有模組都初始化完再顯示主介面”的單一模型，而是分成幾條實際分支：

- 命令列只做檔案處理後直接返回
- 正常桌面啟動並進入嚮導或啟動視窗
- 上次異常退出後，先詢問是否禁用外掛再繼續啟動

## 主程式啟動鏈路

從 `ColorVision/App.xaml.cs` 當前實現看，常見啟動順序大致是：

1. 設定工作目錄並按需預載入關鍵 DLL。
2. 初始化配置、日誌、主題和語言。
3. 解析命令列參數。
4. 如果是 `input` 或 `export` 這類檔案處理分支，直接處理後返回。
5. 進行單例項檢查，並在需要時把命令列參數轉發給已執行例項。
6. 非除錯狀態下清理殭屍程序。
7. 根據上次啟動狀態決定是否禁用外掛。
8. 初始化 WinForms 視覺樣式。
9. 根據嚮導完成狀態顯示 `WizardWindow` 或 `StartWindow`。

這裡最重要的不是記住所有步驟，而是知道啟動並不總會直接進主視窗。

## 外掛在什麼時候進入

當前主程式會在進入嚮導或啟動視窗之前決定是否載入外掛。

外掛載入的幾個關鍵點是：

- 掃描 `Plugins/` 目錄
- 讀取每個外掛目錄中的 `manifest.json`
- 可選讀取 `.deps.json`
- 檢查 `ColorVision.*` 依賴版本
- 最後用 `Assembly.LoadFrom(...)` 裝載外掛程式集

如果上次異常退出，啟動時會先詢問是否禁用外掛，這說明外掛是否參與本次執行時是一個明確分支，而不是總是無條件載入。

## 進入主工作區後，哪些執行時物件最關鍵

### 服務樹

`ServiceManager` 會在資料庫連線可用後載入服務樹，把資源表中的資料組織成：

- `TypeServices`
- `TerminalServices`
- `DeviceServices`
- `GroupResources`

之後再把流程顯示區和各裝置顯示控制元件裝進統一顯示管理器裡。

### 註冊中心與服務令牌

`MQTTRCService` 在執行時負責：

- 保持和註冊中心的連線狀態
- 查詢當前可用服務令牌
- 更新服務狀態
- 把服務狀態同步回裝置服務物件

所以很多“流程跑不起來”“裝置線上但狀態不更新”的問題，本質上都不是 UI 問題，而是這裡的執行時狀態沒有準備好。

### 模板註冊

模板系統不是手工逐個硬編碼註冊。當前 `TemplateControl` 會在資料庫連線可用後掃描已載入程式集中的 `IITemplateLoad` 實現，並呼叫它們的 `Load()` 完成模板註冊。

這意味著模板是否可見、是否能被編輯，依賴兩個前提：

- 相關程式集已經載入
- 資料庫連線已建立

## 流程執行在執行時怎麼接上去

當使用者進入流程視窗後，執行時主鏈會繼續延伸到：

- `DisplayFlow` 負責重新整理當前流程模板、啟動或停止流程
- `FlowControl` 負責啟動和停止執行中的流程
- `FlowEngineLib` 負責具體節點執行
- `MQTTRCService` 提供流程執行所需的服務令牌和狀態更新

執行過程中，執行時還會持續更新：

- 當前執行節點
- 執行日誌文字
- 批次進度
- 節點記錄和訊息記錄

## 執行時最常見的失敗點

### 啟動階段

- 外掛依賴不滿足
- 上次異常退出導致需要手動決定是否禁用外掛
- 命令列分支提前返回，誤以為程式沒有繼續啟動

### 服務準備階段

- 資料庫沒有連通，導致服務樹或模板沒有正常裝載
- 註冊中心或 MQTT 側沒有準備好，導致服務令牌為空

### 執行階段

- 流程模板未選中或起始節點缺失
- 裝置服務存在，但執行時狀態還沒同步到可執行狀態
