# 流程執行與除錯

本頁只保留當前程式碼裡能證實的執行入口和排查路徑。對於大多數使用者，流程執行最關鍵的不是“高階除錯術語”，而是先確認選對了流程、服務是否線上、起始節點是否存在，以及失敗時怎麼縮小範圍。

## 當前可直接使用的執行入口

從當前實現看，流程執行至少有這兩類明確入口：

- 開始執行：`F6`
- 停止執行：`F7`

如果你看到舊文件裡寫的是 `F5`、`F10`、斷點或單步執行，請以當前介面和程式碼繫結為準，不要繼續按舊快捷鍵排查。

## 執行前先確認什麼

### 已選中有效流程模板

如果當前沒有選中有效模板，執行不會真正開始。先確認下拉框裡選中的就是你要跑的那條流程，而不是隻開啟了流程視窗。

### 流程裡存在起始節點

當前執行前會檢查起始節點。沒有起始節點時，流程會直接失敗，不會進入真正的執行階段。

### 註冊中心和服務令牌可用

當前實現會先檢查註冊中心連線和可用服務列表。服務令牌為空時，介面會先提示重新整理服務後再重試。

### 預處理透過

流程正式開始前還有一層預處理。預處理失敗時，流程會被取消，不會繼續往下跑。

## 常見執行順序

1. 先在 [流程設計](./design.md) 中確認當前流程內容和起始節點。
2. 在執行視窗選中正確的流程模板。
3. 確認相關裝置服務已經線上。
4. 按 `F6` 或點選執行入口開始執行。
5. 執行過程中觀察日誌區、當前執行節點和進度變化。
6. 需要中止時按 `F7` 或使用停止入口。

## 執行時你能看到什麼

### 當前執行節點

執行過程中，介面會持續顯示當前正在執行的節點名稱。如果一條流程看起來“卡住了”，先看停在哪個節點，而不是立刻懷疑整條流程。

### 執行進度和耗時

當前實現會記錄流程耗時、上次執行時間，並根據上次耗時估算當前進度。這個估算值適合快速判斷大致階段，但不應該當成嚴格的業務完成度。

### 結果和狀態

流程完成後，會寫入這次批次的狀態、耗時和結果摘要；停止執行時，會把狀態記為取消。

## 除錯時應該怎麼縮小範圍

### 先找第一個失敗節點

比起“整條流程失敗了”這種判斷，更有價值的是先找出第一個變紅或沒有繼續推進的節點，再回到對應裝置、模板或輸入資料去查。

### 先分清是沒啟動，還是中途失敗

常見兩類問題：

- 執行前就被攔住：模板未選中、起始節點缺失、註冊中心未連線、服務令牌為空、預處理失敗
- 已經開始執行，但中途失敗：某個節點返回錯誤、超時或訊息不匹配

### 日誌區優先於猜測

當前執行視窗會持續更新日誌文字。看到失敗時，先看最後停在哪個節點、前後是否出現了預處理失敗、取消執行或狀態訊息，再決定回哪一層排查。

## 常見問題

### 按了執行，但流程沒有開始

- 先確認註冊中心是否已連線
- 再確認服務列表是否已經重新整理出來
- 檢查是否選中了有效流程模板
- 檢查流程裡是否真的存在起始節點

### 流程開始了，但很快停止

- 先看是不是預處理失敗
- 再看日誌裡最後一個節點和對應狀態
- 如果是裝置相關節點，回到對應裝置頁確認連線和配置

### 進度不動，看起來像卡住

- 先看當前執行節點名稱
- 再判斷是不是某個節點正在等待裝置或訊息返回
- 必要時按 `F7` 停止後，先單獨驗證那個節點依賴的裝置服務

### 手動執行成功，流程裡卻失敗

- 先確認流程引用的是不是同一套裝置和模板
- 再確認執行前的環境是否和手動測試一致
- 需要時聯動看 [裝置服務概覽](../devices/overview.md) 和 [日誌檢視器](../interface/log-viewer.md)
