# 裝置服務概覽

本頁作為裝置章節入口，優先回答“有哪些裝置頁可看、通常怎麼配置、遇到問題先查哪裡”。

## 裝置服務是什麼

在 ColorVision 裡，裝置通常以“服務”的形式被管理。主程式會維護一個裝置服務列表，使用者在裝置視窗中檢視、配置、啟用和操作這些服務。

裝置相關實現主要位於：

- `Engine/ColorVision.Engine/Services/`
- `Engine/ColorVision.Engine/Services/Devices/`

當前程式碼目錄中可以看到的典型裝置分類包括：

- Camera
- Calibration
- Motor
- FileServer
- FlowDevice
- Sensor
- SMU
- Spectrum

## 常見使用順序

1. 先看 [新增與配置裝置](./configuration.md)，瞭解新增和儲存裝置的基本流程。
2. 再進入具體裝置頁，確認該裝置有哪些參數和操作。
3. 如果涉及相機，繼續看 [相機管理](./camera-management.md) 和 [相機參數配置](./camera-configuration.md)。
4. 如果需要讓裝置參與自動化流程，再看 [工作流程概覽](../workflow/README.md)。

## 使用時通常會遇到什麼

- 一個裝置服務可能繫結真實硬體，也可能只是某類通訊或檔案型服務。
- 裝置列表順序、啟用狀態和配置內容通常會影響後續視窗和流程裡的可見性。
- 某些裝置除了基礎配置，還會有獨立的物理裝置管理、標定或參數配置頁。

## 裝置交接要記錄什麼

現場交付裝置時，不要只寫「裝置已配置」。後續接手人員至少需要知道這台裝置怎麼識別、怎麼連通、怎麼證明它能被流程使用。

| 記錄項 | 要寫清楚什麼 | 用途 |
| --- | --- | --- |
| 裝置身份 | 名稱、Code、裝置類型、對應真實硬體或服務 | 防止流程綁到錯誤裝置 |
| 通訊配置 | IP、端口、串口、波特率、MQTT topic、檔案路徑等 | 復現現場連線條件 |
| 最小動作 | 相機拍圖、電機回零、SMU 點測、檔案服務列目錄等 | 證明裝置不是只出現在列表裡 |
| 流程綁定 | 被哪條流程、哪個節點或哪個專案視窗引用 | 排查「手動能用、流程失敗」 |
| 日誌證據 | 成功連線時間點、失敗錯誤、超時或權限資訊 | 給遠端排查留下定位線索 |
| 回退方式 | 上一版配置、驅動、韌體或專案包位置 | 升級後異常時能退回 |

## 現場最小驗收

| 步驟 | 操作 | 通過標準 |
| --- | --- | --- |
| 1 | 打開裝置列表並刷新 | 目標裝置可見，名稱和 Code 與交接記錄一致 |
| 2 | 打開裝置詳情或專用視窗 | 關鍵參數可見，保存狀態明確 |
| 3 | 執行一個不會破壞現場狀態的最小動作 | 有返回、有日誌、狀態變化可解釋 |
| 4 | 打開依賴它的流程或專案視窗 | 能選到同一台裝置，而不是相似名稱的舊裝置 |
| 5 | 跑一條最小流程或模擬流程 | 裝置節點能執行到，並能定位結果或失敗原因 |

如果第 3 步失敗，先查裝置層；如果第 3 步通過但第 5 步失敗，優先查流程綁定、模板參數和專案映射。

## 裝置問題分層定位

| 現象 | 先查 | 再查 |
| --- | --- | --- |
| 列表裡沒有裝置 | 是否建立並保存、類型是否正確 | 插件/專案包是否提供該裝置入口 |
| 裝置有但打不開 | 硬體在線、驅動、端口/IP、權限 | 日誌裡的連線異常和超時 |
| 手動動作成功但流程失敗 | 流程節點引用的裝置 Code | 節點參數、模板版本、專案視窗選擇 |
| 結果有但圖像或資料不對 | 裝置參數和採集條件 | 後續模板、匯出欄位或資料庫記錄 |
| 升級後才異常 | 配置是否被覆蓋 | 舊包、舊配置和新 DLL 是否混用 |

## 排查建議

### 裝置沒有出現在列表裡

- 先確認是否已經在裝置配置視窗中建立並儲存。
- 再確認對應裝置依賴是否已經滿足，例如物理硬體、驅動或通訊環境。

### 裝置出現了，但無法工作

- 優先檢查該裝置專題頁裡的參數說明。
- 再檢查日誌和連線狀態。
- 若是流程裡呼叫失敗，再聯動檢視 [流程執行與除錯](../workflow/execution.md)。
