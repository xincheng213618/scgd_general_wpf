# 流程裝置服務

流程裝置服務適合放在“流程會呼叫它，但你平時不會頻繁手動操作它”的位置來理解。當前程式碼裡它更像一個給流程系統使用的裝置入口，重點在於配置、連線和被流程正確引用，而不是複雜的獨立操作面板。

## 什麼時候應該看這一頁

- 你要在裝置列表裡新增一個流程裝置服務
- 你要讓某個流程節點引用這類裝置
- 你在排查“裝置已經建了，但流程裡調不到”這類問題
- 你想確認這類服務是否只是配置入口，而不是獨立業務視窗

## 先怎麼理解它

在當前實現裡，流程裝置服務主要提供兩件事：

- 作為一個標準裝置服務例項被系統管理
- 作為流程執行時可以引用的裝置物件被 MQTT 服務層識別

也就是說，先保證它被正確建立、儲存並能被流程引用，比研究更多內部訊息細節更重要。

## 常見使用順序

1. 先在裝置列表裡建立並儲存流程裝置服務。
2. 確認裝置程式碼、主題或相關連線參數已經配置正確。
3. 回到 [工作流程概覽](../workflow/README.md) 或具體流程視窗裡引用這臺裝置。
4. 執行流程時觀察是否真的命中了這臺裝置，而不是隻確認它出現在列表裡。

## 使用時要特別注意什麼

### 裝置存在，不代表流程已經在用它

裝置能顯示在列表裡，只說明它作為資源被建立了；真正要看的是流程節點裡是否引用了正確的裝置例項。

### 先查配置，再查流程節點

當流程裡調不到裝置時，優先先確認：

- 當前裝置程式碼是否正確
- 流程裡引用的是不是這臺裝置
- 相關通訊配置是否匹配

### 不要假設它有很多獨立手動動作

從當前實現來看，這類裝置頁更偏配置和接入層。排查時不要先去找複雜的獨立控制按鈕，先把裝置配置和流程引用關係查清。

## 常見問題

### 裝置已經建立，但流程裡找不到

- 先確認裝置是否已經儲存成功
- 再確認流程節點引用的是不是正確的裝置物件
- 如果流程裡按程式碼或名稱篩選，檢查是否存在不匹配

### 流程執行時裝置沒有響應

- 先確認裝置服務本身的連線配置
- 再確認流程執行到的是否真的是對應節點
- 同時結合 [日誌檢視器](../interface/log-viewer.md) 看是不是訊息層沒有命中這臺裝置

### 不確定問題在裝置層還是流程層

- 先把問題拆成兩段：裝置是否存在、流程是否引用
- 如果裝置配置本身就不完整，先修裝置
- 如果裝置沒問題，再回到 [流程執行與除錯](../workflow/execution.md) 縮小範圍
