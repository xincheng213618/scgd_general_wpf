# Socket 通訊模組最佳化路線

本文件面向 `UI/ColorVision.SocketProtocol`，用於把當前可用的 Socket 管理能力繼續推進成更穩定、更容易排查、更適合長期執行的通訊模組。

## 當前結構判斷

`ColorVision.SocketProtocol` 當前已經形成完整閉環：

- `SocketInitializer` 按配置啟動或停止服務
- `SocketManager` 承擔 TCP 監聽、客戶端管理和請求分發
- `SocketJsonDispatcher` / `SocketTextDispatcher` 負責 JSON 與文字模式
- `SocketMessageManager` 將訊息寫入 SQLite
- `SocketManagerWindow` 和 `SocketStatusBarProvider` 提供桌面側運維入口

這說明它已經不是“協議示例程式碼”，而是帶 UI、持久化和執行時狀態的通訊子系統。最佳化時應優先保護現有行為，再逐步補齊協議邊界和長期執行能力。

## 已完成的體驗最佳化

管理視窗已經補齊一批低風險能力：

- 頂部服務狀態：啟用狀態、監聽地址、協議模式、連線狀態、客戶端數量
- 開啟失敗提示：最後一次啟動錯誤會顯示在視窗頂部和服務診斷頁
- 訊息過濾：按客戶端、事件名、MsgID、響應碼、內容過濾
- 方向過濾：全部、接收、傳送
- 列表虛擬化：降低歷史訊息瀏覽時的 UI 壓力
- 詳情格式化：JSON 內容可格式化檢視，也保留原文複製
- 自動滾動：新訊息到達時可自動定位，也可關閉
- 重發兜底：優先匹配原客戶端，匹配不到時使用當前選中的客戶端
- 佈局收斂：右側用“訊息詳情 / 客戶端 / 服務診斷”標籤頁承載輔助資訊
- 快捷鍵：`Ctrl+F`、`Esc`、`F5`、`Ctrl+C`、`Delete`

## P0 穩定性路線

目標：先把服務生命週期和 TCP 收發邊界收緊，減少現場“偶現卡住、埠占用、訊息截斷”的問題。

| 專案 | 當前風險 | 建議做法 | 驗收標準 |
| --- | --- | --- | --- |
| 服務重複啟動 | 多次呼叫 `StartServer()` 會建立新的後臺任務 | 增加 `_serverTask`、`CancellationTokenSource` 和啟動狀態保護 | 連續點選啟停不會產生多個監聽迴圈 |
| 停止服務 | `AcceptTcpClient()` 依賴 `TcpListener.Stop()` 拋異常退出 | 顯式取消、捕獲取消路徑，區分正常停止和錯誤停止 | 停止服務日誌不再表現為異常錯誤 |
| TCP 邊界 | 當前按一次 `Read` 當作一條訊息 | 引入長度字首、分隔符或可配置幀解析器 | 大訊息和連續小訊息不會粘包/半包 |
| 客戶端集合 | `ObservableCollection<TcpClient>` 與後臺執行緒互動需要保持 UI 執行緒約束 | 封裝 Add/Remove，所有集合變更走 Dispatcher | 長時間連線/斷開不拋跨執行緒異常 |
| 客戶端清理 | 斷開、停止、異常路徑分散 | 增加統一 `CloseClient` 方法 | 客戶端斷開後列表、資源和日誌一致 |

## P1 可觀測性路線

目標：讓視窗和日誌能回答“現在誰連著、剛剛發生了什麼、失敗集中在哪”。

| 專案 | 建議做法 | 驗收標準 |
| --- | --- | --- |
| 連線生命週期 | 增加連線建立、斷開、異常原因記錄 | 管理視窗可看到連線歷史或日誌可定位 |
| 訊息匯出 | 支援匯出當前過濾結果為 JSON/CSV | 現場可以直接帶走問題樣本 |
| 錯誤統計 | 統計 JSON 解析失敗、Handler 未找到、傳送失敗 | 狀態列或視窗能顯示失敗計數 |
| 處理耗時 | 記錄 Handler 執行耗時 | 可以定位慢事件 |
| 資料庫保留策略 | 按數量或時間清理歷史 | 長期執行不會無限增長 |

## P2 協議化路線

目標：把外部裝置對接從“約定俗成”推進到“可校驗、可相容、可遷移”。

| 專案 | 建議做法 | 驗收標準 |
| --- | --- | --- |
| 訊息模型 | 明確 `Version`、`MsgID`、`EventName`、`SerialNumber`、`Params` 的必填性 | 文件和執行時校驗一致 |
| 錯誤碼 | 定義統一錯誤碼範圍 | 客戶端能根據 Code 做穩定處理 |
| Handler 後設資料 | 為處理器增加描述、版本、示例參數 | 管理視窗或文件可列舉支援事件 |
| Schema | 為重點事件提供 JSON Schema | 對接方可提前校驗 |
| 相容策略 | 約定新老版本欄位相容方式 | 升級不破壞已有裝置 |

## P3 效能與容量路線

目標：支援更長時間執行、更高訊息頻率和更大的訊息歷史。

| 專案 | 建議做法 | 驗收標準 |
| --- | --- | --- |
| UI 分頁 | 管理視窗按分頁或增量載入歷史 | 開啟視窗不因歷史庫變大而卡頓 |
| 批次寫庫 | 高頻訊息時批次寫入或後臺佇列寫入 | 訊息高峰下 UI 不被資料庫寫入拖慢 |
| 索引 | 為 `MessageTime`、`ClientEndPoint`、`EventName` 增加索引 | 查詢和過濾歷史資料可控 |
| 背壓 | 對極端訊息頻率設定佇列上限和降級策略 | 過載時可提示而不是拖垮程序 |
| 壓測 | 增加本地 TCP 壓測指令碼 | 可復現容量上限和迴歸結果 |

## 建議實施順序

1. 先做 P0 的服務生命週期保護和幀解析，避免協議層隱患繼續放大。
2. 再補 P1 的匯出、統計和保留策略，提升現場問題處理效率。
3. 然後推進 P2 的協議文件、錯誤碼和 Handler 後設資料。
4. 最後根據真實訊息量決定 P3 的分頁、批次寫庫和壓測投入。
