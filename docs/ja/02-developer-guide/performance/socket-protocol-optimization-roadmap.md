# ソケット通信モジュール最適化ルート

このドキュメントは `UI/ColorVision.SocketProtocol` を対象としており、現在利用可能なソケット管理機能を、より安定し、トラブルシューティングが容易で、長期運用に適した通信モジュールへと継続的に推進するために使用されます。

## 現在の構造判定

`ColorVision.SocketProtocol` は現在、完全な閉ループを形成しています。

- `SocketInitializer` は構成に従ってサービスを開始または停止します
- `SocketManager` は、TCP 監視、クライアント管理、およびリクエストの配布を担当します。
- `SocketJsonDispatcher` / `SocketTextDispatcher` は JSON およびテキスト モードを担当します
- `SocketMessageManager` は SQLite にメッセージを書き込みます
- `SocketManagerWindow` および `SocketStatusBarProvider` は、デスクトップ側の操作およびメンテナンスの入り口を提供します

これは、これがもはや「プロトコル サンプル コード」ではなく、UI、永続性、実行時状態を備えた通信サブシステムであることを示しています。最適化するときは、既存の動作を保護することを優先し、その後、プロトコルの境界と長期的な動作機能を徐々に完成させる必要があります。

## エクスペリエンスの最適化が完了しました

管理ウィンドウは、低リスク機能のバッチを完了しました。

- 最上位のサービスステータス: 有効ステータス、リスニングアドレス、プロトコルモード、接続ステータス、クライアント数
- 失敗プロンプトを開く: 前回の起動エラーがウィンドウの上部とサービス診断ページに表示されます。
- メッセージのフィルタリング: クライアント、イベント名、MsgID、応答コード、内容によるフィルタリング
- 方向フィルタリング: すべて、受信、送信
- リストの仮想化: 履歴メッセージを閲覧する際の UI の負担を軽減します。
- 詳細な書式設定: JSON コンテンツを表示用に書式設定でき、元のテキストをコピーすることもできます
- 自動スクロール: 新しいメッセージが到着したときに自動的に配置することも、オフにすることもできます
- 再送信: 元のクライアントと一致することを優先し、一致するものが見つからない場合は現在選択されているクライアントを使用します。
- レイアウトの収束: 右側の「メッセージの詳細/クライアント/サービス診断」タブを使用して補助情報を伝えます。
- ショートカット キー: `Ctrl+F`、`Esc`、`F5`、`Ctrl+C`、`Delete`

## P0 安定ルート

目標: まず、サービス ライフ サイクルと TCP 送受信境界を強化し、サイトでの「時折発生するスタック、ポート占有、メッセージの切り捨て」の問題を軽減します。

|プロジェクト |現在のリスク |推奨される実践方法 |合格基準 |
| --- | --- | --- | --- |
|サービスが繰り返し開始される | `StartServer()` を複数回呼び出すと、新しいバックグラウンド タスクが作成されます。 `_serverTask`、`CancellationTokenSource`、および起動ステータス保護を追加 |連続してクリックして開始および停止しても、複数のリスニング ループは生成されません。
|サービスを停止する | `AcceptTcpClient()` は `TcpListener.Stop()` に依存しており、例外をスローして終了します。明示的なキャンセル、キャンセル パスのキャプチャ、通常の停止とエラー停止の区別 |サービスを停止してもログに異常なエラーが表示されなくなりました。
| TCP 境界 |現在、1 つのメッセージとして `Read` を押します。長さのプレフィックス、デリミタ、または設定可能なフレーム パーサーを導入します。大きなメッセージと連続した小さなメッセージは貼り付けられません/半分のパケットになります。
|クライアントコレクション | `ObservableCollection<TcpClient>` バックグラウンド スレッドと対話するには、UI スレッド制約を維持する必要があります。追加/削除をカプセル化し、コレクションの変更はすべて Dispatcher | に移動します。長期間の接続/切断ではクロススレッド例外がスローされません。
|クライアントのクリーンアップ |切断・停止・パス分散異常 |統合された `CloseClient` メソッドを追加 |クライアントが切断された後、リスト、リソース、ログの一貫性を保つ |

## P1 可観測性ルート

目標: ウィンドウとログが「現在誰が接続しているか、何が起こったのか、どこに障害が集中しているのか」を答えられるようにします。

|プロジェクト |推奨される実践方法 |合格基準 |
| --- | --- | --- |
|接続のライフサイクル |接続確立、切断、および異常理由レコードを追加 |管理ウィンドウでは接続履歴を確認したり、ログを見つけることができます。
|メッセージのエクスポート |現在のフィルタリング結果を JSON/CSV としてエクスポートすることをサポート |問題のあるサンプルは現場で直接持ち帰ることができます。
|エラー統計 |統計 JSON 解析失敗、ハンドラーが見つかりません、送信失敗 |ステータス バーまたはウィンドウに失敗数を表示できます。
|処理時間 |レコード ハンドラーの実行時間 |遅いイベントを見つけることができます |
|データベース保持ポリシー |量または時間ごとの履歴のクリーンアップ |長期実行は無制限に増加するわけではありません。

## P2プロトコルルート

目標: 外部デバイスのドッキングを「従来型」から「検証可能、互換性、移行可能」に推進します。

|プロジェクト |推奨される実践方法 |合格基準 |
| --- | --- | --- |
|メッセージモデル | `Version`、`MsgID`、`EventName`、`SerialNumber`、`Params` の必須フィールドを明確にします。ドキュメントと実行時の検証の一貫性 |
|エラーコード |統一されたエラー コード範囲を定義する |クライアントはコードに基づいて安定した処理を行うことができます |
|ハンドラーメタデータ |ハンドラーの説明、バージョン、サンプルパラメーターを追加します。管理ウィンドウまたはドキュメントでサポート イベントを列挙できます。
|スキーマ |主要なイベントに JSON スキーマを提供する |ドッキング当事者は事前にそれを確認できます。
|互換性戦略 |新旧バージョン間のフィールド互換性に関する合意 |既存の設備を破壊せずにアップグレード |

## P3 のパフォーマンスと容量のルート

目標: より長い実行、より高いメッセージ頻度、およびより大きなメッセージ履歴をサポートします。

|プロジェクト |推奨される実践方法 |合格基準 |
| --- | --- | --- |
| UI ページング |管理ウィンドウは、ページングまたは段階的に履歴を読み込みます。履歴データベースの拡大により、開くウィンドウが動かなくなることはありません。
|データベースへのバッチ書き込み |高頻度メッセージ中のバッチ書き込みまたはバックグラウンド キュー書き込み |メッセージのピーク時にデータベースの書き込みによって UI が遅くなることはありません。
|インデックス | `MessageTime`、`ClientEndPoint`、`EventName` のインデックスを追加します。履歴データの制御可能なクエリとフィルタリング |
|背圧 |極端なメッセージ頻度に対するキュー キャップと劣化戦略を設定する |過負荷時にプロセスをダウンさせるのではなくプロンプトを表示する |
|ストレステスト |ローカル TCP ストレス テスト スクリプトを追加する |再現可能な容量の上限と回帰結果 |

## 推奨される実装順序

1. プロトコル層での隠れた危険性のさらなる増幅を避けるために、最初にサービス ライフ サイクル保護と P0 のフレーム解析を実行します。
2. P1 のエクスポート、統計、保持戦略を追加して、オンサイトの問題処理の効率を向上させます。
3. 次に、P2 のプロトコル ドキュメント、エラー コード、およびハンドラー メタデータを進めます。
4. 最後に、実際のメッセージ量に基づいて、P3 のページング、バッチ書き込み、ストレス テストへの投資を決定します。
