# Copilot コンテキスト統合 v1

この文書には、最初のバージョンの設計境界、実装された項目、および「独立したチャット パネル」から「ソフトウェア コンテキスト サービス」への ColorVision Copilot の進化のその後の計画が記録されています。

## ターゲット

現在、Copilot にはすでにセッション、モデル構成、添付ファイル、ツール呼び出し、ログ診断、および右側の Dock パネルがありますが、ビジネス オブジェクトの理解は依然として主にファイル パスとユーザーによる手動マウントのコンテキストにとどまっています。

次の段階の目標は、1 つの時点でチャット機能を強化し続けることではなく、Copilot をさまざまなモジュールから呼び出せるパブリック プラットフォーム機能に変えることです。

- ビジネス モジュールは、ColorVision/Copilot ディレクトリの実装に直接依存しません。
- Copilot は、現在のソフトウェア シナリオの構造化されたコンテキストを自動的に伝達できます。
- Copilot への入り口は、ステータス バーから、ソリューション ツリー、イメージ、プロセス、アルゴリズムの結果、例外などのワークフロー ノードまで広がっています。

## 今回実装した簡易版

今回は、まず低リスクで再利用可能なベースレイヤーを完成させました。

1. 公契約層

- `ICopilotService`、`ICopilotContextProvider`、および最小限のリクエスト/コンテキスト モデルを UI/ColorVision.Common/Interfaces/Copilot に追加しました。
- `CopilotPanelService` は `ICopilotService` の実装として機能し、`CopilotServiceRegistry` を介して他のモジュールに公開されます。

2. 最小限のコンテキストプロバイダー

- `CopilotContextRegistry` と `CopilotWorkspaceContextProvider` を追加しました。
- 現在のエージェント リクエストは、ソリューション ルート ディレクトリ、現在のアクティブ コンテンツ、現在の検索ルートを含む「現在のワークスペース」コンテキストに自動的にハングされます。
- このステップはビジネス オブジェクトを直接理解するのではなく、最初に「ソフトウェアの現在の作業面」をパブリックで拡張可能なコンテキスト エントリに変換します。

3. ソリューションツリーのエントリ

- `FileNode` 「AI にこのファイルの解釈を依頼する」と「AI にこのファイル/ログの診断を依頼する」を追加しました。
- `FolderNode` 「AIにこのフォルダーを要約してもらう」を追加しました。
- これらのエントリは、Copilot 実装を直接参照するのではなく、パブリック `ICopilotService` を介してトリガーされます。

4. ブランドの統一

- ステータス バーのエントリ名が `GitHub Copilot` から `ColorVision Copilot` に調整されました。

## なぜ最初にここで停止するのでしょうか?

このバージョンは、次の理由により、意図的に Engine、ImageEditor、Flow、または Device Services レイヤーに直接組み込まれていません。

- これらのモジュールが ColorVision/Copilot を直接参照すると、すぐに逆結合が形成されます。
・現在最も不足しているのは「チャット機能の充実」ではなく、「業務モジュールから安定して呼び出せるコンテキストインターフェース」です。
- 最初に公共契約、最小限のコンテキスト収集、および高頻度の入り口をクリアし、その後、低コストでビジネス オブジェクトを接続します。

## フォローアップ計画

### フェーズ 1: コンテキスト ブリッジの拡張

最初にこれらのプロバイダーを追加します。

- 現在のウィンドウ/現在のページ プロバイダー
- 現在選択されているオブジェクトプロバイダー
- ROI プロバイダーを使用した現在の画像
- 現在のログおよびフィードバック パッケージのプロバイダー
- 現在のプロセス/ノード/最近の障害プロバイダー

この段階の受け入れ基準は次のとおりです。

- ユーザーが任意のキー ページから Copilot を開始する場合、現在のオブジェクトの説明を手動で繰り返す必要はありません。
- エージェントは、単なるファイル パスではなく、構造化されたアプリケーション コンテキストを確認できます。

### フェーズ 2: シーンの入り口

以下の種類の入口からのアクセスが優先されます。

- 画像ページを右クリック: 現在の画像を分析し、現在の ROI を解釈します。
- アルゴリズム結果ページ: この結果と失敗した理由を説明します
- フロー: 現在のプロセスを説明し、最後の失敗を診断します。
- デバイスパネル: 現在のデバイスステータスを分析します。
- 例外ダイアログ ボックス: 分析は Copilot に任せます

この段階では、「パブリック インターフェイスを呼び出す」ことを主張し、これらのモジュールが Copilot 固有のウィンドウや ViewModel に直接依存しないようにする必要があります。

### フェーズ 3: ナビゲーション アクション

解釈/診断が安定したら、リスクの低い UI アクションを導入します。

- パネルを開く
- ウィンドウに焦点を当てる
- 構成アイテムを見つける
- ログディレクトリを開きます

この段階では、ナビゲーションと測位のみが実行され、構成の変更、機器の制御、プロセスの実行などのリスクの高いアクションは実行されません。

### フェーズ 4: ビジネス ツール

既存の SearchFiles、GrepText、ReadLocalFile、および GetRecentLog に基づいて、読み取り専用のビジネス ツールが追加されています。

- `GetActiveWorkspaceContextTool`
- `GetSelectedImageContextTool`
- `GetAlgorithmResultContextTool`
- `GetFlowRunSummaryTool`
- `GetDeviceStatusTool`
- `GetDiagnosticsPackageSummaryTool`

## Agent/ReAct ルートとの関係

Copilot Context Integration v1 は、「一般的なコード エージェントのように動作するかどうか」ではなく、「現在のソフトウェア シナリオを理解できるかどうか」に対処します。

したがって、後続のルートは 2 つの平行な線に分割する必要があります。

- コンテキストの統合: ソフトウェア内のコンテキストとエントリを解決します。
- Agent/ReAct: ツールの計画、低コストの取得、構造化された実行、およびその他の診断ツールを解決します。

どちらのルートも重要ですが、現在の優先順位はコンテキスト統合である必要があります。

## 現在の境界

このバージョン以降:

- パブリック Copilot コントラクト レイヤーがすでに存在します。
- 最小限のワークスペース コンテキスト プロバイダーがすでに存在します。
- ソリューションツリーへの高頻度の入り口があります。
- ImageEditor、Flow、Algorithm、Device のビジネス オブジェクト コンテキストがまだ接続されていません。
- UI アクション カード、ビジネス ツール、およびグローバル設定ページの埋め込みはまだ実装されていません。

その後の設計とスケジュールはこのドキュメントに基づいて行うことをお勧めします。
