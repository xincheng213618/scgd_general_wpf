#FlowEngineLib アーキテクチャ

このページでは、現在のウェアハウスで実際に実行されているプロセスの編集と実行チェーンについてのみ説明します。 FlowEngineLib を独立した階層化フレームワークとして作成するという古いドラフトは維持されなくなります。

## まず、システム内のその場所を確認します。

FlowEngine 関連の機能は `Engine/FlowEngineLib/` に限定されたものではありません。現在の実際の使用チェーンは 2 つの層にまたがっています。

- `FlowEngineLib/` は、ノード エディターでの実行制御、開始ノード、終了ノード、サービス ノードの基本機能を提供します。
- `Engine/ColorVision.Engine/Templates/Flow/` は、プロセス テンプレート、編集ウィンドウ、実行ウィンドウ、バッチ処理をメイン プログラムに接続する役割を果たします。

したがって、FlowEngine アーキテクチャについて議論する場合、ライブラリ自体について書くだけでは、実際の実行時間の半分を見逃してしまいます。

## 現時点で最も重要なオブジェクト

### `FlowEngineControl`

`FlowEngineControl` は、実行制御の中心となるオブジェクトです。現在、次のことを担当しています。

- `STNodeEditor` をバインドする
- エディターにノードを追加するときに開始ノードとサービス ノードを識別します
- `startNodeNames` および `services` を保守します
- ファイルまたは Base64 からキャンバスを読み込みます
- 開始ノードを選択し、プロセスを開始および停止します
- 開始ノードが完了すると `Finished` をスローします

これは、これが抽象的なスケジューリング インターフェイスではなく、ノード エディター インスタンス、ノード オブジェクト、およびサービス コレクションに直接関連付けられたランタイム コントローラーであることを意味します。

### `BaseStartNode`

開始ノードは、プロセス実行を `CVStartCFC` にカプセル化し、開始、停止、および完了イベントをフローチャートに沿ってディスパッチする役割を果たします。

現在の要点は次のとおりです。

- `Start(serialNumber)` は `CVStartCFC` を作成します
- `DoDispatch(...)` はアクションを下流に渡します
- `FireFinished(...)`は実際にプロセス終了イベントが発行される場所です。

したがって、「プロセスの完了」はコントローラー自体によって推論されるのではなく、最終的には開始ノードによって発行されます。

### `CVBaseServerNode`

ほとんどのデバイス ノードとアルゴリズム ノードは `CVBaseServerNode` システムに分類されます。彼らは以下の責任を負います。

- 実行時アクションを送信して待機する
- タイムアウト、障害を処理し、データを返す
- ノードの結果を下流に渡す
- `nodeEndEvent` 経由で個々のノードの終了ステータスをレポートします

ここの `nodeEndEvent` は重要ですが、これはノード レベルの終了を示すだけであり、プロセス全体が終了したことを意味するわけではありません。

### `CVEndNode`

エンド ノードは、プロセス完了チェーンの最後のホップです。現在の実装では、エンド ノードは処理を終了するときに `startAction.FireFinished()` を呼び出し、プロセス全体を完了としてマークします。

これが、システムにおいて「特定のノードの実行が終了した」ことと「プロセス全体が終了した」ということが異なる理由です。

## プロセスは実際にどのように実行されるのでしょうか?

現在のメインチェーンはおおよそ次のとおりです。

1. `TemplateFlow` または `FlowEngineToolWindow` はプロセス データを準備します。
2. `FlowEngineToolWindow` は `FlowEngineLib.dll` をノード エディターにロードします。
3. `FlowEngineControl` は `STNodeEditor` をバインドし、ノードが参加するときに開始ノードとサービス ノードを識別します。
4. `LoadFromBase64(...)` または `Load(...)` は、フローチャートをキャンバスにロードします。
5. `StartNode(...)` 開始ノードを指定するか、デフォルトで最初の開始ノードを選択します。
6. `BaseStartNode` は `CVStartCFC` を作成し、それをダウンストリーム ノードにディスパッチします。
7. 各 `CVBaseServerNode` 派生ノードは、独自の実行、タイムアウト、およびデータ配信を処理します。
8. `CVEndNode` は最後に `startAction.FireFinished()` を呼び出します。
9. `BaseStartNode.Finished` がトリガーされます。
10. `FlowEngineControl.Start_Finished(...)` を独自の `Finished` イベントに変換します。

この補完チェーンは、古いドキュメントの「特定のノードの終了はプロセスの終了を意味する」よりも厳密であり、現在のコードに近いものです。

## エンジン層はどのようにメインプログラムに接続するのでしょうか?

### プロセステンプレート

`TemplateFlow` を使用すると、フローチャートをシステム内にテンプレートとして存在させることができ、以下がサポートされます。

- テンプレートリスト管理
- ダブルクリックしてプロセスエディタを直接開きます
- `.stn` / `.cvflow` インポート
- プロセス パッケージをインポートする際のプロセス関連テンプレート

### 編集ウィンドウ

`FlowEngineToolWindow` は、独立したプロセス編集面です。それは次のことを担当します。

- ベアリング `STNodeEditor`
- `FlowEngineLib.dll` をロードする
- 元に戻す、やり直し、コピー、貼り付け、ズーム、自動位置合わせにアクセスします
- `STNodeEditorHelper` を介してプロパティ パネルとノード ツリーを接続する

したがって、現在の編集エクスペリエンスには `FlowEngineLib` 独自の UI が付属していませんが、エンジン レイヤーは WPF ウィンドウのレイヤーをラップして取得します。

### 実行ウィンドウ

メイン プログラムの実際の日常使用に該当するのは、`DisplayFlow` 行と `FlowControl` 行です。

`DisplayFlow` は現在、次のことを担当しています。

- 現在のプロセス テンプレートを更新します
- 開始前に前処理を実行する
- 監視プロセスが完了しました
- 実行ログ、バッチ情報、進捗状況を書き込みます
- プロセス完了後にカスタムバッチをトリガー

これは、メイン プログラムで実行されているプロセスが単に「グラフを実行している」だけではなく、バッチ レコード、ログ テキスト、および後処理拡張機能にも関連付けられていることを示しています。

## 間違って記述されやすい現在の境界は何ですか?

### `nodeEndEvent` はプロセス完了イベントではありません

`CVCommonNode` の `nodeEndEvent` は、ノードレベルのフィードバックにのみ使用されます。実際のプロセス完了チェーンは次のとおりです。

- EndNode が `startAction.FireFinished()` を呼び出します
- `CVStartCFC.FireFinished()` 開始ノードに戻ります
- `BaseStartNode.Finished` がトリガーされる
- `FlowEngineControl.Finished` を捨ててください

これら 2 つのイベントが混同されると、障害の伝播、進捗状況の更新、および最終的な完了判断のすべてに偏りが生じます。

### 開始ノードはどのノードでもありません

`FlowEngineControl` は、ノードが参加するときに `BaseStartNode` を `startNodeNames` に含めるだけです。プロセスを開始するときに名前が指定されていない場合は、デフォルトで最初の開始ノードが使用されます。

したがって、プロセスを開始できるかどうかは、開始ノードが存在するかどうか、および開始ノードの準備ができているかどうかに直接関係します。

### 障害の伝播はノードの種類によって異なります

プロセスの障害はコントローラーによって一律に判断されるわけではありません。多くの障害、タイムアウト、またはキャンセルはノード内で内部的に生成され、ネットワークに沿って伝播します。特に複数入力ノードの場合、障害の伝播動作は、コントローラーの表面状態だけではなく、特定のノードの実装に依存します。

## 展開時に通常どこに配置されるか

### 新しいノード

新しいプロセス ノードが追加される場合、通常は `FlowEngineLib` 自体のノード実装と、それが開始ノード、終了ノード、またはサービス ノード チェーンにどのように接続されるかに焦点が当てられます。

### 新しいテンプレートのプロセス

新しいタイプの編集可能なプロセス テンプレートが追加される場合、通常は `TemplateFlow` 隣接するテンプレート管理、インポート、エクスポート、および編集ウィンドウ アクセスに重点が置かれます。

### ノードプロパティパネルの拡張機能

新しいプロセス ノード構成 UI を追加する場合、通常はノード クラス自体を変更するのではなく、`STNodeEditorHelper` または `NodeConfigurator` 領域に分類されます。

## 推奨される読む順序

次の行を読むことをお勧めします。

1. `Engine/FlowEngineLib/FlowEngineControl.cs`
2. `Engine/FlowEngineLib/Start/BaseStartNode.cs`
3. `Engine/FlowEngineLib/End/CVEndNode.cs`
4. `Engine/FlowEngineLib/Base/CVBaseServerNode.cs`
5. `Engine/ColorVision.Engine/Templates/Flow/TemplateFlow.cs`
6. `Engine/ColorVision.Engine/Templates/Flow/FlowEngineToolWindow.xaml.cs`
7. `Engine/ColorVision.Engine/Templates/Flow/DisplayFlow.xaml.cs`

このようにして、最初にメインの実行チェーンを確立し、その後編集とメイン プログラムの統合に戻すことができます。

## このページはそれ以上何もしません

このページでは、次のような危険性の高いコンテンツは管理されなくなります。

- FlowEngineLib を現在の実装とは切り離された標準の階層化フレームワークとして説明する
- 一連の抽象的な設計パターンですべてのノードの動作をカバーします
- MQTT、ロギング、シリアル化、その他の周辺機器を独立したインフラストラクチャ層コミットメントにパッケージ化します。

後で再構築の方向性について議論したい場合は、特定の実行チェーンと実際のノード システムを出発点として取り上げる必要があります。
