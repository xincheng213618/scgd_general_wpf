# テンプレートモジュールの分析

このページは、現在のウェアハウス内のテンプレート システムの現在のステータスを分析するだけであり、「追加されたドキュメントの数」や「計画された最適化計画の数」などの無効な概要コンテンツは保持されなくなりました。

## まずこのモジュールが何であるかを見てみましょう

`Engine/ColorVision.Engine/Templates/` は単一のアルゴリズム ディレクトリではなく、次の種類の責任を同時に引き受けるコードのセットです。

- テンプレートの抽象化と登録
- テンプレート管理・編集画面
- さまざまなビジネスドメイン向けのテンプレートの実装
- プロセステンプレート関連の機能
- テンプレートの検索、作成、サンプル保存などの補助ツール

したがって、純粋なモデル コードと、明らかな WPF ウィンドウおよびエディター コードの両方が含まれています。読む際は「アルゴリズムパラメータの定義だけ」と誤解しないでください。

## 現時点で最初に知っておく価値のあるファイル

### コア入口

- `ITemplate.cs`: テンプレート抽象化機能への主要な入り口
- `ModelBase.cs`、`ParamModBase.cs`: テンプレート モデルとパラメトリック モデルの基本タイプ
- `TemplateContorl.cs`: テンプレート登録センターと読み込み入り口

### 管理および編集 UI

- `TemplateManagerWindow.xaml(.cs)`: テンプレート管理ウィンドウ
- `TemplateEditorWindow.xaml(.cs)`: テンプレート編集ウィンドウ
- `TemplateCreate.xaml(.cs)`: テンプレート作成エントリ
- `TemplateSettingEdit.xaml(.cs)`: テンプレート構成編集の入り口

### 検索とサンプル

- `TemplateSearchProvider.cs`
- `TemplateSampleLibrary.cs`
- `TemplateSampleSaveWindow.xaml(.cs)`

「テンプレートがどのように検出され、どのように開かれ、どのように編集されるか」を知りたいだけの場合は、通常、アルゴリズムのサブディレクトリに直接アクセスするよりも、最初にこれらのファイルを確認する方が効率的です。

## カレントディレクトリの分割方法

倉庫の現在の状況から判断すると、このディレクトリは少なくともいくつかのカテゴリに分類できます。

### 1. コアフレームワーク層

この部分は、テンプレートの抽象化、登録、基本モデル、およびパブリック UI を担当します。

典型的な文書には次のようなものがあります。

- `ITemplate.cs`
- `ModelBase.cs`
- `ParamModBase.cs`
- `TemplateContorl.cs`
- `TemplatesExtension.cs`

### 2. プロセステンプレート層

`Flow/` は、プロセス テンプレートとプロセスの編集と実行に関連する機能を引き受けます。一般的なアルゴリズムテンプレートと同じ大規模なシステム内にありますが、使用シナリオは明らかに異なります。

### 3. ビジネステンプレートファミリー

ビジネス テンプレート ディレクトリの複数のグループは、引き続き現在のウェアハウスで直接参照できます。次に例を示します。

- `ARVR/`
- `POI/`
- `Compliance/`
- `JND/`
- `Matching/`
- `FindLightArea/`
- `FocusPoints/`
- `ImageCropping/`
- `LedCheck/`
- `LEDStripDetection/`
- `Validate/`
- `DataLoad/`

これらのディレクトリは、統一されたルールに従って完全に分割されているわけではありません。アルゴリズム ドメインごとにグループ化されているもの、処理リンクごとにグループ化されているもの、および歴史的な進化から残された機能パッケージに近いものもあります。

### 4. JSON テンプレート ファミリー

`Jsons/` は、最も誤解されている領域の 1 つです。通常、JSON 構成をコアとしてテンプレート実装のバッチを実行し、従来のディレクトリ テンプレートと共存させます。

名前が似ているがディレクトリが異なるテンプレート実装を見つけた場合、それらが重複したコードであるとは考えないでください。過去のバージョン、構成方法、またはビジネス アクセス方法が異なる可能性が高くなります。

## 現在の実行時にテンプレートをシステムに接続するにはどうすればよいですか?

テンプレート システムへの最も重要な現在のランタイム リンクは次のとおりです。

1. メイン プログラムとプラグインは、最初にアセンブリを読み込みます。
2. `TemplateContorl` は、データベース接続が利用可能になった後、現在ロードされているアセンブリをスキャンします。
3. `IITemplateLoad` を実装する非抽象型を探します。
4. これらのタイプは、テンプレートを `ITemplateNames` から `Load()` に登録します。
5. テンプレート管理画面、編集画面、処理画面、業務機能を使用して、登録したテンプレートを利用します。

このリンクは、2 つの実際的な制約を示しています。

- テンプレートは静的マスターテーブルで手動で宣言されません。
- テンプレートが表示されるかどうかは、アセンブリのロードとデータベースの可用性ステータスに影響されます。

## このモジュールを読むときによくある誤解

### 誤解 1: 純粋なアルゴリズム層として扱う

また、ウィンドウ、エディター、検索、サンプルの保存、テンプレートの作成、プロセス関連の UI も含まれています。これは、単純なアルゴリズム パラメーター定義ウェアハウスではありません。

### 誤解 2: すべてのディレクトリが同時に同じルールに従って設計されていると考えている

いいえ、現在のディレクトリ構造には明らかな進化の痕跡があり、完全に一貫した階層モデルを自然に満たすことは期待できません。

### 誤解3: テンプレートが見つからない場合は、まずエディタを確認してください

テンプレートの問題の多くは、登録チェーンの早い段階で発生します。アセンブリが読み込まれていない、データベースが接続されていない、`IITemplateLoad` が実行されていない、またはテンプレート名が繰り返し上書きされているなどです。

## 今すぐ読み続けたい場合は

推奨される順序は次のとおりです。

1. まず `TemplateContorl.cs` を読んで、テンプレートがどのように検出され登録されるかを理解してください。
2. `ITemplate.cs`、`ModelBase.cs`、および `ParamModBase.cs` をもう一度見て、テンプレート オブジェクト自体がどのようなものかを理解します。
3. 次に、`TemplateManagerWindow` と `TemplateEditorWindow` を見て、ユーザーがテンプレートをどのように操作するかを理解します。
4. 最後に、`ARVR/`、`POI/`、`Flow/` などの特定のビジネス ディレクトリを入力します。

これにより、最初からテンプレートのサブディレクトリを詳しく調べるよりも、全体的な理解を容易に確立できます。
