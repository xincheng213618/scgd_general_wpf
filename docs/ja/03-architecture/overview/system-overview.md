# システムアーキテクチャの概要

このページでは、一連の抽象レイヤーを使用してウェアハウス全体を標準的な教科書として説明することはもう行いません。代わりに、現在のコード ウェアハウスのメイン ディレクトリに従ってシステムがどのように編成されているか、およびコードを読み取るときにどこから始めるべきかを直接説明します。

## まずこの倉庫を理解するにはどうすればよいでしょうか?

現在のディレクトリ構造から判断すると、ColorVision は、デスクトップのメイン プログラムをコアとして、エンジン、UI、プラグイン、プロジェクト パッケージ、インストールおよび更新システムを中心に展開する Windows WPF プラットフォームに近いものです。

最も重要なトップレベルの領域は次のとおりです。

- `ColorVision/`: メインアプリケーションの入り口とメインウィンドウ
- `UI/`: WPF UI フレームワーク、テーマ、プロパティ エディター、イメージ エディター、データベースおよびデスクトップ メニューなど。
- `Engine/`: デバイス サービス、テンプレート システム、プロセス実行、OpenCV 統合、ファイル処理
- `Plugins/`: ランタイムプラグイン拡張機能
- `Projects/`: 顧客プロジェクトとカスタマイズされたビジネス ポートフォリオ
- `ColorVisionSetup/`: インストーラーおよびアップデート関連プログラム
- `Web/Backend/`: プラグイン マーケット バックエンド
- `Scripts/`: スクリプトのビルド、パッケージ化、リリース

## システムの役割ごとに構造を表示する

### メインプログラム層

`ColorVision/` はデスクトップ アプリケーションの入り口であり、メイン ウィンドウ、アプリケーションの起動、グローバル構成、更新の入り口、およびワークベンチ全体の構成を担当します。

「プログラムの開始後に最初に何が起こるか」を追いかけている場合は、通常はここから始めて、`UI/` と `Engine/` を組み合わせて見ていきます。

### UI レイヤー

`UI/` は単一のプロジェクトではなく、インターフェイス関連のモジュールの集合です。現在の主要なものには次のようなものがあります。

- `ColorVision.UI/`: メニュー、パネル、プロパティ エディターなどのユニバーサル UI フレームワークと機能。
- `ColorVision.Themes/`: テーマ別およびビジュアル リソース
- `ColorVision.ImageEditor/`: 画像の表示、注釈、および結果の表示
- `ColorVision.Database/`: データベース ブラウザなどのデータベース関連の UI 機能
- `ColorVision.UI.Desktop/`: デスクトップレベルのメニューと設定エントリ

### エンジン層

`Engine/` はシステムのビジネス コアですが、単一プロジェクトの名前空間ではありません。現在、それは主にいくつかの部分で構成されています。

- `ColorVision.Engine/`: デバイス サービス、テンプレート システム、プロセス ウィンドウ、MQTT、およびビジネス調整
- `FlowEngineLib/`: プロセスノードの編集および実行ベース
- `cvColorVision/`: 低レベルのビジョン処理と OpenCV 関連の統合
- `ColorVision.FileIO/`: ファイルの読み書き処理
- `ColorVision.ShellExtension/`: 外部統合関連の拡張機能

### プラグインとプロジェクト層

- `Plugins/` は、Conscope、Spectrum、SystemMonitor などのランタイム プラグイン拡張機能を提供します。
- `Projects/` 通常、既存のエンジンと UI 機能を特定のソリューションに再結合することによって、顧客プロジェクトまたはビジネス パッケージを実装します。

### 配信層と周辺層

- `ColorVisionSetup/` はサイド プログラムのインストールと更新を担当します
- `Web/Backend/` はプラグイン マーケットのバックエンドを担当します
- `Scripts/` およびルート バッチ スクリプトは、ビルド、パッケージ化、公開を担当します。

## 実行時に最も一般的なメインリンク

ユーザー操作を徹底的に調べてみると、通常、最も一般的なリンクは次のとおりです。

1. ユーザーは `ColorVision/` のメイン ウィンドウから関数を入力します。
2. `UI/` の対応するウィンドウまたはパネルは、表示と対話を担当します。
3. `Engine/ColorVision.Engine/` のデバイス サービス、テンプレート、またはプロセス ロジックがビジネス処理を引き継ぎます。
4. プロセスの実行が必要な場合は、さらに `Engine/FlowEngineLib/` を呼び出します。
5. 画像またはアルゴリズムの処理が必要な場合は、引き続き `Engine/cvColorVision/`、`UI/ColorVision.ImageEditor/`、または特定のテンプレート実装をリンクします。
6. 関数が外部拡張からのものである場合は、実装を `Plugins/` または `Projects/` に入力します。

## このページはそれ以上何もしません

このページでは、これらの簡単に歪められるコンテンツは今後維持されません。

- 架空の標準化された 6 層アーキテクチャの命名
- 現在のディレクトリと一致しないモジュール名のリスト
- 「依存性注入コンテナ」、「オブジェクトプール」、「レポートテンプレート」などを教科書風にまとめて一般化。

トピックでより詳細な実行時関係、プロセス実行チェーン、またはテンプレート構造が必要な場合は、ここですべてを一度に説明するのではなく、対応するトピック ページに移動して説明する必要があります。
