# アーキテクチャランタイム

このページでは、現在のコードで確認できるメイン プログラムのランタイム リンクのみを説明しており、英語のドラフト メタデータと一般的な起動シーケンス図は維持されなくなります。

## 最初にランタイムを理解する方法

デスクトップ プログラムの現在のランタイムは、「すべてのモジュールを一度に初期化してからメイン インターフェイスを表示する」という単一のモデルではなく、実際にはいくつかのブランチに分割されています。

- コマンドラインはファイル処理のみを実行し、直接戻ります。
- 通常のデスクトップ起動とウィザードまたは起動ウィンドウの開始
- 最後の異常終了後、起動を続行する前にプラグインを無効にするかどうかを確認します。

## メインプログラムがリンクを開始します

`ColorVision/App.xaml.cs` の現在の実装から判断すると、一般的な起動シーケンスは大まかに次のとおりです。

1. 作業ディレクトリを設定し、必要に応じて重要な DLL をプリロードします。
2. 構成、ログ、テーマ、言語を初期化します。
3. コマンドラインパラメータを解析します。
4. `input` や `export` などのファイル処理ブランチの場合は、直接処理して戻ります。
5. 単一インスタンスのチェックを実行し、必要に応じてコマンド ライン パラメータを実行中のインスタンスに転送します。
6. 非デバッグ状態のゾンビプロセスをクリーンアップします。
7. 前回の起動ステータスに基づいて、プラグインを無効にするかどうかを決定します。
8. WinForms のビジュアル スタイルを初期化します。
9. ウィザードの完了ステータスに応じて、`WizardWindow` または `StartWindow` が表示されます。

ここで最も重要なことは、すべての手順を覚えておくことではなく、起動すると必ずしもメイン ウィンドウが直接表示されるわけではないことを知っておくことです。

## プラグインはいつ導入されますか?

現在のメイン プログラムは、ウィザードまたは起動ウィンドウに入る前にプラグインをロードするかどうかを決定します。

プラグインのロードに関する重要なポイントは次のとおりです。

- `Plugins/` ディレクトリをスキャンします
- 各プラグインディレクトリの`manifest.json`を読み込みます
- オプションの読み取り `.deps.json`
- `ColorVision.*` 依存関係のバージョンを確認する
- 最後に `Assembly.LoadFrom(...)` を使用してプラグイン アセンブリをロードします

前回異常終了した場合は、起動時にプラグインを無効にするかどうかを尋ねられます。これは、プラグインがこの実行に参加するかどうかは明確な分岐であり、常に無条件にロードされるわけではないことを意味します。

## メイン ワークスペースに入った後、どのランタイム オブジェクトが最も重要であるか

### サービスツリー

`ServiceManager` は、データベース接続が利用可能になった後にサービス ツリーをロードし、リソース テーブル内のデータを次のように編成します。

- `TypeServices`
- `TerminalServices`
- `DeviceServices`
- `GroupResources`

次に、プロセス表示領域と各デバイスの表示コントロールを統合表示マネージャーにインストールします。

### 登録センターとサービス トークン

`MQTTRCService` は以下を担当します。

- 登録センターとの接続状態を維持する
- 現在利用可能なサービス トークンをクエリする
- サービスステータスの更新
- サービスステータスをデバイスサービスオブジェクトに同期します

したがって、「プロセスが実行できない」や「デバイスはオンラインだがステータスが更新されない」などの多くの問題は、本質的に UI の問題ではなく、ここでのランタイム ステータスが準備ができていないことを意味します。

### テンプレートの登録

テンプレート システムは、手動でハードコーディングされた登録ではありません。現在、`TemplateControl` は、`IITemplateLoad` 実装のロードされたアセンブリをスキャンし、データベース接続が利用可能になると、その `Load()` を呼び出してテンプレートの登録を完了します。

これは、テンプレートが表示され、編集できるかどうかは、次の 2 つの前提条件によって決まることを意味します。

- 関連するアセンブリがロードされています
- データベース接続が確立されました

## 実行時にプロセス実行を接続する方法

ユーザーがプロセス ウィンドウに入ると、ランタイム メイン チェーンは引き続き次のように拡張されます。

- `DisplayFlow` は、現在のプロセス テンプレートの更新、プロセスの開始または停止を担当します。
- `FlowControl` は実行中のプロセスの開始と停止を担当します
- `FlowEngineLib` は特定のノードの実行を担当します
- `MQTTRCService` は、プロセスの実行に必要なサービス トークンとステータスの更新を提供します

実行中、ランタイムは更新を続けます。

- 現在実行中のノード
- 実行ログテキスト
- バッチの進行状況
- ノードレコードとメッセージレコード

## 実行時に最も一般的な障害ポイント

### スタートアップフェーズ

- プラグインの依存関係が満たされていない
- 最後の異常終了により、プラグインを無効にするかどうかを手動で決定する必要がありました。
- プログラムが開始し続けないと誤って認識され、コマンド ライン ブランチが早期に戻ります。

### サービス準備フェーズ

- データベースが接続されていないため、サービス ツリーまたはテンプレートが正しく読み込まれません。
- 登録センターまたは MQTT 側の準備ができていないため、サービス トークンが空になります

### 実行フェーズ

- プロセス テンプレートが選択されていないか、開始ノードがありません
- デバイス サービスは存在しますが、ランタイム状態が実行可能状態と同期されていません。
