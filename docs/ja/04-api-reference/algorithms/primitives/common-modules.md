# 一般的なアルゴリズムモジュール

このページでは、現在のリポジトリをスタンドアロンの「ユニバーサル アルゴリズム プラットフォーム」として説明することはなくなりました。ソース コードの現在のステータスによれば、共有アルゴリズム コンポーネントのナビゲーション ページとしてより適しています。

## さて、このページでは実際に何を取り上げるべきでしょうか?

複数のアルゴリズム、テンプレート、フロー ノードによって繰り返し再利用される現在のウェアハウス内の共通コンポーネントは、主に次のグループに集中しています。

- ROI / 発光領域の位置決め
- POI ポイント セットと関連テンプレート
- マッチングテンプレートマッチング
- JSON 形式のトリミングまたはエッジ検出テンプレート

これらのコンポーネントに共通しているのは、「純粋なアルゴリズム コア」ではありませんが、すべて次の特徴を持っています。

- テンプレート編集の入り口
- アルゴリズムホストを表示
- MQTTコマンドのパッケージ化
- 場合によっては結果処理またはフローアクセスを伴う

したがって、このページの機能は、存在しない統一フレームワークを抽象化し続けるというよりは、「共有コンポーネントのどのセットをどこで確認するか」に似ています。

## 現時点で最も優先すべきもの

### ROI

従来の ROI は主に `Templates/FindLightArea` に当てはまります。

- `TemplateRoi`
- `RoiParam`
- `AlgorithmRoi`
- `DisplayRoi`

これは現在、グローバルな汎用 ROI SDK ではなく、Luminous Regional Positioning Template Chain を表しています。

さらに、ROI エントリの JSON バージョンもあります。

- `TemplateImageROI`
- `TemplateSFRFindROI`

ROI を求めるトリミングまたは ARVR を検討している場合、これら 2 つは従来の `TemplateRoi` よりも現状に近いものになります。

### POI

POI は現在、最も一般的な共有プリミティブです。少なくとも次の内容がカバーされます。

- `TemplatePoi`
- `PoiParam`
- `PoiPoint`
- `AlgorithmPOI`
- `AlgorithmBuildPoi`
- 関連するテンプレートのフィルター、修正、調整、および出力

また、JSON アルゴリズムとフロー ノードによって引き続き参照されるため、POI は単一のアルゴリズム ページではなく、モジュール間のデータ構造になります。

### マッチング

現在、マッチングも完全ではありますが、比較的合理化された共有チェーンです。

- `TemplateMatch`
- `MatchParam`
- `AlgorithmMatching`
- `DisplayMatching`
- `ViewHandleMatching`

`AlgorithmMatching` 現在:

- `TemplateMatch` を開きます
- オプションの開口部 `TemplatePoi`
- `TemplateFile` の指定を許可します
- `Event_MatchTemplate` をリリース

したがって、Matching は単なる計算機能ではなく、完全なテンプレートとコマンド ホストです。

## これらの共有モジュールをシステムに組み込むにはどうすればよいでしょうか?

現在の実装によれば、これらは通常、同じ動作モードに従います。

1. `TemplateEditorWindow` またはカスタム編集コントロールを使用してテンプレートを管理します。
2. `DisplayAlgorithmBase` 派生クラスを通じて UI とコマンドを公開します。
3. `CVTemplateParam` とその他の入力パラメータをアルゴリズム クラスにアセンブルします。
4. MQTT イベントを通じてサービス側にリクエストを送信します。
5. 結果プロセッサまたはフロー ノードは、必要に応じて消費を続けます。

現在の実装では UI、テンプレート、コマンド ホストが統合されているため、これらのモジュールを単に「アルゴリズム ライブラリ」として記述すると歪んでしまうのはこのためです。

## 必要に応じて今すぐソースコードを読みたい場合

### 領域選択やトリミングを確認したい

最初に [ROI](./roi.md) を読みます。

### ポイント セット テンプレート、ポイント セットの構築、または POI の再利用を確認したい

最初に [POI](./poi.md) と [POI テンプレート](../templates/poi-template.md) を読んでください。

### 画像テンプレートマッチングを見たい

最初に `Engine/ColorVision.Engine/Templates/Matching/AlgorithmMatching.cs` と `TemplateMatch.cs` をお読みください。

### これらのコンポーネントがプロセスにどのように配置されるかを確認したい

最初に [プロセス エンジン](../templates/flow-engine.md) と `Templates/Flow/NodeConfigurator` を読み取ります。

## 現在、最もよくある間違いのいくつかが犯されています

### ユニバーサルは独立したフレームワークと同等ではありません

これらの共有モジュールは、個別にリリースされたパブリック SDK を形成しませんが、`ColorVision.Engine/Templates` の下で配布され、メイン プログラム ホストによって均一にホストされます。

### 共有モジュールは純粋ではありません

これらは通常、テンプレート、UI、MQTT メッセージ、結果表示の組み合わせです。厳格な 3 層アーキテクチャに従って書き続けると、現状とのズレが生じやすくなります。

### POI、ROI、およびマッチングの間には相互参照があります

たとえば、マッチングは引き続き `TemplatePoi` を開くことができ、ARVR の `SFR_FindROI` は POI テンプレートを要求します。これらのモジュールは互いに完全に独立したアイランドではありません。
