#ColorVision.FileIO

このページでは、現在のウェアハウスで実際に利用可能な `ColorVision.FileIO` モジュールのみを説明しており、「ユニバーサル ファイル フレームワーク + JSON/YAML/バッチ処理プラットフォーム」の古いドラフトは維持されなくなります。

## まず、このモジュールが現在どのようなものであるかを見てみましょう

現在のソース コードのステータスによると、`ColorVision.FileIO` は一般化されたファイル処理フレームワークではなく、ColorVision のカスタム イメージ ファイル形式を中心に編成された専用の I/O モジュールです。現時点で最も明確な着地点は次のとおりです。

- `CVCIE` / `CVRAW` / `CVSRC` クラス ファイルを識別して読み取ります。
- ファイルヘッダー、バージョン、露出、サイズ、チャンネル、生データ領域を解析します。
- チャンネルまたはファイル タイプごとにローカル ColorVision 画像ファイルを開きます。
- `CVCIEFile` をコア データ キャリアとして使用します。

したがって、古いドキュメントで説明されている「汎用 JSON/XML/YAML/画像バッチ処理のサポート」よりもはるかに範囲が狭く、現在の実際のコードに近いものになります。

## 現時点で最も重要なファイル

- `Engine/ColorVision.FileIO/CVFileUtil.cs`
- `Engine/ColorVision.FileIO/CVCIEFile.cs`

少なくとも現在のモジュールの読み取り可能な実装から判断すると、これら 2 つの場所でコア フォーマットの識別、ヘッダーの解析、およびデータ キャリアの定義がカバーされています。

## 現在処理されている形式は何ですか?

`CVCIEFile.cs` 現在定義されている `CVType` には次のものが含まれます。

- `Raw`
- `Src`
- `CIE`
- `Calibration`
- `Tif`
- `Dat`

しかし、`CVFileUtil` の現在最も集中的な実装から判断すると、実際に焦点を当てているのは、一般的なオフィス ファイル形式や構成ファイル形式ではなく、ColorVision カスタム イメージ ファイル `CVCIE` / `CVRAW` のセットです。

## 現在の読み取りチェーンの仕組み

`CVFileUtil` 現在のコアプロセスはおおよそ次のとおりです。

1. ファイル ヘッダーを使用して、それが `CVCIE` システム ファイルかどうかを判断します。
2. ヘッダーとバージョンを読み取ります。
3. さまざまなバージョンに応じて、ファイル名、ゲイン、チャンネル番号、露出、サイズなどのメタデータを解析します。
4. 次に、データ領域のオフセットに従って元のバイト ブロックを読み取ります。
5. この情報を `CVCIEFile` にまとめます。

現在、次の両方をサポートしています。

- ファイルパスから読み取る
- バイト配列からの読み取り

そして、多くの詳細は「範囲外、例外、OOM、無効なヘッダーの防止」を中心に展開されます。これらは一般的な形式解析コードであり、一般的なファイル サービス層ではありません。

## `CVCIEFile` あなたの現在の役割は何ですか?

`CVCIEFile` はモジュール内のコア データ構造となり、以下を運ぶ役割を果たします。

- ファイルバージョン
- ファイルの種類
- ランクとビット深度
- チャンネル数
- ゲインおよび露出アレイ
- ソースファイル名
- 生データバイト
- ファイルパス

また、それ自体も `IDisposable` を実装しており、解放時にデータ配列の大きなブロックを積極的にクリアします。これは、現在のモジュールが軽量のテキスト構成ではなく、主に大容量の画像データに直面していることを再度示しています。

## 現在、どのような実用的な入り口が利用可能ですか?

`CVFileUtil` の現在の実装から判断すると、より重要なエントリには次のものがあります。

- `IsCIEFile(...)`
- `IsCVCIEFile(...)`
- `ReadCIEFileHeader(...)`
- `ReadCIEFileData(...)`
- `Read(...)`
- `OpenLocalFileChannel(...)`
- `OpenLocalCVFile(...)`

これらの入り口は基本的に次の 2 つの目標に焦点を当てています。

- ファイルが ColorVision 独自の形式であるかどうかを確認します
- 独自の形式をメモリ オブジェクトに安全に解析します

## 現在、最もよくある間違いのいくつかが犯されています

### これは一般化されたファイル システム プラットフォームではありません

現在のソース コードには、古いドキュメントにあるユニバーサル `FileIOManager`、JSON プロセッサ、YAML プロセッサ、バッチ エグゼキュータなどの完全なシステムが実装されていません。その書き方を使い続けると、存在しない抽象化レイヤーが事実になってしまいます。

### コア オブジェクトはバイナリ イメージ キャリアであり、テキスト構成モデルではありません

`CVCIEFile` は現在、主に画像メタデータと生のバイト配列の大きなブロックを運びます。これは、構成、圧縮、シリアル化サービスに焦点を当てた古いドラフトの説明とはまったく異なります。

### バージョン ブランチが実装の焦点です

`ReadCIEFileHeader(...)` さまざまなバージョンの分岐解決は、現在の複雑さの実際の原因の 1 つです。文書に「カスタム形式ファイルを読み取る」とのみ記載されている場合、この詳細レイヤーは消去されます。

### セキュリティは主に防御的な読みに反映されます

現在のコードは、いわゆる「ホット更新非同期ファイル フレームワーク」を実装する代わりに、ファイルの長さ、オフセット、配列割り当て、および例外を厳しくチェックします。これを理解することによってのみ、コードがヘッダー/データの解析に焦点を当てている理由を理解することができます。

## 推奨される読む順序

1. `Engine/ColorVision.FileIO/CVCIEFile.cs`
2. `Engine/ColorVision.FileIO/CVFileUtil.cs`

最初に古い文書を調べるよりも、最初にデータキャリアを調べてから、特定の解析ロジックを調べる方がはるかに効果的です。
