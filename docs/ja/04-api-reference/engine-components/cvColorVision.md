#cvColorVision

このページでは、現在のウェアハウスで実際に利用可能な `cvColorVision` モジュールのみを説明し、「機能のプロモーション + 多数の架空のサンプル + 純粋にマネージド アルゴリズム ライブラリ」という古いドラフトは維持されなくなります。

## まず、このモジュールが現在どのようなものであるかを見てみましょう

現在のソース コードの状況によると、`cvColorVision` は主に C# に依存してビジネス アルゴリズムを実装するモジュールではなく、ネイティブ相互運用性ブリッジの厚い層です。現在の中核的な役割は次のとおりです。

- `cvCamera.dll` および `cvOled.dll` の機能を `DllImport` を通じて C# に公開します。
- カメラ、カラー、グラフィックス カード、ソース テーブル、OLED アルゴリズムなどの低レベル インターフェイスを統一された名前空間に集約します。
- `ColorVision.Engine`、プラグイン、およびデバイス サービスに薄いラッパー呼び出しサーフェスを提供します。

したがって、古いドキュメントのような純粋にマネージドなビジュアル フレームワークよりも、「ネイティブ機能バインディング レイヤー」に近いものになります。

## 現時点で最も重要なファイル

- `Engine/cvColorVision/cvCameraCSLib.cs`
- `Engine/cvColorVision/ConvertXYZ.cs`
- `Engine/cvColorVision/CvOledDLL.cs`
- `Engine/cvColorVision/PG.cs`
- `Engine/cvColorVision/PassSx.cs`
- `Engine/cvColorVision/Algorithms.cs`

モジュールが基礎となる DLL にどのように接続するか、および現在公開されている機能を知りたいだけの場合、これらのコードはすでに本体をカバーしています。

## 現在のコントロール プレーンをブロックに分割する方法

### カメラと一般的な視覚インターフェイス

`cvCameraCSLib.cs` は、現在最大のバインディング サーフェスです。コードを見ると、カメラ スイッチだけでなく、次のようなネイティブ エントランスの大規模なコレクションもカバーされています。

- カメラのオン、オフ、リアルタイム プレビュー、フレーム キャプチャ
- JSON の読み取りと書き込みを構成する
- 自動露出、ROI、コールバック登録
- XYZ/xy/uv/CCT/波形サンプリング
- TIFF エクスポートとデータの分割/結合
- オートフォーカス、レンズ位置、キヤノン関連のコントロール
- 多彩な視覚検出・画像処理機能

したがって、これはわずか数十個のカメラ API を備えた小さなパッケージではなく、現在最も密度の高い P/Invoke コンバージェンス ポイントです。

### カラーとクロマのサンプリング

`ConvertXYZ.cs` は、`cvCamera.dll` の XYZ 関連の入り口をさらに焦点を絞ったバインディング サーフェスに分割し、現在次の点に焦点を当てています。

- XYZバッファの初期化と解放
- 円/長方形領域のサンプリング
- xyz、uv、CCT、主波長などをエクスポートします。
- バッチポイントサンプリング

これは、現在のカラー サンプリング チェーンがスタンドアロンの C# 計算機ではなく、ネイティブのバッファリング関数とサンプリング関数を中心に実行されていることを示しています。

### OLED 固有のアルゴリズム

`CvOledDLL.cs` は現在、特に `cvOled.dll` にバインドされており、以下を提供します。

- パラメータの読み込み
- 画像の読み込み
- ピクセル検索
- ピクセル再構成
- モアレフィルター

したがって、OLED 関連の機能は現在、カメラ インターフェイス内で混合されるのではなく、別個の DLL に実装されています。

### グラフィックカードと周辺機器インターフェイス

`PG.cs` は現在、チャート カード デバイス制御用のシン ラッパーであり、以下を提供します。

- PGの初期化
- TCP/シリアル接続
- スタート/ストップ/リセット
・上下切り替えと指定フレーム切り替え

`PassSx.cs` は、ソース テーブル/電源側でネイティブ呼び出しパッケージを提供し、以下をカバーします。

- デバイスの電源をオン/オフにする
- ソースモードの設定
- フロントポートとリアポートを備えた 2 線式/4 線式のセットアップ
- 電圧と電流を読み取る
- ステッピングとスキャンの実行

これは、`cvColorVision` が現在画像処理を処理するだけでなく、複数のタイプの周辺機器の基礎となるバインディングも引き受けていることを示しています。

### 非常に薄いアルゴリズムのエントリ

`Algorithms.cs` この種のファイルは、モジュールの別の特性を示しています。一部のパッケージは非常に薄く、基礎となる関数を 1 つだけ最も直接的な形式で公開しています。

したがって、この層の責任は、すべての API スタイルを均一に設計することではなく、基礎となる機能を可能な限り完全にマッピングすることです。

## 現在、最もよくある間違いのいくつかが犯されています

### これは純粋な C# アルゴリズム センターではありません

現在、主要な機能のほとんどはネイティブ DLL から提供されており、C# コードは主に宣言、少量の補助的なパッケージ化、およびデータ型のブリッジングを担当します。 「主要なアルゴリズムはホスティング層に実装される」と書き続けると、実際のコード構造に反します。

### `cvCameraCSLib` はカメラだけではありません

ファイル名は誤解されやすいものですが、現時点では実際には多くのカラー サンプリング、画像処理、オートフォーカス、検出機能が公開されており、合計バインディング エントリの 1 つです。

### ここでのインターフェイスの粒度は均一ではありません。

`cvCameraCSLib.cs` のような一部のファイルは非常に厚く、`Algorithms.cs`、`PG.cs`、`CvOledDLL.cs` のような一部のファイルは非常に薄いです。ドキュメントは、きちんとした均一な階層化された API システムにハードライトされるべきではありません。

### これは、「上位層によって呼び出される」ベース層に似ています。

現在、`ColorVision.Engine`、デバイス サービス、および一部のプラグインは、ここで公開されているネイティブ インターフェイスを呼び出します。 `cvColorVision` 自体は、ホストレベルのウィンドウ、テンプレート、またはワークフロー オーケストレーションを担当しません。

## 推奨される読む順序

1. `Engine/cvColorVision/cvCameraCSLib.cs`
2. `Engine/cvColorVision/ConvertXYZ.cs`
3. `Engine/cvColorVision/CvOledDLL.cs`
4. `Engine/cvColorVision/PG.cs`
5. `Engine/cvColorVision/PassSx.cs`

このようにして、最初に最も厚い合計結合面を確認し、次に OLED、グラフィックス カード、ソース メーターなどの専用インターフェイスに拡張することができます。
