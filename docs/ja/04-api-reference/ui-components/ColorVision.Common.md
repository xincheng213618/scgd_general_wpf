#ColorVision.Common

このページでは、現在 UI/ColorVision.Common によって実行されている共有の基本機能についてのみ説明し、古いドキュメントの「パブリック SDK インターフェイスの大規模かつ包括的なコレクション」については継続しません。

## モジュールの配置

ColorVision.Common は現在、UI レイヤーの共有ベース ライブラリであり、主に次のコンテンツを提供します。

- MVVMの基本タイプ
- コマンドのカプセル化
- 共通のインターフェースとメタデータモデル
- 大まかな権限制御
- Windows ネイティブ メソッドのカプセル化
- よく使用されるツール

これは、独立して実行されるビジネス モジュールというよりは、モジュール間で再利用される基本的な構成要素のセットに似ています。

## 現在最も重要なディレクトリ

プロジェクト ディレクトリを見て、最初に知っておくべき最も重要な点は次のとおりです。

- MVVM/: `ViewModelBase`、`RelayCommand` およびその他の基本タイプ
- インターフェース/: 構成、メニュー、ステータスバー、イニシャライザ、ビューなどの共有インターフェース。
- 権限/: `Authorization`、`AccessControl`、`PermissionMode`
- NativeMethods/: Windows API ラッパー
- ユーティリティ/: ファイル、コレクション、ウィンドウなどの一般的なツール。
- 入力/: 入力関連の機能
- ThirdPartyApps/: サードパーティアプリケーションのアクセスに関する定義

## キー入力タイプ

### ビューモデルベース

`ViewModelBase` は現在、最も基本的なバインド可能なオブジェクト基本クラスであり、`INotifyPropertyChanged` を実装し、多数の構成クラス、マネージャー、およびビュー モデルによって継承されます。

### RelayCommand とコマンド

現在、コマンド層には主に 2 つの共通の入り口があります。

- `RelayCommand` / `RelayCommand<T>`: ユニバーサル コマンドのカプセル化
- `Commands`: 少数のグローバル `RoutedUICommand`

古いドキュメントでは、コマンド システムは独立したフレームワークの完全なセットとして記述されていましたが、現在のコードから判断すると、実際に頻繁に使用されるのは `RelayCommand` です。

### インターフェース/

`Interfaces/` は、完全なビジネス実装ではなく、共有境界の定義を担当します。現在一般的なインターフェイスのグループには次のものがあります。

- `IConfig`、`IConfigSettingProvider`
- `IInitializer`、`InitializerBase`
- `IMenuItemProvider`
- `IStatusBarProvider`、`IStatusBarProviderUpdatable`
- `View` や `IViewManager` などの関連タイプを表示します。

これらのタイプのほとんどは最小限のコントラクトのみを定義し、実際の登録、検出、および実行ロジックは通常、上位レベルのモジュールにあります。

### ステータスバーメタ

`StatusBarMeta` は、古いドキュメントのようなアイコンとコマンドのみを備えた簡素化されたモデルではありません。現在、すでに以下をホストしています。

- 一意の識別名と名前
- 説明テキスト
- 左右の整列と並べ替え
- `Command` または `Popup` の 2 種類のアクション
- バインディング元オブジェクト
- アイコンリソースまたは直接アイコンコンテンツ
- ターゲット ウィンドウのスコープとデフォルトの可視性

したがって、これは単なる軽量 DTO ではなく、すでに UI ステータス バー システムの中核となるメタデータです。

### 認可 / アクセス制御 / 許可モード

`Authorizations/` で提供されるのは、現在の一般的な粗粒度のアクセス許可制御です。

- `Authorization.Instance.PermissionMode`
- `AccessControl.Check(...)`
- `RequiresPermissionAttribute`

ここでは境界に特に注意してください。

- 共通レイヤーは、グローバルな粗粒度の権限モードのみを提供します
- `UI/ColorVision.Solution/Rbac` のきめ細かいローカル RBAC

Common の権限システムをプロジェクト全体の唯一の完全な RBAC として記述しないでください。

## 現在の実装はどのようなものですか?

ColorVision.Common は現在、外部ユーザーにリリースされる安定したパブリック フレームワークというよりも、「共有プロトコル層 + 基本ツール層」に近いものです。多くのインターフェイスには共通の名前がありますが、その実際の機能は、ウェアハウス内の UI モジュールに統一されたコントラクトを提供することです。

たとえば:

- `IConfig` 自体は単なるマーカー インターフェイスです
- `InitializerBase` は、デフォルトの名前、順序付け、依存構造のみを提供します。
- `View` はインデックス、タイトル、アイコンを備えた共有 ViewModel であり、完全なビュー フレームではありません

## 現在の実装の境界

### 完全なプラグイン プラットフォームのドキュメントではありません

Common には多くの拡張インターフェイスが定義されていますが、実際のプラグインの検出、メニューの登録、設定の集約、およびステータス バーの更新はすべて上位モジュールの実装に分散されています。これは単なる共有コントラクトであり、統合ランタイム ハブとして記述するべきではありません。

### 全権を持つセンターではない

Common でのアクセス許可チェックは、グローバル モードのスイッチや粗粒度の制限には適していますが、ソリューション側のローカル RBAC と同等ではありません。

### 多くのインターフェースは「最終的な抽象化」ではなく「最小限の形状」です。

`IConfig` や `IInitializer` のようなインターフェイスは非常に軽量です。後で読むときは、インターフェイス定義自体に留まるのではなく、最初に実装側に従って実際の制御チェーンを確認する必要があります。

## このページはそれ以上何もしません

このページでは、次のような危険性の高いコンテンツは管理されなくなります。

- バージョン番号とパッケージのリリース情報の大きなセクション
- 理想的なパブリック SDK リスト
- すべてのインターフェースを完全なフレームワーク機能に拡張します
- Common Permission System をグローバルに一意な RBAC と誤って書きました
