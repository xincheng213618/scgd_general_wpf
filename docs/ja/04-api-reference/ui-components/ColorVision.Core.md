#ColorVision.Core

このページでは、現在 UI/ColorVision.Core によって実装されているネイティブ相互運用性レイヤーについてのみ説明します。 「High-Level Image API Manual」および古いドキュメントに存在しないマネージド メソッドの例は継続されなくなりました。

## モジュールの配置

ColorVision.Core は現在、ネイティブの画像およびビデオ機能のブリッジング レイヤーに近く、主に次の役割を果たします。

- マネージド/アンマネージドの境界を越える `HImage` のようなデータ構造を定義する
- P/Invoke 経由で `opencv_helper.dll`、`opencv_cuda.dll` を呼び出す
- WPF 側でビットマップ変換と更新支援を提供します。
- 偽色、画像強調、焦点評価、ビデオ関連などのネイティブの入り口を公開します。

これは、カプセル化された高レベルの画像処理フレームワークではありません。現在、多くの機能は依然として `extern` メソッド レベルでネイティブにエクスポートされたラッパーです。

## 現時点で最も重要なファイル

プロジェクト ディレクトリの中で、最初に読む価値のあるものは次のとおりです。

- `HImage.cs`: 画像データ構造体
- `HImageExtension.cs`: `HImage` と WPF 画像オブジェクト間のブリッジ
- `OpenCVMediaHelper.cs`: ネイティブ エクスポート ラッパーのメイン コレクション
- `OpenCVCuda.cs`: CUDA 関連のネイティブ入口
- `ColormapTypes.cs`: 擬似カラーの列挙
- `NativeLogBridge.cs`: ネイティブ ログ ブリッジング
- `nvcuda.cs`: CUDA関連のP/Invoke定義

## キー入力タイプ

### H画像

`HImage` は現在、古いドキュメントのような多数のインスタンス メソッドを備えたマネージ クラスではなく、ネイティブ イメージ バッファーを保持する構造です。その中核となる分野は次のとおりです。

- `rows`
- `cols`
- `channels`
- `depth`
- `stride`
- `pData`

同時に、`Marshal.AllocHGlobal` によって割り当てられた画像メモリを解放する役割を担う `Dispose()` を実装します。

これは、現在のモジュールの最も重要な役割の 1 つは、ネイティブ境界と管理境界を越えて画像バッファーを安全に渡すことであることを意味します。

### HImageExtension

`HImageExtension` は、完全な処理アルゴリズム ライブラリではなく、ブリッジ支援を提供します。現在は主に以下の業務を担当しています。

- チャネル数とビット深度に基づいて `PixelFormat` を導出する
- `HImage` の内容を `WriteableBitmap` にコピーします。
- 非同期ビットマップ更新パスを提供します
- ネイティブ画像データを WPF 表示可能なオブジェクトに変換するのを支援します

したがって、その値はアルゴリズム チェーンではなく、主に表示チェーンにあります。

### OpenCVMediaHelper

名前は `OpenCVMediaHelper` ですが、現在、ビデオ関連のインターフェイスだけでなく、次のような `opencv_helper.dll` のエクスポート パッケージが多数含まれています。

- 擬似カラーとオートレンジ擬似カラー
- 最小値/最大値の抽出
- 自動明るさ、自動カラー、自動色合い
- チャンネル抽出
- 輝度コントラスト、ガンマ、反転、閾値、シャープ化、フィルタリング、エッジ検出
- SFRと重点評価
- 複数の識別または検出入り口
- ビデオ関連の構造と機能

したがって、現在のより正確な理解は、これは単なる「ビデオ ヘルプ クラス」ではなく、メインのネイティブ イメージ機能エクスポート サーフェスであるということです。

### OpenCVCuda

`OpenCVCuda` は現在、古いドキュメントで主張されているユニバーサル CUDA デバイス管理レイヤーではありません。現在、次のような融合関連のエントリに焦点を当てた、少数の `opencv_cuda.dll` エクスポートが公開されています。

- `CM_Fusion`
- `CM_Fusion_Async`
- `CM_Fusion_Batch`

したがって、CUDA 機能を説明するときは、現在の実際のエクスポートに従って記述する必要があり、GPU 機能の完全な一般的な入り口に拡張すべきではありません。

### ColormapType と NativeLogBridge

- `ColormapTypes` は、統合疑似カラー マッピング列挙を担当します。
- `NativeLogBridge` は、ネイティブ側のログを管理対象ログ システムにブリッジする役割を果たします。

どちらのファイルも小さいですが、それぞれ擬似カラー チェーンとデバッグ チェーンの重要な境界点です。

## 現在のランタイムのメインチェーン

このモジュールのセットは現在、次のチェーンのように見えます。

1. 上位モジュールは、P/Invoke を通じて `OpenCVMediaHelper` または `OpenCVCuda` を呼び出します。
2. ネイティブ DLL は `HImage` を返すか、`HImage` 出力パラメータを書き込みます。
3. WPF 表示チェーンは、画像データを `WriteableBitmap` から `HImageExtension` まで更新します。
4. `ColorVision.ImageEditor` などの上位層モジュールは、これらのビットマップの周囲で対話、描画、表示を続けます。

## 現在の実装の境界は何ですか?

### 高レベルの OO API として記述しないでください

現在のコードには、古いドキュメントで記述されている次のような典型的な高レベル インターフェイスがありません。

- `HImage.Load(...)`
- `HImage.ToBitmapSource()`
- `OpenCVCuda.GetCudaDeviceCount()`
- `OpenCVCuda.IsCudaAvailable()`

これらの書き方では、存在しない管理パッケージを見つけてしまうという誤解を読者に与えてしまいます。

### HImage のリソース セマンティクスは非常に重要です

`HImage` は通常の管理オブジェクトではなく、アンマネージ ポインターと明示的な解放ロジックが含まれています。このモジュールについて議論する場合、「クラス設計」よりもメモリと所有権の境界の方が重要です。

### 上位レベルのビジネス セマンティクスはここにはありません

Core はネイティブ機能のブリッジングのみを担当し、ツールバー、インタラクション、または ImageEditor のようなドキュメント状態のオーケストレーションを担当しません。読むときは、それが下位レベルの能力ベースにすぎないことを明確にしてください。

## このモジュールの読み方は現在、より適切です

### 画像データ構造と表示ブリッジングを確認したい

まずはご覧ください:

- `HImage.cs`
- `HImageExtension.cs`

### ネイティブのエクスポート インターフェイスを確認したい

まずはご覧ください:

- `OpenCVMediaHelper.cs`
- `OpenCVCuda.cs`

### 疑似カラーやログボーダーを見たい

まずはご覧ください:

- `ColormapTypes.cs`
- `NativeLogBridge.cs`

## このページはそれ以上何もしません

このページでは、次のような危険性の高いコンテンツは管理されなくなります。

- 存在しないマネージド高レベルメソッドの例
- `OpenCVCuda` を完全なデバイス管理レイヤーとして書き込みます
- 大量のアップデートログとバージョンリスト
- Core を完全な上位レベルの画像処理フレームワークとして呼び出す
