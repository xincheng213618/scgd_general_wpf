#ColorVision.データベース

このページでは、現在 UI/ColorVision.Database に実装されているデータ アクセスおよびデータベース ブラウズ機能についてのみ説明します。旧テンプレートの「データベースチュートリアル＋サンプルフラグメント＋構築検証記録」の混在記述方法は維持されなくなります。

## モジュールの配置

`ColorVision.Database` には現在 2 種類の責任があります。

- ビジネスエンティティとDAOの基本的なデータアクセス層
- ランタイムメンテナンスのためのデータベースブラウザとプロバイダシステム

その中で、現在優先的に注目すべき主要なラインは、従来のエンティティ スキャン モードではなく、「データベース ファースト」のブラウザ チェーンです。

## 現在最も重要なディレクトリとファイル

プロジェクト ディレクトリを見て、最初に知っておくべき最も重要な点は次のとおりです。

- `DatabaseBrowserWindow.xaml(.cs)`: データベース ブラウザのメイン ウィンドウ
- `DatabaseBrowserProviderRegistry.cs`: プロバイダーの登録と遅延読み込みの入り口
- `IDatabaseBrowserProvider.cs`: ブラウザプロバイダ契約
- `DatabaseBrowserModels.cs`: ライブラリ、テーブル、列、ページング モデル
- `MySqlControl.cs`: MySQL の構成とプロバイダーの作成
- `BaseTableDao.cs`、`EntityBase.cs`、`ViewEntity.cs`: ビジネス エンティティのアクセス層の基本タイプ

## キー入力タイプ

### データベースブラウザウィンドウ

`DatabaseBrowserWindow` は、現在のデータベース メンテナンス エクスペリエンスへの主要な入り口です。それは次のことを担当します。

- データソース、ライブラリ、テーブルのツリー構造を表示します。
- 右側の `DataTable` を押して結果セットを参照します
- 検索、ページング、並べ替えをサポート
- 追加、更新、削除などの一般的なテーブルレベルの操作を実行します。

その重要な機能は、現在のブラウザーが UI を駆動するために C# エンティティ定義に依存しなくなったことです。代わりに、最初に実際のデータベース接続からライブラリ、テーブル、および列の情報を取得し、次に表示方法と書き戻し方法を決定します。

### データベースブラウザプロバイダレジストリ

`DatabaseBrowserProviderRegistry` は、ブラウズ可能なデータ ソースの統合管理を担当します。現在、デフォルトのプロバイダーを遅延ロードし、ブラウザーに公開します。

- MySQL のデフォルトプロバイダー
- 他の発信者自身が登録したプロバイダー

したがって、これは現在のデータベース ブラウザ システムのスケジューリング エントリ ポイントです。

### IDatabaseBrowserProvider

`IDatabaseBrowserProvider` は、データベース ブラウザにとって最も重要な抽象境界です。現在、実装では以下を提供する必要があります。

- ライブラリリスト
- テーブルリスト
- コラム情報
- ページングクエリ
- 挿入、更新、削除

したがって、このモジュールのコア拡張ポイントは「エンティティ クラスの追加」ではなく、「新しいプロバイダーの登録」です。

### MySqlコントロール

`MySqlControl` は現在、単なる接続構成オブジェクトではなく、次のことも前提としています。

- MySQL 設定の永続性
- 接続文字列の構造
- MySQL ブラウザプロバイダの作成

したがって、MySQL 関連のエントリは、`BaseTableDao<T>` だけを見るのではなく、それに沿って直接表示する必要があります。

### BaseTableDao/EntityBase/ViewEntity

これらのタイプは、現在のビジネス層エンティティ アクセスの基礎のままです。

- `IEntity` 統一 `Id`
- `EntityBase` は主キー マッピング基本クラスを提供します
- バインド可能なエンティティの場合は `ViewEntity`
- `BaseTableDao<T>` 既存のビジネス コードを引き続き提供します

しかし、これらはもはや現在のデータベース UI ブラウジング チェーンの唯一の中心ではありません。

## 現在のランタイムのメインチェーン

このモジュールのセットは現在、次のチェーンに近いものです。

1. `DatabaseBrowserWindow` は、`DatabaseBrowserProviderRegistry` から利用可能なプロバイダーを取得します。
2. プロバイダーは、ライブラリ、テーブル、および列の情報を返します。
3. ブラウザは、テーブル構造に従って `DataTable` の結果を動的に表示します。
4. データの追加、編集、削除は、プロバイダーのユニバーサル書き込みインターフェイスを通じてデータベースに返されます。
5. ビジネス コードの場合、エンティティと DAO システムは引き続き並行して使用できますが、ブラウザ UI は制御されなくなります。

## 現在の実装の境界は何ですか?

### ブラウザのメインラインはすでに「データベースファースト」です

これは現在最も重要な境界変更です。古い考え方は、「最初にエンティティを持ち、次にテーブル インターフェイスを作る」という傾向が強かったです。現在では、実際のデータベース構造から直接ブラウジングおよびメンテナンス インターフェイスを生成することがより重要になっています。

### プロバイダーはエンティティよりも重要です

新しいデータベース ソースを拡張する場合、現在の優先事項は、エンティティ クラスのバッチをシステムに追加するのではなく、`IDatabaseBrowserProvider` を実装して登録することです。

### DAO システムはまだ存在しますが、入り口はそれだけではありません。

`BaseTableDao<T>` などの型は引き続き既存のビジネス コードを提供しますが、このモジュールを読むと、データベース機能の唯一の中心として記述することはできなくなります。

## このページはそれ以上何もしません

このページでは、次のような危険性の高いコンテンツは管理されなくなります。

- チュートリアル形式のサンプルコードスタック
- 「ベスト プラクティス」の一般的な段落
- 検証レコードを手動で作成する
- エンティティ クラスのみを扱う古いモデルとしてデータベース モジュールを作成する
