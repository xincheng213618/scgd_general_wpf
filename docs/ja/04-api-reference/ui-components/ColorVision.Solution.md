#ColorVision.ソリューション

このページには、現在の `UI/ColorVision.Solution/` の最も重要で安定したエントリ タイプとサブモジュールのみが保持され、古いドキュメントの「完全な API ホワイト ペーパー + バージョン リスト + 包括的な RBAC 引き継ぎ」という記述スタイルは維持されなくなります。

## モジュールの配置

`ColorVision.Solution` は現在、単純な「ソリューション マネージャー」ではなく、デスクトップ ワークスペース シェルとして理解されています。

現在、実際に取り組んでいることには次のものが含まれます。

- `.cvsln` ソリューションの作成、オープン、および最近のファイルの管理
- ツリー状のプロジェクトの参照と左側の新しいプロジェクトのエントリ
- ファイル/フォルダエディタの選択と開き
- AvalonDockのドキュメントエリアとパネルレイアウトの管理
- 内蔵ターミナルコントロール
- 複数の画像ビューアとサムネイル キャッシュ
- マークダウンプレビュー
- ソリューション側のローカル RBAC サブモジュール

これは、これが単一のウィンドウではなく、`SolutionManager` の周りに編成された UI の薄い層でもないことを意味します。

## 現在最も重要なディレクトリ

プロジェクト ディレクトリを見て、最初に知っておくべき最も重要な点は次のとおりです。

- `Editor/`: ファイルおよびフォルダーのエディターの登録、選択、開きます。
- `Explorer/`: ソリューション ツリー、ノード モデル、新しい項目、コンテキスト メニュー
- `Workspace/`: AvalonDock ドキュメント領域とパネル レイアウト管理
- `Terminal/`: 組み込み端末制御および ConPTY パッケージ化
- `MultiImageViewer/`: フォルダーの複数画像のプレビューとサムネイルのキャッシュ
- `RecentFile/`: 最近のファイル履歴
- `Rbac/`: ソリューション側のローカル ユーザー、ロール、権限、セッション、および監査モジュール
- ルート ディレクトリの `SolutionManager.cs`: ソリューションの開き、作成、現在のワークスペースの切り替えの入り口

## キー入力タイプ

### `SolutionManager`

`SolutionManager` は、現在のワークスペース エントリの中心オブジェクトです。それは次のことを担当します。

- `.cvsln` を開くか作成します
- 最近開いたソリューションのリストを維持する
- 電流 `SolutionExplorer` を生成します
- コマンド ラインまたは最近使用したファイルに基づいて、起動時にどのソリューションを開くかを決定します

「ソリューションがどのようにして得られたか」を追跡したい場合は、通常、ツリー コントロールを最初に見るのではなく、ソリューションを最初に確認します。

### `SolutionExplorer`

`SolutionExplorer` および `Explorer/` ディレクトリ内のノード タイプは、ディレクトリ、ファイル、新しい項目、および右クリック アクションをツリー ワークスペースに編成する役割を果たします。

この部分が「ユーザーからプロジェクトの構造をどう見るか」の入り口となります。

### `EditorManager`

`EditorManager` は編集者の登録と配布を担当します。現在の実装機能は非常に明確です。

- ロードされたアセンブリをスキャンして、`IEditor` を実装する型を探します。
- `EditorForExtensionAttribute`、`GenericEditorAttribute`、`FolderEditorAttribute` 経由で登録する
- 拡張機能のデフォルトエディターの設定を許可します
- フォルダーエディターにも対応

したがって、現在のエディタ システムは手書きのスイッチ テーブルではなく、プロパティ駆動型の登録メカニズムです。

### `WorkspaceManager` および `DockLayoutManager`

これら 2 つは、現在のドキュメント ワークスペースのドッキングと復元を担当します。

- 現在のドキュメントを検索してアクティブ化します
- `ContentId` とドキュメントの選択ステータスを維持する
- AvalonDockレイアウトの保存とロード
- レイアウト復元時にレジストリによってパネルとドキュメントのコンテンツを復元します

「タブはどこへ行った?」という問題が発生した場合は、 「レイアウトが復元されない」または「ドキュメント領域が失われる」場合は、通常、最初にこのリンクを参照してください。

### `TerminalControl`

ターミナル機能は現在、別個の外部サービスではなく、このプロジェクト内にあります。 `TerminalControl` 現在の担当者:

- `cmd` または `powershell` を開始します
- ConPTY出力を引き受ける
- 画面バッファとコマンド履歴を維持する
- スクリプトを実行し、URL クリックを処理します

したがって、これは単に「システム端末を呼び出す」というよりも、組み込みの端末 UI コンポーネントに近いものになります。

### `MultiImageViewer`

`MultiImageViewer` は、独立した `UserControl` として使用することも、`MultiImageViewerEditor` を介してエディタ システムに接続することもできます。

現在は主に以下の業務を担当しています。

- フォルダー内の複数の画像を読み込みます
- 拡張子のフィルタリングをサポート
- サムネイル表示
- ワークスペースのドキュメントタブと連動して開いたり解放したりできます

## RBAC に関して、このモジュールは現在何を担当していますか?

古いドキュメントの最大の問題は、`ColorVision.Solution` が「プロジェクト全体の統合された RBAC 権限制御層」と書かれていることです。現在のコードはこの状態ではありません。

### 現在の実際の状況

`Rbac/` は確かに `ColorVision.Solution` の重要なサブモジュールであり、すでに以下が含まれています。

- `RbacManager`
- `LoginWindow`、`UserManagerWindow`、`PermissionManagerWindow`
- ユーザー、ロール、権限、セッション、監査関連のエンティティおよびサービス
- ネイティブ SQLite の永続性
- `PermissionChecker` のきめ細かい権限コードのキャッシュ

### ただし、現在の境界も明確に書かなければなりません

この RBAC セットは現在、主に独自の管理ウィンドウとソリューション側のローカル許可サブシステムで機能します。

現在の検索結果から判断すると、`HasPermissionAsync` および `PermissionChecker` への詳細な呼び出しのほとんどすべてが依然として `Rbac/` サブディレクトリに残っています。同時に、多くのウィンドウ エントリは依然として大まかな判断のためにグローバル `Authorization.Instance.PermissionMode` に依存しています。

したがって、より正確な説明は次のようになります。

- `ColorVision.Solution` にはネイティブ RBAC サブモジュールが含まれています
- グローバル `PermissionMode` と共存します
- ソリューション ツリー全体、すべてのエディター、およびすべてのファイル操作が、きめ細かい権限コード制御に完全にアクセスできるとは言えません。

## このページはそれ以上何もしません

このページでは、次のような危険性の高いコンテンツは管理されなくなります。

- 古いバージョン番号とターゲット フレームワーク リスト
- 完全なパブリック API の存在を前提とした大きな疑似コード
- `RbacManager` をプロジェクト全体の統一された権限エントリとして記述します
- すべてのファイル操作を、きめ細かい権限によって完全にインターセプトされるように書き込みます。

具体的なクラスやメソッドを追加したい場合は、疑似 API のページ全体をここに積み続けるのではなく、対応するサブモジュール ページで個別に展開する必要があります。
