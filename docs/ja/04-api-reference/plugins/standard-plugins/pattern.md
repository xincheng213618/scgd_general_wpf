# パターン・絵カード生成機能

このページでは、パターンを「現在のウェアハウスに完全に存在し、直接開発できる標準プラグイン」として説明することはなくなりました。現在のソース コードの状況によると、古いドキュメントに記載されている独立した `Plugins/Pattern/` プラグイン実装はウェアハウス内で一致できなくなりました。

## 現在の結論

現在のソース ツリーを見ると、次のようになります。

- `Plugins/` ディレクトリに実際に存在するのは、`Conoscope/`、`EventVWR/`、`Spectrum/`、`SystemMonitor/`、`WindowsServicePlugin/` などのプロジェクトです。
- `Plugins/Pattern/` ディレクトリは現在のウェアハウスに存在せず、対応する `.csproj`、ウィンドウ、`manifest.json`、または独立したプラグイン エントリの実装もありません。
- `IPattern`、`IPatternBase<T>`、`PatternManager`、バッチ生成インターフェイス、および古いドキュメントで言及されているその他のタイプには、現在のウェアハウス ソース コードには対応する実際の定義がありません。

したがって、このページは現在、「ウェアハウス内のピクチャー カード関連機能の残りの配置ポイントの説明」としてのみ使用でき、独立したプラグイン API リファレンスとして機能し続けることはできません。

## ウェアハウス内で実際に照合できる関連コード

独立したパターン プラグイン プロジェクトはありませんが、PG/ピクチャー カードの切り替えに関連する 2 つのコード行がリポジトリにまだ保持されています。

### `Engine/cvColorVision/PG.cs`

この部分は、`cvCamera.dll` における PG 関連のネイティブ インターフェイスの P/Invoke カプセル化です。現在、明確に確認できる機能には次のものがあります。

- `CM_InitPG`
- `CM_ConnectToPG`
- `CM_StartPG`
- `CM_StopPG`
- `CM_ReSetPG`
- `CM_SwitchUpPG`
- `CM_SwitchDownPG`
- `CM_SwitchFramePG`

これは、現在ウェアハウスにあるより信頼性の高い「カード関連機能」は、独自のパターン エディターを備えた高レベルのプラグイン プロジェクトではなく、PG デバイス コントロールであることを示しています。

### `Engine/FlowEngineLib/PG/PGLoopNode.cs`

FlowEngine 側にも `PGLoopNode` があり、ループ ノードの PG パラメーターをコマンド リストに変換し、プロセス実行チェーンを通じて発行します。

- `开始` -> `CM_StartPG`
- `停止` -> `CM_StopPG`
- `重置` -> `CM_ReSetPG`
- `上` / `下` -> 絵カードの切り替え
- `指定` -> `CM_SwitchFramePG`

これは、古いドキュメントのように 11 のパターンをローカルで生成し、PNG/JPEG/BMP をエクスポートする完全なプラグインではなく、「PG デバイスを制御してプロセス中にグラフィックス カードを切り替える」機能に似ています。

## 古い文書がそのまま使用できなくなった理由

古いページには、ソース コードでは証明できない種類のコンテンツがいくつかあります。

- 独立した `Pattern` プラグイン項目とメニュー エントリの存在を主張します。
- `IPattern` / `IPatternBase<T>` などの拡張インターフェイスが存在すると主張します。
- 11 パターン、ローカル テンプレート管理、バッチ エクスポート、プレビューの最適化などの高度な機能の完全なセットをサポートすると主張します。
- 現在のリポジトリに存在しないサンプル API や拡張コードを多数提供します。

これらのコンテンツを保持し続けると、読者は対応する実装が現在のウェアハウスに直接見つかると誤って考えるだけになります。

## 現在のより合理的な理解方法

あなたが今この倉庫で「絵カード」の能力を追い求めているなら、まずそれを次のように理解する必要があります。

1. まず、スタンドアロンの WPF プラグインとしてではなく、PG デバイス制御チェーンの一部として考えてください。
2. まず `cvColorVision/PG.cs` を見て、どの PG コマンドが最下層で送信できるかを確認します。
3. `FlowEngineLib/PG/PGLoopNode.cs` をもう一度読んで、プロセス内のバッチまたはサイクルで PG を切り替える方法を理解します。
4. さらに上位レベルのチャート生成 UI を確認する必要がある場合は、まず、対応するソース コードがウェアハウスから移動されているか、他のプロジェクトにあるか、または単なるドキュメントの残骸であるかを確認してください。

## 現在のこれらのエントリを理解する方法

- ルート ディレクトリの [Plugins/README.md](../../../../Plugins/README.md) には、「現在のソース コードに実際に存在するプラグイン ディレクトリ」と過去の残りの名前が明確に区別されるようになりました。
- API リファレンスのこのページでは、現在のウェアハウスにあるパターン関連機能の残りの配置ポイントの説明のみが保持され、既存の標準プラグインとしては記述されなくなりました。
- 2 つのエントリの表現に違いがある場合は、ソース コード ディレクトリとランタイムの読み込み動作が優先されます。
