# プロセス機器サービス

プロセスデバイスサービスは、「プロセスから呼び出されますが、手動で頻繁に操作することはありません」という立場で理解するのが適しています。現在のコードでは、これはプロセス システムのデバイスの入り口に似ています。複雑な独立した操作パネルではなく、構成、接続、およびプロセスによって正しく参照されることに重点が置かれています。

## このページはいつ読むべきですか?

- プロセスデバイスサービスをデバイスリストに追加する必要があります
- プロセス ノードでこのタイプのデバイスを参照する必要がある
・「設備を構築したものの、途中で移管できない」などのトラブルシューティングを行っている。
- この種のサービスが単なる設定ポータルであり、独立した業務窓口ではないかどうかを確認したい

## まずそれを理解する方法

現在の実装では、プロセス デバイス サービスは主に次の 2 つを提供します。

- 標準のデバイス サービス インスタンスとしてシステムによって管理されます
- プロセス実行時に参照できるデバイスオブジェクトとしてMQTTサービス層に認識される

言い換えれば、内部メッセージの詳細を調査するよりも、メッセージが正しく作成、保存され、プロセスによって参照できることを確認することの方が重要です。

## 一般的な使用シーケンス

1. まずプロセスデバイスサービスを作成し、デバイスリストに保存します。
2. デバイス コード、テーマ、または関連する接続パラメータが正しく設定されていることを確認します。
3. [ワークフローの概要](../workflow/README.md) に戻るか、特定のプロセス ウィンドウでこのデバイスを参照します。
4. プロセスを実行するときは、リストにデバイスが表示されることを確認するだけでなく、実際にデバイスがヒットするかどうかを観察します。

## 使用する際に特に注意すべきことは何ですか?

### デバイスが存在するからといって、プロセスがすでにそのデバイスを使用しているとは限りません。

デバイスがリストに表示されるということは、それがリソースとして作成されたことを意味するだけです。本当に重要なのは、正しいデバイス インスタンスがプロセス ノードで参照されているかどうかです。

### 最初に構成を確認してから、プロセス ノードを確認してください

プロセス中にデバイスを調整できない場合は、まず次のことを確認してください。

- 現在のデバイスコードは正しいですか?
- このデバイスはプロセスで参照されますか?
・ 該当する通信設定が一致しているか

### 独立した手動アクションが多数あるとは考えないでください

現在の実装から判断すると、このタイプのデバイス ページは、構成層とアクセス層に重点を置いています。トラブルシューティングを行うときは、複雑な独立したコントロール ボタンを最初に探すのではなく、まずデバイスの構成とプロセスの参照関係を調べます。

## よくある質問

### デバイスは作成されましたが、プロセス内で見つかりません。

- まずデバイスが正常に保存されたかどうかを確認します
- プロセスノードが正しいデバイスオブジェクトを参照しているか再度確認してください
- プロセス内でコードまたは名前でフィルタリングする場合は、不一致がないか確認します

### プロセスを実行してもデバイスが応答しません。

- まずデバイスサービス自体の接続構成を確認します
- 実行したプロセスが本当に該当ノードであるか再確認
- 同時に、[ログビューア](../interface/log-viewer.md)を結合して、メッセージ層がこのデバイスにヒットしないかどうかを確認します。

### 問題がデバイス層にあるのかプロセス層にあるのかわからない

- まず質問を 2 つの部分に分割します。装置が存在するかどうかと、プロセスが参照されているかどうかです。
- デバイスの設定自体が不完全な場合は、まずデバイスを修復してください
- デバイスに問題がない場合は、[プロセスの実行とデバッグ](../workflow/execution.md)に戻り、範囲を絞り込みます。
