# 소켓 통신 모듈 최적화 경로

이 문서는 `UI/ColorVision.SocketProtocol`을 지향하며 현재 사용 가능한 소켓 관리 기능을 보다 안정적이고 문제 해결이 더 쉬우며 장기적인 작업에 더 적합한 통신 모듈로 계속 승격하는 데 사용됩니다.

## 현재 구조 판단

`ColorVision.SocketProtocol`은 이제 완전한 폐쇄 루프를 형성했습니다.

- `SocketInitializer`는 구성된 대로 서비스를 시작하거나 중지합니다.
- `SocketManager`는 TCP 모니터링, 클라이언트 관리 및 요청 배포를 담당합니다.
- `SocketJsonDispatcher` / `SocketTextDispatcher`는 JSON 및 텍스트 모드를 담당합니다.
- `SocketMessageManager`는 SQLite에 메시지를 씁니다.
- `SocketManagerWindow` 및 `SocketStatusBarProvider`는 데스크톱 측 작업 및 유지 관리 입구를 제공합니다.

이는 더 이상 "프로토콜 샘플 코드"가 아니라 UI, 지속성 및 런타임 상태를 갖춘 통신 하위 시스템임을 보여줍니다. 최적화할 때 기존 동작을 보호하는 데 우선순위를 두고 프로토콜 경계와 장기적인 운영 기능을 점진적으로 완성해야 합니다.

## 경험 최적화 완료

관리 창에서는 위험도가 낮은 기능을 일괄적으로 완료했습니다.

- 상위 서비스 상태 : 활성화 상태, 수신 주소, 프로토콜 모드, 연결 상태, 클라이언트 수
- 열기 실패 프롬프트: 마지막 시작 오류가 창 상단과 서비스 진단 페이지에 표시됩니다.
- 메시지 필터링: 클라이언트, 이벤트 이름, MsgID, 응답 코드, 내용별로 필터링
- 방향 필터링 : 전체, 수신, 전송
- 목록 가상화: 기록 메시지를 탐색할 때 UI 부담을 줄입니다.
- 세부 서식 지정: JSON 콘텐츠를 보기 위한 서식으로 지정할 수 있으며, 원본 텍스트도 복사할 수 있습니다.
- 자동 스크롤: 새 메시지가 도착하면 자동으로 위치를 지정하거나 끌 수 있습니다.
- 재전송: 원본 클라이언트와 일치하는 것을 우선으로 하고, 일치하는 것이 없으면 현재 선택된 클라이언트를 사용합니다.
- 레이아웃 수렴 : 우측의 "메시지 내용/클라이언트/서비스 진단" 탭을 이용하여 보조 정보 전달
- 단축키: `Ctrl+F`, `Esc`, `F5`, `Ctrl+C`, `Delete`

## P0 안정성 경로

목표: 첫째, 사이트에서 "간헐적인 중단, 포트 점유 및 메시지 잘림" 문제를 줄이기 위해 서비스 수명 주기와 TCP 송수신 경계를 강화합니다.

| 프로젝트 | 현재위험 | 권장 사례 | 합격기준 |
| --- | --- | --- | --- |
| 서비스가 반복적으로 시작됩니다 | `StartServer()`를 여러 번 호출하면 새로운 백그라운드 작업이 생성됩니다 | `_serverTask`, `CancellationTokenSource` 및 시작 상태 보호 추가 | 시작 및 중지를 위한 연속 클릭은 여러 청취 루프를 생성하지 않습니다 |
| 서비스 중지 | `AcceptTcpClient()`는 `TcpListener.Stop()`을 사용하여 예외를 발생시키고 종료합니다 | 명시적 취소, 취소 경로 캡처, 정상 중지와 오류 중지 구분 | 서비스 중지 로그에 더 이상 비정상적인 오류가 표시되지 않습니다. |
| TCP 경계 | 메시지로 '읽기'를 한 번 누르세요 | 길이 접두사, 구분 기호 또는 구성 가능한 프레임 파서 도입 | 큰 메시지와 연속적인 작은 메시지가 고정되지 않거나 절반 패킷 |
| 클라이언트 수집 | `ObservableCollection<TcpClient>`는 백그라운드 스레드와 상호작용할 때 UI 스레드 제약 조건을 유지해야 합니다. | 추가/제거 캡슐화, 모든 컬렉션 변경 사항은 Dispatcher | 장기간의 연결/연결 끊김이 크로스 스레드 예외를 발생시키지 않습니다. |
| 클라이언트 정리 | 연결 끊김, 정지 및 비정상 경로 분산 | 통합 `CloseClient` 메소드 추가 | 클라이언트 연결이 끊어진 후 목록, 리소스 및 로그의 일관성을 유지 |

## P1 관찰 경로

목표: 창과 로그가 "지금 연결된 사람이 누구인지, 방금 무슨 일이 일어났는지, 오류가 집중된 위치"에 대해 답변하도록 합니다.

| 프로젝트 | 권장 사례 | 합격기준 |
| --- | --- | --- |
| 연결 수명주기 | 연결설정, 연결해제, 이상사유 기록 추가 | 관리 창에서는 연결 기록을 확인하거나 로그를 찾을 수 있습니다 |
| 메시지 내보내기 | 현재 필터링 결과를 JSON/CSV로 내보내기 지원 | 문제 샘플을 현장에서 직접 채취 가능 |
| 오류 통계 | 통계 JSON 구문 분석 실패, 처리기를 찾을 수 없음, 전송 실패 | 상태 표시줄 또는 창에 실패 횟수 표시 가능 |
| 처리 시간 | 레코드 핸들러 실행 시간 | 느린 이벤트를 찾을 수 있습니다 |
| 데이터베이스 보존 정책 | 수량 또는 시간별 이력 정리 | 장기 실행은 무한정 증가하지 않습니다 |

## P2 프로토콜 경로

목표: 외부 장치 도킹을 "기존"에서 "검증 가능, 호환 가능 및 마이그레이션 가능"으로 승격합니다.

| 프로젝트 | 권장 사례 | 합격기준 |
| --- | --- | --- |
| 메시지 모델 | `Version`, `MsgID`, `EventName`, `SerialNumber`, `Params`의 필수 필드를 명시합니다 | 문서 및 런타임 검증이 일관됩니다 |
| 오류 코드 | 통합 오류 코드 범위 정의 | 클라이언트는 Code | Code |
| 핸들러 메타데이터 | 핸들러에 대한 설명, 버전, 예제 매개변수 추가 | 관리 창 또는 문서에서 지원 이벤트를 열거할 수 있음 |
| 스키마 | 주요 이벤트에 대한 JSON 스키마 제공 | 도킹당사자가 사전에 확인할 수 있다 |
| 호환성 전략 | 구버전과 신버전 간 현장 호환성 합의 | 기존 장비를 파괴하지 않고 업그레이드 |

## P3 성능 및 용량 경로

목표: 더 긴 실행, 더 높은 메시지 빈도 및 더 많은 메시지 기록을 지원합니다.

| 프로젝트 | 권장 사례 | 합격 기준 |
| --- | --- | --- |
| UI 페이징 | 관리 창은 페이징 또는 증분 방식으로 기록을 로드합니다 | 히스토리 데이터베이스 확장으로 인해 열리는 창이 막히지 않습니다 |
| 데이터베이스에 일괄 쓰기 | 빈도가 높은 메시지 중 일괄 쓰기 또는 백그라운드 큐 쓰기 | 메시지 피크 동안 데이터베이스 쓰기로 인해 UI 속도가 느려지지 않습니다. |
| 색인 | `MessageTime`, `ClientEndPoint`, `EventName`에 대한 인덱스 추가 | 기록 데이터의 제어 가능한 쿼리 및 필터링 |
| 배압 | 극단적인 메시지 빈도에 대한 큐 캡 및 성능 저하 전략 설정 | 프로세스를 중단시키는 대신 과부하 시 메시지 표시 |
| 스트레스 테스트 | 로컬 TCP 스트레스 테스트 스크립트 추가 | 재현 가능 용량 상한 및 회귀 결과 |

## 권장 구현 순서

1. 프로토콜 계층에서 숨겨진 위험이 더 이상 증폭되지 않도록 서비스 수명주기 보호 및 P0의 프레임 구문 분석을 먼저 수행합니다.
2. P1의 내보내기, 통계 및 보존 전략을 추가하여 현장 문제 처리의 효율성을 향상시킵니다.
3. 그런 다음 P2의 프로토콜 문서, 오류 코드 및 핸들러 메타데이터를 진행합니다.
4. 마지막으로 실제 메시지 양을 기준으로 P3의 페이징, 일괄 쓰기 및 스트레스 테스트 투자를 결정합니다.
