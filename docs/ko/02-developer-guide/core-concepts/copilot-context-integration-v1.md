# Copilot 컨텍스트 통합 v1

이 문서에는 "독립 채팅 패널"에서 "소프트웨어 컨텍스트 서비스"로의 ColorVision Copilot 진화에 대한 설계 경계, 구현 항목 및 후속 계획의 첫 번째 버전이 기록되어 있습니다.

## 타겟

현재 Copilot에는 이미 세션, 모델 구성, 첨부 파일, 도구 호출, 로그 진단 및 오른쪽 Dock 패널이 있지만 비즈니스 개체에 대한 이해는 여전히 주로 파일 경로 및 사용자의 수동 마운트 컨텍스트에 남아 있습니다.

다음 단계의 목표는 단일 지점에서 채팅 기능을 지속적으로 향상시키는 것이 아니라 Copilot을 다양한 모듈에서 호출할 수 있는 공용 플랫폼 기능으로 전환하는 것입니다.

- 비즈니스 모듈은 ColorVision/Copilot 디렉토리 구현에 직접적으로 의존하지 않습니다.
- Copilot은 현재 소프트웨어 시나리오의 구조화된 컨텍스트를 자동으로 전달할 수 있습니다.
- Copilot의 입구는 상태 표시줄에서 솔루션 트리, 이미지, 프로세스, 알고리즘 결과 및 예외와 같은 워크플로 노드까지 확장됩니다.

## 이번에 구현한 Simple 버전

이번에는 먼저 위험도가 낮고 재사용 가능한 기본 레이어를 완성했습니다.

1. 공공 계약 레이어

- UI/ColorVision.Common/Interfaces/Copilot 아래에 'ICopilotService', 'ICopilotContextProvider' 및 최소 요청/컨텍스트 모델을 추가했습니다.
- `CopilotPanelService`는 이제 `ICopilotService`의 구현 역할을 하며 `CopilotServiceRegistry`를 통해 다른 모듈에 노출됩니다.

2. 최소 컨텍스트 제공자

- `CopilotContextRegistry` 및 `CopilotWorkspaceContextProvider`를 추가했습니다.
- 현재 에이전트 요청은 솔루션 루트 디렉터리, 현재 활성 콘텐츠 및 현재 검색 루트를 포함하여 "현재 작업 영역" 컨텍스트에 자동으로 중단됩니다.
- 이 단계에서는 비즈니스 객체를 직접 이해하지 않지만 먼저 "소프트웨어의 현재 작업 표면"을 공개적이고 확장 가능한 컨텍스트 항목으로 바꿉니다.

3. 솔루션 트리 항목

- `FileNode`에 "AI에게 이 파일 해석을 요청" 및 "AI에게 이 파일/로그 진단을 요청"이 추가되었습니다.
- 'FolderNode'에 "AI에게 이 폴더를 요약하라고 요청하세요"가 추가되었습니다.
- 이러한 항목은 Copilot 구현을 직접 참조하는 대신 공개 'ICopilotService'를 통해 트리거됩니다.

4. 브랜드 통일

- 상태 표시줄 항목 이름이 'GitHub Copilot'에서 'ColorVision Copilot'으로 조정되었습니다.

## 여기서 먼저 멈춰야 하는 이유는 무엇입니까?

이 버전은 다음과 같은 이유로 의도적으로 Engine, ImageEditor, Flow 또는 Device Services 레이어로 직접 이동하지 않습니다.

- 이러한 모듈이 ColorVision/Copilot을 직접 참조하면 신속하게 백 커플링을 형성합니다.
- 현재 가장 부족한 것은 '채팅 기능의 추가'가 아니라 '비즈니스 모듈에서 안정적으로 호출할 수 있는 상황별 인터페이스'입니다.
- 먼저 공개 계약을 클리어하고 최소한의 컨텍스트 수집과 고주파 접속을 수행한 후 나중에 저렴한 비용으로 비즈니스 개체를 연결합니다.

## 후속 기획

### 1단계: 컨텍스트 브리지 확장

먼저 다음 공급자를 추가하세요.

- 현재 창/현재 페이지 제공자
- 현재 선택된 객체 제공자
- ROI 제공자가 포함된 현재 이미지
- 현재 로그 및 피드백 패키지 제공자
- 현재 프로세스/노드/최근 장애 제공자

이 단계의 승인 기준은 다음과 같습니다.

- 사용자가 주요 페이지에서 Copilot을 시작할 때 현재 개체에 대한 설명을 수동으로 반복할 필요가 없습니다.
- 에이전트는 파일 경로만이 아닌 구조화된 애플리케이션 컨텍스트를 볼 수 있습니다.

### 2단계: 현장 입장

다음 유형의 출입구를 통한 접근이 우선적으로 이루어집니다.

- 이미지 페이지를 마우스 오른쪽 버튼으로 클릭: 현재 이미지를 분석하고 현재 ROI를 해석합니다.
- 알고리즘 결과 페이지: 이 결과와 실패한 이유를 설명합니다.
- 흐름: 현재 프로세스를 설명하고 마지막 실패를 진단합니다.
- 장치 패널: 현재 장치 상태 분석
- 예외 대화 상자: 분석을 위해 Copilot에 맡기십시오.

이 단계에서는 "공용 인터페이스 호출"을 강조해야 하며 이러한 모듈이 Copilot 특정 창이나 ViewModel에 직접 의존하도록 허용하지 않아야 합니다.

### 3단계: 탐색 작업

해석/진단이 안정된 후 위험도가 낮은 UI 작업을 도입합니다.

- 패널 열기
- 창에 집중
- 구성 항목 찾기
- 로그 디렉터리를 엽니다.

이 단계에서는 내비게이션과 위치 확인만 수행되며 구성 변경, 장비 제어, 프로세스 실행 등 위험도가 높은 작업은 수행되지 않습니다.

### 4단계: 비즈니스 도구

기존 SearchFiles, GrepText, ReadLocalFile 및 GetRecentLog를 기반으로 읽기 전용 비즈니스 도구가 추가되었습니다.

- `GetActiveWorkspaceContextTool`
-`GetSelectedImageContextTool`
- `GetAlgorithmResultContextTool`
-`GetFlowRunSummaryTool`
-`GetDeviceStatusTool`
- `GetDiagnosticsPackageSummaryTool`

## Agent/ReAct 경로와의 관계

Copilot Context Integration v1은 "일반 코드 에이전트처럼 작동합니까?"가 아니라 "현재 소프트웨어 시나리오를 이해합니까?"를 다룹니다.

따라서 후속 경로는 두 개의 평행선으로 나누어야 합니다.

- 컨텍스트 통합: 소프트웨어 내에서 컨텍스트와 항목을 해결합니다.
- Agent/ReAct: 도구 계획, 저비용 검색, 구조화된 실행 및 더 많은 진단 도구를 해결합니다.

두 경로 모두 중요하지만 현재 우선순위는 컨텍스트 통합이어야 합니다.

## 현재 경계

이 버전 기준:

- 이미 공개 Copilot 계약 레이어가 있습니다.
- 이미 최소 작업공간 컨텍스트 제공자가 있습니다.
- 솔루션 트리에 고주파 입구가 있습니다.
- ImageEditor, Flow, Algorithm 및 Device의 비즈니스 개체 컨텍스트가 여전히 연결되지 않았습니다.
- UI 액션 카드, 비즈니스 도구 및 전역 설정 페이지 임베딩은 아직 구현되지 않았습니다.

후속 설계 및 일정은 이 문서를 기반으로 하는 것이 좋습니다.
