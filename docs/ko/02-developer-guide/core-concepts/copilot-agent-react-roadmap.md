# 부조종사 에이전트 현황 및 ReAct 진화 로드맵

이 문서에서는 ColorVision의 현재 Copilot Agent가 실제로 작동하는 방식, ReAct 스타일 에이전트와 어떻게 다른지, 로컬 파일 읽기 및 저렴한 컨텍스트 검색을 지원하기 위해 어떻게 발전할지 설명합니다.

Copilot 컨텍스트 통합 v1도 참조하세요. 이 문서는 Copilot이 독립 채팅 모듈에서 이 문서의 Agent/ReAct 경로를 보완하는 소프트웨어 컨텍스트 서비스로 어떻게 발전하는지에 중점을 둡니다.

## 한 문장 결론

현재 구현은 표준 ReAct 에이전트가 아니라 "정적 후보 선별 + 최소 모델 플래너 + 최대 여러 라운드의 도구 실행 + 로컬 구조 파일 읽기 매개변수 + 일회성 압축 컨텍스트 + 대형 모델에 대한 일회성 호출"의 읽기 전용 에이전트입니다.

최신 라운드에서는 "사용자의 원래 키워드 임계값"의 정적 제한을 약화시키기 시작했습니다. 도구 레지스트리는 더 이상 플래너에게 표시되는 도구를 처음 6개로 자르지 않으며 SearchFiles, GrepText 및 GetRecentLog도 런타임 기능을 기반으로 한 가시성 판단으로 변경되어 플래너가 검색 루트 또는 최근 로그가 있을 때 저렴한 검색을 먼저 수행할지 여부를 결정할 수 있습니다. 플래너 구문 분석이 실패하면 더 이상 레지스트리의 첫 번째 도구를 맹목적으로 실행하지 않고 현재 모드와 알려진 컨텍스트를 기반으로 보수적인 폴백을 수행합니다. 적합한 후보가 없으면 도구 단계가 바로 종료됩니다.

이미 최소한의 도구 계층과 최소한의 계획자-실행자 프로토타입이 있지만 아직 다음 기능은 없습니다.

- 코드 에이전트처럼 계속해서 검색, 수정, 빌드, 오류 검색 및 반복 작업을 수행합니다.
- 모델이 ReadLocalFile, SearchFiles, GrepText, GetRecentLog 및 FetchUrl뿐만 아니라 더 많은 진단/환경 도구를 위해 실제로 구조화된 매개변수를 출력하도록 합니다.
- 라운드당 단일 단계로 진행되는 기존의 최소 플래너 대신 더욱 강력해진 멀티툴 연속 플래너 지원

## 현재 실제 실행 링크

현재 통화 링크는 다음과 같습니다.

``인어
흐름도 TD
    A[사용자 입력] --> B[CopilotChatViewModel.RunAgentTurnAsync]
    B --> C[CopilotAgentRequest 구성]
    C --> D[CopilotAgentService.RunAsync]
    D --> E[CopilotToolRegistry.FindTools]
    E --> F [규칙에 따라 사용 가능한 도구 선택]
    F --> G [순차 실행 도구]
    G --> H[CopilotToolResult 목록 생성]
    H --> I[CopilotAgentContextBuilder.BuildMessages]
    I --> J[질문 + 첨부 파일 + 도구 결과를 최종 프롬프트 단어로 압축]
    J --> K[CopilotChatService.StreamReplyAsync]
    K --> L[스트리밍 출력 추론 및 최종 답변]
```
키 코드 위치:

- 에이전트 주요 프로세스: ColorVision/Copilot/Agent/CopilotAgentService.cs
- 도구 레지스트리: ColorVision/Copilot/Agent/CopilotToolRegistry.cs
- 도구 인터페이스: ColorVision/Copilot/Agent/ICopilotTool.cs
- 컨텍스트 압축: ColorVision/Copilot/Agent/CopilotAgentContextBuilder.cs
- UI 액세스: ColorVision/Copilot/CopilotChatViewModel.cs

## 현재 도구 레이어의 역할은 정확히 무엇인가요?

현재 8개의 읽기 전용 도구와 3개의 제어 실행 도구를 포함하여 11개의 도구가 기본적으로 등록되어 있습니다.

1.실행메뉴
   기능: "옵션", "VAM", "업데이트 확인"과 같은 메뉴 이름이나 경로로 기본 메뉴 명령을 실행하고 테마 및 언어와 같은 메뉴 하위 항목도 실행합니다.

2. 테마 설정
   기능: 밝은, 어두운, 분홍색, 청록색과 같은 사용자의 명시적인 의도에 따라 애플리케이션 테마를 전환하거나 시스템을 따릅니다.

3.언어 설정
   기능: 사용자의 명시적 의도에 따라 인터페이스 언어를 전환하고 기존 재시작 확인 프로세스를 재사용합니다.

4. 검색문서
   기능: 게시된 ColorVision 온라인 문서 인덱스를 쿼리하고 소프트웨어 사용, 메뉴, 장비, 플러그인, 개발 가이드 및 아키텍처 설명 문제에 적합한 페이지 내의 장, 페이지 및 제목별로 가장 관련성이 높은 조각을 반환합니다.

5. FetchUrl
   기능: 쿼리를 통해 기획자가 지정한 URL에 대한 크롤링 우선 순위를 지정합니다. 플래너가 URL을 제공하지 않으면 사용자 텍스트의 URL로 돌아가 웹 페이지 본문을 크롤링합니다.

6.파일 검색
   기능: 파일 이름 또는 경로 조각으로 현재 솔루션 검색 루트에서 후보 파일을 찾습니다.

7. 그렙텍스트
   기능: 현재 솔루션 텍스트 파일에서 키워드 또는 식별자로 히트 라인을 찾습니다.

8.목록디렉토리
   기능: 현재 메시지에서 사용자가 명시적으로 언급한 로컬 폴더의 내용을 나열하고 읽을 수 있는 후속 파일 후보를 생성합니다.

9. 첨부파일 읽기
   기능: "현재 세션에 마운트된 첨부 파일"을 읽습니다.

10. 로컬파일 읽기
   기능: 현재 메시지에서 사용자가 명시적으로 언급한 로컬 텍스트 파일을 읽습니다.

11.최근 로그 가져오기
   기능: 최신 로그를 읽고 플래너가 제공한 쿼리에 따라 결과를 필터링합니다.

이는 현재 에이전트가 여전히 명확하게 제어 가능한 기능 경계를 가지고 있음을 보여줍니다.

- 이미 세션에 첨부된 파일을 읽을 수 있습니다.
- 현재 사용자의 메시지에 명시적으로 나타나는 로컬 텍스트 파일 경로를 읽을 수도 있습니다.
- 또한 현재 솔루션 루트 디렉터리, 활성 문서 디렉터리 및 첨부 파일이 있는 디렉터리를 기반으로 가벼운 파일 이름/텍스트 검색 라운드를 수행할 수도 있습니다.
- 이제 출시 후 ColorVision 온라인 문서 색인에 대한 액세스를 제공하여 보다 적은 맥락으로 소프트웨어 사용 및 개발 문서 관련 질문에 답할 수 있습니다.
- 이제 메인 메뉴 명령 실행, 테마 전환, 인터페이스 언어 등 명시적인 사용자 의도로 소수의 제어된 작업을 수행할 수도 있습니다.
- 대화에서 모델에 의해 일시적으로 생성된 새 경로를 기반으로 하는 로컬 파일을 읽을 수 없으며 부작용도 수행할 수 없습니다.
- 이미 최소한의 구조화된 매개변수 레이어가 있지만 매개변수 표면은 여전히 ​​상대적으로 좁습니다.

## ReAct와 ReAct의 핵심 차이점

ReAct의 일반적인 패턴은 다음과 같습니다.

1. 모델은 현재 어떤 정보가 누락되어 있는지 먼저 생각합니다.
2. 모델은 Search, ReadFile, Grep, RunTests와 같은 작업을 제안합니다.
3. 시스템은 작업을 실행하고 관찰 결과를 반환합니다.
4. 모델은 관찰을 기반으로 다음 단계를 계속 결정합니다.
5. 여러 라운드를 거친 후 최종 답변을 제공합니다.

현재 구현과 이상적인 완전한 코딩 에이전트 사이에는 여전히 격차가 있습니다. 네 가지 주요 사항이 있습니다.

### 1. 도구는 모델에 따라 결정되는 것이 아니라 규칙에 따라 미리 결정됩니다.

현재 CopilotToolRegistry.FindTools는 요청에 따라 CanHandle을 직접 실행합니다.

즉, 다음과 같습니다.

- 도구 실행 여부는 로컬 C# 규칙에 따라 다릅니다.
- 모델이 컨텍스트에 따라 "다음에 읽을 파일"을 동적으로 결정하는 대신

### 2. 도구에는 이미 경량의 구조화된 매개변수가 있지만 매개변수 표면은 여전히 제한적입니다.

ICopilotTool의 현재 서명은 다음과 같습니다.``csharp
bool CanHandle(CopilotAgentRequest 요청);
Task<CopilotToolResult> ExecuteAsync(CopilotAgentRequest 요청, CopilotAgentToolInput toolInput, CancellationToken cancelToken);
```
즉, 도구는 전체 요청을 얻는 것 외에도 최소한의 구조화된 입력 개체도 가져옵니다. 예를 들면 다음과 같습니다.

```json
{ "도구": "read_file", "경로": "...", "startLine": 1, "endLine": 200 }
```
구조화된 입력의 현재 계층은 주로 쿼리, 경로, startLine 및 endLine의 네 가지 필드를 포함합니다. 검색, 파일 읽기 및 SetTheme/SetLanguage와 같은 작은 애플리케이션 제어 작업을 지원하는 데 충분합니다. 그러나 완전한 함수 호출과 같은 더 자세한 스키마, 권한 모델 및 매개변수 확인 시스템은 아직 없습니다.

### 3. 서비스 계층에는 이미 최소 계획자-실행자 루프가 있지만 아직 완전한 폐쇄 루프는 아닙니다.

CopilotAgentService.RunAsync의 패턴은 다음과 같습니다.

- 각 라운드마다 먼저 기획자가 현재 사용 가능한 도구에서 작업을 선택할 수 있습니다.
- 도구를 실행하고 관찰 내용을 기록합니다.
- 플래너가 완료되거나 도구 단계가 수렴될 때까지 MaxToolRounds 범위 내에서 다음 라운드를 계속합니다.
- 마지막으로 누적된 도구 관찰 결과를 모델에 공급하여 최종 답변을 출력합니다.

따라서 더 이상 "모든 도구를 먼저 실행한 다음 통일된 답변을 제공"하는 단일 라운드 모델이 아니지만 아직 편집, 테스트 및 검증이 포함된 완전한 코딩 루프로 발전하지 않았습니다.

### 4. 코드 프록시를 위한 값싼 검색 레이어가 없습니다.

사용자가 기대하는 프로세스는 다음과 같습니다.

``텍스트
사용자 작업
-> 에이전트가 어떤 컨텍스트가 필요한지 결정합니다.
-> 먼저 저렴한 도구를 사용하여 후보자 정보를 찾으십시오.
-> 후보 결과 압축
-> 대형 모델로 보내기
-> 수정 계획 개발
-> 편집
-> 테스트/빌드
-> 오류 보기
-> 계속 검색/계속 수정
```
현재 구현에서는 그 중 일부만 다루고 있습니다.

``텍스트
사용자 작업
-> 규칙 노출 도구
-> 플래너가 다음 단계를 선택합니다.
->제어된 도구 실행
-> 압축 결과
-> 모델 크기 늘리기
-> 끝
```
grep, 기호 검색, AST 검색, 내장 검색, Git 기록, 진단, 테스트, 수정 및 확인 루프와 같은 저렴한 후보 컨텍스트 도구가 없습니다.

## 아직 완전한 로컬 파일이 아닌 이유는 무엇인가요?

현재 버전에서는 이미 "C:\\Users\\...\\remote_control.py를 읽어주세요"와 같은 명시적인 경로 입력을 처리할 수 있지만 여전히 명확한 경계가 있습니다.

1. 현재로서는 모델이 허용 목록에서 벗어나 새 경로를 읽는 것을 허용할 수 없습니다.
2. 요청에는 이미 최소한의 통합 도구 매개변수 개체가 있지만 인터페이스 계층과 권한 정책 개체는 아직 완전히 독립적이지 않았습니다.
3. 구조적 매개변수는 현재 ReadLocalFile 및 ListDirectory의 경로뿐만 아니라 SearchFiles, GrepText, GetRecentLog, SearchDocs 및 FetchUrl의 쿼리도 포함합니다. 그중에서도 SearchFiles, GrepText 및 GetRecentLog의 가시성 또한 "사용자의 원래 키워드"에서 "기능 우선순위"로 완화되기 시작했습니다. SearchDocs는 출시 후 안정적인 문서 색인을 따릅니다. FetchUrl은 이미 구조화된 쿼리 실행을 지원하지만 도구 가시성은 여전히 주로 요청 수준 URL 추출에 의존합니다.
4. 서비스 계층은 이미 최소 계획자-실행자 루프를 수행할 수 있지만 아직 더 강력한 다중 도구 계획 폐쇄 루프는 아닙니다.
5. 현재 파일 읽기는 라인 범위별 집중 읽기를 지원하지만 아직은 더 자세한 조각 위치 지정, 기호 수준 읽기 및 AST 수준 컨텍스트를 지원하지 않습니다.

그래서 첫 번째 단계를 밟았지만 아직 실제로 코드 검색 에이전트는 아닙니다.

## 추천 진화 방향

한 단계에서 "완전한 코딩 에이전트"로 바로 이동하는 것은 권장되지 않습니다. 보다 안정적인 경로는 3개 계층으로 진화하는 것입니다.

### 1단계: 먼저 명시적 경로를 사용한 로컬 파일 읽기 지원

목표: 사용자가 명시적으로 경로를 제공하면 에이전트가 파일을 안전하게 읽을 수 있습니다.

이 단계에는 완전한 ReAct가 필요하지 않으며 현재 최소 다중 라운드 에이전트만 확장하면 됩니다.

현재 상태: 현재 사용자 메시지에서 명시적인 로컬 경로를 추출하고 명시적인 파일과 폴더를 오프로드할 수 있는 최소 버전이 구현되었습니다. 또한 솔루션 검색 루트를 기반으로 SearchFiles 및 GrepText를 추가하고 로컬 폴더에 대한 ListDirectory를 추가했으며 `ListDirectory(path) -> ReadLocalFile(batch-all)` 및 `SearchFiles/GrepText -> ReadLocalFile(path, startLine, endLine)`을 지원합니다. 이 최소 2륜 체인 실행입니다. 명시적인 디렉터리 분석 시나리오의 경우 첫 번째 ReadLocalFile은 파일별로 라운드를 계속 사용하는 대신 현재 디렉터리의 모든 후보 파일을 일괄적으로 읽는 데 우선 순위를 둡니다.

#### 새로운 기능 제안

1. CopilotAgentRequest에 실행 컨텍스트 추가

현재 액세스된 필드:

-SearchRootPaths
-ActiveDocumentPath

앞으로도 필드를 추가하는 것이 좋습니다.

-AllowedReadRoots
-외부 읽기 허용
-MaxToolCalls

2. ReadLocalFile 도구 추가

권장 입력 매개변수:

-경로
-startLine
- 엔드라인
- 이유

제안된 반품 콘텐츠:

- 실제 구문 분석된 경로
- 문서 요약
- 조각 내용 읽기
- 정보 자르기
- 오류 메시지

3. 명시적인 경로 트리거링을 먼저 수행하고 모델 계획을 서두르지 마십시오.

가장 저렴한 첫 번째 버전은 다음과 같습니다.

- 사용자 메시지에 파일 경로로 보이는 텍스트가 나타나는 경우
- 경로가 허용 범위 내에 있습니다.
- 그런 다음 자동으로 ReadLocalFile을 호출합니다.

이 단계는 스크린샷의 장면을 커버하기에 충분합니다.

#### 보안 경계

이 단계에는 파일 액세스 정책이 있어야 합니다. 그렇지 않으면 허가 없이 쉽게 읽을 수 있습니다.

- 기본적으로 작업공간 내의 경로만 허용됩니다.
- 작업공간 외부 경로에는 사용자의 명시적인 승인이 필요합니다.
- .cs .xaml .py .json .md .txt .log와 같은 화이트리스트 텍스트 확장자만 허용합니다.
- 단일 읽기에서 최대 문자 및 줄 수를 제한합니다.
- 바이너리 파일은 직접 거부됩니다.
- 실행 과정에서 "어떤 경로를 읽었는지, 몇 줄을 읽었는지" 표시

### 2단계: “저렴한 검색 도구 레이어” 완성

목표: 첨부 파일과 URL에만 의존하는 대신 에이전트가 대규모 모델을 호출하기 전에 자체적으로 후보 컨텍스트를 수집하도록 합니다.현재 상태: 2단계에서 최소 구현 버전이 완료되었습니다. SearchFiles, GrepText, GetRecentLog 및 SearchDocs가 기본 도구 목록에 연결되었습니다. 그 중 SearchDocs는 출시된 docs-search-index.json을 통해 ColorVision 온라인 문서를 쿼리하므로 사용자는 더 이상 URL을 먼저 제공할 필요가 없습니다. 도구 레지스트리의 처음 6개 하드 잘림 항목도 제거되었습니다. SearchFiles, GrepText 및 GetRecentLog는 이제 런타임 기능을 기반으로 플래너에 먼저 노출되지만 아직 glob/regular/symbol 수준에서는 구체화되지 않았습니다. 실행 계층은 또한 구조화된 쿼리, 중복 감지 및 실행 요약을 통해 FetchUrl을 보완했습니다. 더 이상 "사용자의 원래 문장에 URL이 첨부된 특수 도구"로 간주되지 않습니다.

향후 제안되는 새롭거나 향상된 읽기 전용 도구:

1.파일 검색
   기능: glob, 디렉터리 제약 조건 및 결과 정렬을 추가로 지원합니다.

2. 그렙텍스트
   기능: 정규식, 컨텍스트 라인 및 보다 안정적인 쿼리 추출을 추가로 지원합니다.

3. 로컬파일 읽기
   기능: 경로와 줄 번호로 파일 조각을 읽습니다.

4. 기호 요약 읽기
   기능: 클래스 이름, 메서드 이름 및 속성 이름으로 정의 근처의 조각을 찾습니다.

5.진단 받기
   목적: 최근 빌드/진단 오류 요약을 수집합니다.

6. GetGitDiff/GetGitHistory
   기능: 현재 변경 사항과 역사적 의미를 이해합니다.

이 레이어가 완료되면 사용자가 기대하는 프로세스에 더 가까워질 것입니다.

``텍스트
질문
-> 저렴한 검색
-> 후보 결과 압축
-> 대형 모델로 보내기
```
### 3단계: 실제 ReAct/Planner-Executor 폐쇄 루프로 진화

목표: 도구를 한꺼번에 선택하는 로컬 규칙 대신 모델이 "다음에 수행할 작업"을 먼저 결정하도록 합니다.

2레이어 모델로 변경하는 것이 좋습니다.

1.플래너
   다음 액션 출력만 담당하고 사용자에게 직접 답변하지는 않습니다.

2. 답변자
   도구가 충분히 관찰한 후 최종 답을 생성하는 역할을 담당합니다.

제안된 루프:

``텍스트
1..N 단계의 경우:
  플래너 -> 구조화된 작업 출력
  executor -> 작업 확인 및 실행
  관찰 -> 궤적에 추가
  작업 종료 == final_answer
```
최소한 다음을 지원하려면 구조화된 작업을 수행하는 것이 좋습니다.

- 검색_파일
- grep_text
-읽기_파일
-읽기_로그
-fetch_url
- 최종_답변

이렇게 하면 파일을 읽은 후에도 모델이 계속될 수 있습니다.

- 또한 특정 기호를 잡아야 합니다.
- 다른 파일을 열어야 합니다.
- 최신 빌드 오류도 살펴봐야 합니다.

이것이 실제로 ReAct에 접근하는 핵심입니다.

## 제안된 코드 수정 사항

### 1. CopilotAgentModels.cs

현재 상태: 'CopilotToolCall', 'CopilotToolObservation' 및 'CopilotAgentStepRecord'의 최소 궤적 모델이 추가되었습니다. 서비스 계층은 또한 각 도구 실행 라운드를 단계 기록으로 기록합니다. 그러나 독립적인 런타임 컨텍스트, 도구 스키마 및 실제 '도구 호출' 실행 계약은 없습니다.

다음을 추가하는 것이 좋습니다.

- CopilotAgentRuntimeContext
- CopilotToolArgument

### 2. ICopilotTool.cs

현재 상태: 실행 중인 링크에는 쿼리/경로/startLine/endLine을 전달하기 위한 최소한의 통합 도구 매개변수 개체가 도입되었습니다. 기존 구조화된 도구는 이 개체를 직접 사용했지만 'ICopilotTool.ExecuteAsync(request, ...)'는 여전히 요청 집계 개체에 의존하고 호환 가능한 접근자도 요청 모델에 남아 있습니다.

제안은 다음과 같이 발전합니다.

```csharp
공용 인터페이스 ICopilotTool
{
    문자열 이름 { get; }
    문자열 설명 { get; }
    ToolSchema 스키마 { 가져오기; }
    Task<CopilotToolResult> ExecuteAsync(CopilotToolCall 호출, CopilotAgentRuntimeContext 컨텍스트, CancellationToken cancelToken);
}
```
이러한 방식으로 ReadLocalFile은 모델에서 지정한 경로, startLine 및 endLine을 읽을 수 있습니다.

현재 상태: 최소한의 모델 기반 루프가 있지만 단계 기록이 기록되고 관찰이 최종 답변으로 전송되지만 아직 완전한 계획자-실행자는 아닙니다.

## "로컬 파일 읽기"에 대한 가장 실용적인 다음 단계 제안

스크린샷의 문제를 최대한 빨리 해결하는 것이 목표라면, ReAct 전체를 직접 진행하지 말고, 다음과 같은 순서로 진행하는 것이 좋습니다.1. ReadLocalFile을 "전체 파일 읽기"에서 "경로 + 줄 범위별로 조각 읽기"로 업그레이드합니다.
2. SearchFiles에 보다 안정적인 파일 이름/글로브 추출 및 정렬을 추가합니다.
3. GrepText에 일반 규칙, 컨텍스트 라인 및 더 엄격한 검색 예산을 추가합니다.
4. 기본적으로 작업공간에서 경로 직접 읽기만 허용합니다.
5. 작업 영역 밖의 경로가 1회 팝업으로 나타나거나 사용자의 승인을 먼저 요구하는 경우
6. 실행 프로세스 패널에 보다 구조화된 검색 요약 표시
7. 마지막으로 최소 플래너를 구조화된 매개변수와 다중 도구 의사 결정을 지원하는 완전한 Planner-Executor 루프로 업그레이드합니다.

이렇게 하면 이점이 가장 높고 위험은 가장 낮습니다.

## 권장되는 다음 버전 타겟

다음 버전의 목표를 다음과 같이 정의하는 것이 좋습니다.

> 에이전트가 읽기 전용 모드로 작업 공간에서 명시적 경로 파일을 안전하게 읽을 수 있도록 허용하고 응답하기 전에 자동으로 1~2개의 저렴한 검색 작업을 수행합니다.

이는 명확하고 검증 가능하며 위험을 통제할 수 있는 중간 단계입니다.

이 이정표를 완료한 후에는 완전한 ReAct를 향해 계속해서 발전하는 것이 더욱 안정적일 것입니다.

## 요약

현재 에이전트에는 이미 올바른 시작점이 있습니다.

- 독립적인 도구 추상화가 있습니다.
- 도구 레지스트리가 있습니다.
- 실행 프로세스 패널이 있습니다.
- 도구 결과가 포함된 압축 레이어

하지만 ReAct에서는 여전히 세 가지 핵심 계층이 떨어져 있습니다.

1. 구조화된 도구 호출
2. 더욱 강력한 다단계 계획자-실행자 폐쇄 루프
3. 코드 시나리오를 위한 저렴한 검색 도구 세트

ReAct에 가능한 한 빨리 계속해서 접근하고 싶다면 가장 좋은 다음 단계는 완전한 코딩 에이전트를 직접 사용하는 것이 아니라 계속해서 더 많은 진단 도구를 추가하고, 현재 구현된 명시적 도구-인수 인터페이스를 기반으로 플래너-실행자 폐쇄 루프의 품질을 개선하고, 코드 구조 검색 기능에 대한 저렴하면서도 더 나은 이해를 점진적으로 도입하는 것입니다.
