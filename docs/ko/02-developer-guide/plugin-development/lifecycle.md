## 소개

이 문서에서는 플러그인 검색, 로딩, 초기화, 런타임 관리 및 제거 프로세스를 포함하여 ColorVision 플러그인 시스템의 전체 수명 주기를 자세히 설명합니다. 오류 격리 메커니즘과 버전 호환성 전략도 다룹니다.

## 디렉토리
# 플러그인 수명주기

이 페이지에서는 코드에서 직접 확인할 수 있는 현재 웨어하우스의 플러그인 실행 경로를 설명합니다. "독립 플러그인 호스트 + 비동기 라이프사이클 인터페이스" 설명의 이전 버전은 더 이상 사용되지 않습니다.

## 시작부터 가용성까지의 일반적인 프로세스

``인어
흐름도 TD
    A[주 프로그램 시작] --> B[스캔 플러그인 디렉토리]
    B --> C[manifest.json 읽기]
    C --> D [플러그인 목록 캐시 구성]
    D --> E[.deps.json 및 ColorVision 종속성 버전 확인]
    E --> F[플러그인 DLL 로드]
    F --> G [로드된 어셈블리 새로 고침]
    G --> H[플랫폼 스캔 확장 유형]
    H --> I[메뉴/설정/도구 입구가 보입니다.]
```
## 1. 플러그인 검색

`PluginLoader.LoadPlugins()`는 실행 디렉터리에서 `Plugins/`를 검색합니다. 각 하위 디렉터리는 후보 플러그인 디렉터리로 간주됩니다.

플랫폼은 다음을 찾고 있습니다.

-`manifest.json`
- 플러그인 마스터 DLL
- 선택사항 `.deps.json`

디렉터리에 `manifest.json`이 있는 경우 플랫폼은 플러그인 ID, 이름, 설명, DLL 경로 및 기타 정보를 읽고 이 정보를 내부 구성 캐시에 동기화합니다.

디렉터리에 매니페스트가 없는 경우에도 플랫폼은 "디렉터리 이름과 동일한 이름을 가진 DLL" 방법으로 이를 로드하려고 시도하지만 이는 호환성 동작일 뿐이며 공식적인 전달 방법으로는 권장되지 않습니다.

## 2. 플러그인 목록 정리 및 동기화

스캔 시작 시 플랫폼은 먼저 구성에 기록되었지만 디스크에 더 이상 존재하지 않는 플러그인 ID를 캐시에서 제거합니다. 즉, 플러그인 디렉토리의 존재는 플랫폼의 알려진 플러그인 목록에 역 영향을 미칩니다.

이것이 플러그인 디렉토리를 삭제한 후 다음에 플러그인을 시작할 때 관리 목록에서 해당 플러그인이 사라지는 이유입니다.

## 3. 종속성 확인

플러그인 디렉터리에 `.deps.json`이 있는 경우 플랫폼은 종속성을 읽고 `ColorVision.*` 관련 어셈블리를 확인하는 데 중점을 둡니다.

- 대상 DLL이 기본 프로그램 디렉터리에 존재하는지 여부
- 실제 버전이 플러그인에서 선언한 최소 버전을 충족하는지 여부

버전이 만족스럽지 않으면 플러그인 로드가 중지되고 로그 또는 프롬프트 메시지가 표시됩니다.

## 4. 어셈블리 로드

매니페스트 및 종속성 검사가 통과되면 플랫폼은 다음을 수행합니다.

1. 플러그인의 기본 DLL에 대한 실제 경로를 계산합니다.
2. 'Assembly.LoadFrom(...)'을 사용하여 어셈블리를 로드합니다.
3. 어셈블리 이름, 버전, 경로, 빌드 시간 및 기타 정보를 기록합니다.
4. 모든 플러그인이 로드된 후 어셈블리 목록을 새로 고칩니다.

현재 코드 경로에서 플러그인 로딩은 각 플러그인에 대해 독립적인 호스트나 재활용 가능한 로딩 컨텍스트를 설정하는 것이 아니라 "기본 프로세스에 어셈블리를 추가하고 후속 유형 검색에 참여하는 것"입니다.

## 5. 확장 지점이 적용됩니다.

DLL이 로드된 후 플랫폼은 로드된 어셈블리에서 확장 유형을 계속 검색합니다. 일반적인 결과는 다음과 같습니다.

- 메뉴 항목 제공업체를 찾았습니다.
- 설정 페이지 또는 구성 항목 공급자가 검색되었습니다.
- 상태바, 도구창, 기타 확장 입구가 발견됩니다.

따라서 플러그인이 "사용 가능한 것으로 보이는지" 여부는 단순히 로드되었는지 여부가 아니라 어셈블리가 플랫폼에서 예상하는 공급자 인터페이스를 구현하는지 여부에 따라 결정되는 경우가 많습니다.

## 6. 업데이트 및 관리

플러그인 정보가 기록된 후 플랫폼은 캐시의 플러그인 정보를 기반으로 프롬프트를 관리하고 업데이트할 수 있습니다. 업데이트 로직 및 플러그인 시장 통합은 UI 레이어 플러그인 관련 모듈에 있지만 이전 스캔 및 로드 결과 세트를 기반으로 구축됩니다.

## 문제 발생 시 가장 먼저 확인해야 할 사항

### 플러그인 디렉터리가 존재하지만 전혀 인식되지 않습니다.

- `manifest.json`이 존재하고 파싱이 가능한지 확인하세요.
- `dllpath`가 올바른지 확인하세요.
- DLL이 실제로 플러그인 디렉터리에 복사되었는지 확인하세요.

### 플러그인이 인식되었지만 로드에 실패했습니다.

- `.deps.json`에서 `ColorVision.*`에 대한 버전 요구 사항을 확인하세요.
- 기본 프로그램 디렉터리에 필요한 종속 DLL이 있는지 확인하세요.
- 종속성 버전이 부족하거나 DLL 프롬프트가 누락되었는지 로그를 확인하세요.

### 플러그인이 로드되었으나 메뉴나 기능이 나타나지 않습니다.

- 해당 Provider 인터페이스가 구현되어 있는지 확인
- 입력 유형이 non-abstract, non-open generic, public no-parameter 구성 등 기본 요구 사항을 충족하는지 확인합니다.

## 설명

- 현재 문서에는 창고에서 직접 눈에 보이는 로드 경로만 설명되어 있습니다.
- 'PluginContext', 권한 시스템, 격리된 호스트, 제거 가능한 컨텍스트 등에 관한 이전 문서의 내용은 현재 기본 경로 구현의 기본 기반으로 사용할 수 없습니다.
