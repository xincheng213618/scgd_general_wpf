# 템플릿 모듈 분석

이 페이지는 현재 창고에 있는 템플릿 시스템의 현재 상태만 분석하며, "문서가 추가된 수", "최적화 계획이 계획된 수"와 같은 유효하지 않은 요약 내용은 더 이상 유지하지 않습니다.

## 먼저 이 모듈이 무엇인지 살펴보겠습니다.

`Engine/ColorVision.Engine/Templates/`는 단일 알고리즘 디렉토리가 아니라 동시에 이러한 유형의 책임을 맡는 코드 세트입니다:

- 템플릿 추상화 및 등록
- 템플릿 관리 및 편집창
- 다양한 비즈니스 도메인에 대한 템플릿 구현
- 프로세스 템플릿 관련 기능
- 템플릿 검색, 생성, 샘플 저장 등을 위한 보조 도구

따라서 순수한 모델 코드와 명확한 WPF 창 및 편집기 코드가 모두 있습니다. 읽으실 때 "오직 알고리즘 매개변수 정의"라고 오인하지 마십시오.

## 현재 가장 먼저 알아야 할 파일

### 코어 입구

- `ITemplate.cs`: 템플릿 추상화 기능의 주요 입구
- `ModelBase.cs`, `ParamModBase.cs`: 템플릿 모델 및 매개변수 모델의 기본 유형
- `TemplateContorl.cs`: 템플릿 등록 센터 및 로딩 입구

### 관리 및 편집 UI

- `TemplateManagerWindow.xaml(.cs)` : 템플릿 관리 창
- `TemplateEditorWindow.xaml(.cs)`: 템플릿 편집 창
- `TemplateCreate.xaml(.cs)` : 템플릿 생성 입구
- `TemplateSettingEdit.xaml(.cs)` : 템플릿 구성 편집 입구

### 검색 및 샘플

- `TemplateSearchProvider.cs`
- `TemplateSampleLibrary.cs`
- `TemplateSampleSaveWindow.xaml(.cs)`

"템플릿이 어떻게 검색되는지, 어떻게 열리는지, 어떻게 편집되는지"를 알고 싶다면 일반적으로 알고리즘 하위 디렉터리로 직접 들어가는 것보다 먼저 이러한 파일을 살펴보는 것이 더 효과적입니다.

## 현재 디렉토리를 나누는 방법

창고의 현재 상황으로 판단하면 이 디렉토리는 최소한 여러 범주로 나눌 수 있습니다.

### 1. 핵심 프레임워크 레이어

템플릿 추상화, 등록, 기본 모델 및 공용 UI를 담당하는 부분입니다.

일반적인 문서는 다음과 같습니다.

-`ITemplate.cs`
- `ModelBase.cs`
- `ParamModBase.cs`
- `TemplateContorl.cs`
-`TemplatesExtension.cs`

### 2. 프로세스 템플릿 레이어

`Flow/`는 프로세스 템플릿과 프로세스 편집 및 실행과 관련된 기능을 제공합니다. 일반 알고리즘 템플릿과 동일한 시스템에 있지만 사용 시나리오는 분명히 다릅니다.

### 3. 비즈니스 템플릿 계열

비즈니스 템플릿 디렉토리의 여러 그룹은 현재 창고에서 계속해서 직접 볼 수 있습니다. 예를 들면 다음과 같습니다.

-`ARVR/`
-`POI/`
- `규정 준수/`
-`JND/`
- `매칭/`
-`FindLightArea/`
- `포커스포인트/`
-`이미지 자르기/`
- `LedCheck/`
- `LEDStripDetection/`
- `검증/`
-`데이터 로드/`

이러한 디렉토리는 통일된 규칙에 따라 완전히 구분되지 않습니다. 일부는 알고리즘 도메인별로 그룹화되고 일부는 처리 링크별로 그룹화되며 일부는 역사적 진화에서 남은 기능 패키지에 더 가깝습니다.

### 4. JSON 템플릿 계열

`Jsons/`는 가장 오해받는 영역 중 하나입니다. 일반적으로 기존 디렉터리 템플릿과 공존하면서 JSON 구성을 핵심으로 사용하여 일괄 템플릿 구현을 수행합니다.

이름은 비슷하지만 디렉터리가 다른 템플릿 구현을 본 경우 중복 코드라고 가정하지 마세요. 기록 버전, 구성 방법 또는 비즈니스 액세스 방법이 다를 가능성이 높습니다.

## 현재 런타임 중에 템플릿을 시스템에 어떻게 연결합니까?

템플릿 시스템에 대한 가장 중요한 현재 런타임 링크는 간단합니다.

1. 메인 프로그램과 플러그인이 먼저 어셈블리를 로드합니다.
2. 'TemplateContorl'은 데이터베이스 연결이 가능해진 후 현재 로드된 어셈블리를 검색합니다.
3. 'IITemplateLoad'를 구현하는 비추상 유형을 찾습니다.
4. 이러한 유형은 `Load()`를 통해 `ITemplateNames`에 템플릿을 등록합니다.
5. 등록된 템플릿을 이용하려면 템플릿 관리창, 편집창, 프로세스창, 비즈니스 기능을 이용하세요.

이 링크는 두 가지 실제 제약 조건을 보여줍니다.

- 템플릿은 정적 마스터 테이블에서 수동으로 선언되지 않습니다.
- 템플릿 표시 여부는 어셈블리 로딩 및 데이터베이스 가용성 상태에 영향을 받습니다.

## 이 모듈을 읽을 때 가장 흔히 저지르는 오해

### 오해 1: 순수 알고리즘 레이어로 취급

또한 창, 편집기, 검색, 샘플 저장, 템플릿 생성 및 프로세스 관련 UI가 포함되어 있습니다. 단순한 알고리즘 매개변수 정의 창고가 아닙니다.

### 오해 2: 모든 디렉토리가 동시에, 동일한 규칙에 따라 설계되었다고 생각

아니요. 현재 디렉토리 구조에는 명백한 진화의 흔적이 있으며 완전히 일관된 계층 모델을 자연스럽게 만족시킬 수는 없습니다.

### 오해 3: 템플릿이 누락된 경우 먼저 편집기를 확인하세요

등록 체인 초기에 많은 템플릿 문제가 발생합니다. 즉, 어셈블리가 로드되지 않거나, 데이터베이스가 연결되지 않거나, 'IITemplateLoad'가 실행되지 않거나, 템플릿 이름이 반복적으로 덮어쓰여집니다.

## 지금 계속 읽고 싶다면

권장 순서는 다음과 같습니다.

1. 템플릿이 어떻게 검색되고 등록되는지 이해하려면 먼저 'TemplateContorl.cs'를 살펴보세요.
2. 'ITemplate.cs', 'ModelBase.cs' 및 'ParamModBase.cs'를 다시 살펴보고 템플릿 개체 자체가 어떻게 생겼는지 이해하세요.
3. 그런 다음 'TemplateManagerWindow' 및 'TemplateEditorWindow'를 살펴보고 사용자가 템플릿을 작동하는 방법을 이해하세요.
4. 마지막으로 'ARVR/', 'POI/' 또는 'Flow/'와 같은 특정 비즈니스 디렉토리를 입력합니다.

이렇게 하면 처음부터 템플릿 하위 디렉터리를 파헤치는 것보다 전반적인 이해를 더 쉽게 할 수 있습니다.

## 이 페이지에서는 더 이상 아무것도 수행하지 않습니다.

이 페이지에서는 더 이상 다음 콘텐츠를 유지하지 않습니다.

- 문서완료 결과에 대한 통계
- 너무 구체적이지만 아직 구현되지 않은 리팩토링 로드맵
- 현재 창고 상태와 단절된 단일 계층 모델

향후 실제 리팩토링 제안이 있을 경우 "현재 상황 분석"에 혼합하는 대신 독립적인 디자인 페이지에서 별도로 시연해야 합니다.
