# 공정 장비 서비스

프로세스 장치 서비스는 "프로세스가 호출하지만 수동으로 자주 조작하지는 않을 것"이라는 입장으로 이해하는 것이 적합합니다. 현재 코드에서는 프로세스 시스템의 장치 입구에 가깝습니다. 복잡한 독립 운영 패널보다는 구성, 연결 및 프로세스에서 올바르게 참조하는 데 중점을 둡니다.

## 이 페이지는 언제 읽어야 할까요?

- 장치 목록에 처리 장치 서비스를 추가해야 합니다.
- 프로세스 노드가 이 유형의 장치를 참조하기를 원합니다.
- "장비를 제작했는데, 그 과정에서 이전할 수 없습니다." 등의 문제를 해결하고 있는 경우
- 이 서비스가 독립된 업무창구가 아닌 구성포털일 뿐인지 확인하고 싶으신 분

## 먼저 이해하는 방법

현재 구현에서 프로세스 장치 서비스는 주로 다음 두 가지를 제공합니다.

- 표준 장치 서비스 인스턴스로 시스템에서 관리됩니다.
- MQTT 서비스 계층에서 프로세스 실행 중에 참조할 수 있는 장치 개체로 인식됩니다.

즉, 내부 메시지 세부 사항을 자세히 연구하는 것보다 메시지가 올바르게 생성되고 저장되었으며 프로세스에서 참조될 수 있는지 확인하는 것이 더 중요합니다.

## 일반적인 사용 순서

1. 먼저 장치 목록에 프로세스 장치 서비스를 생성하고 저장합니다.
2. 장치 코드, 테마 또는 관련 연결 매개 변수가 올바르게 구성되었는지 확인하십시오.
3. [워크플로 개요](../workflow/README.md)로 돌아가거나 특정 프로세스 창에서 이 장치를 참조합니다.
4. 프로세스 실행 시 단순히 목록에 나타나는 것만 확인하기보다는 실제로 장치가 히트하는지 관찰해 보세요.

## 사용 시 특별히 주의해야 할 점은 무엇인가요?

### 장치가 존재한다고 해서 프로세스가 이미 해당 장치를 사용하고 있는 것은 아닙니다.

장치가 목록에 표시될 수 있다는 사실은 해당 장치가 리소스로 생성되었음을 의미할 뿐입니다. 실제로 중요한 것은 올바른 장치 인스턴스가 프로세스 노드에서 참조되는지 여부입니다.

### 구성을 먼저 확인한 다음 프로세스 노드를 확인하세요.

프로세스 중에 장치를 조정할 수 없는 경우 먼저 다음을 확인하십시오.

- 현재 디바이스 코드가 맞나요?
- 이 장치가 프로세스에서 참조됩니까?
- 해당 통신 구성이 일치하는지 여부

### 독립적인 수동 조치가 많다고 가정하지 마세요.

현재 구현을 보면 이 유형의 장치 페이지는 구성 및 액세스 계층에 더 중점을 둡니다. 문제를 해결할 때 복잡한 독립 제어 버튼을 먼저 찾지 말고 먼저 장치 구성 및 프로세스 참조 관계를 찾으십시오.

## FAQ

### 장치가 생성되었지만 프로세스에서 찾을 수 없습니다.

- 먼저 장치가 성공적으로 저장되었는지 확인하십시오.
- 프로세스 노드가 올바른 장치 객체를 참조하는지 다시 확인
- 해당 과정에서 코드나 이름으로 필터링하는 경우, 불일치하는 부분이 있는지 확인하세요.

### 프로세스가 실행될 때 장치가 응답하지 않습니다.

- 먼저 장치 서비스 자체의 연결 구성을 확인하십시오.
- 실행된 프로세스가 실제로 해당 노드인지 다시 확인
- 동시에 [로그 뷰어](../interface/log-viewer.md)를 결합하여 메시지 레이어가 이 장치에 닿지 않는지 확인합니다.

### 문제가 장치 계층에 있는지 프로세스 계층에 있는지 확실하지 않습니다.

- 먼저 질문을 장비 존재 여부와 프로세스 참조 여부의 두 부분으로 나눕니다.
- 기기 구성 자체가 불완전한 경우 먼저 기기를 수리하세요.
- 기기에 문제가 없다면 [프로세스 실행 및 디버깅](../workflow/execution.md)으로 돌아가서 범위를 좁혀보세요.
