# 일반 알고리즘 모듈

이 페이지에서는 더 이상 현재 저장소를 독립형 "범용 알고리즘 플랫폼"으로 설명하지 않습니다. 소스 코드의 현재 상태에 따라 공유 알고리즘 구성 요소에 대한 탐색 페이지로 더 적합합니다.

## 이제 이 페이지에서는 실제로 무엇을 다루어야 할까요?

여러 알고리즘, 템플릿 및 흐름 노드에서 반복적으로 재사용되는 현재 웨어하우스의 공통 구성 요소는 주로 다음 그룹에 집중되어 있습니다.

- ROI / 발광 영역 포지셔닝
- POI 지점 설정 및 관련 템플릿
- 매칭 템플릿 매칭
- JSON 형식의 자르기 또는 가장자리 찾기 템플릿

이러한 구성 요소의 공통점은 "순수한 알고리즘 핵심"이 아니지만 모두 다음과 같습니다.

- 템플릿 편집 입구
- 알고리즘 호스트 표시
- MQTT 명령 패키징
- 때로는 결과 처리 또는 흐름 액세스가 포함됨

따라서 이 페이지의 기능은 존재하지 않는 통합 프레임워크를 계속해서 추상화하기보다는 "어떤 공유 구성 요소 집합을 볼 수 있는지"에 더 가깝습니다.

## 현재 가장 우선시되어야 할 가치

### ROI

이제 클래식 ROI는 주로 `Templates/FindLightArea`에 속합니다.

-`템플릿Roi`
- `로이파람`
- `알고리즘Roi`
- `디스플레이로이`

현재는 글로벌 일반 ROI SDK가 아닌 발광 영역 포지셔닝 템플릿 체인을 나타냅니다.

또한 ROI 항목의 JSON 버전도 있습니다.

- `템플릿이미지ROI`
- `템플릿SFRFindROI`

자르기 또는 ARVR ROI를 살펴보면 이 두 가지가 기존 'TemplateRoi'보다 현상 유지에 더 가깝습니다.

### POI

POI는 현재 가장 일반적인 공유 프리미티브입니다. 최소한 다음 내용을 다룹니다.

- `템플릿 포이`
- `포이파람`
- '포이포인트'
- `알고리즘POI`
-`알고리즘빌드포이`
- 관련 템플릿 필터링, 수정, 교정 및 출력

그리고 JSON 알고리즘과 Flow 노드에서 계속 참조되므로 POI는 단일 알고리즘 페이지가 아니라 모듈 간 데이터 구조입니다.

### 매칭

매칭은 현재 완전하지만 상대적으로 간소화된 공유 체인입니다.

- `템플릿 일치`
- `MatchParam`
- `알고리즘 매칭`
- `디스플레이 매칭`
- `ViewHandleMatching`

'AlgorithmMatching'은 현재 다음을 수행합니다.

- 'TemplateMatch' 열기
- 선택적으로 `TemplatePoi`를 엽니다.
- `TemplateFile` 지정 허용
- `Event_MatchTemplate` 게시

따라서 매칭은 단순한 계산 기능이 아니라 완전한 템플릿이자 명령 호스트입니다.

## 이러한 공유 모듈을 시스템에 어떻게 연결합니까?

현재 구현에 따르면 일반적으로 동일한 작동 모드를 따릅니다.

1. 'TemplateEditorWindow' 또는 사용자 정의 편집 컨트롤을 통해 템플릿을 유지합니다.
2. 'DisplayAlgorithmBase' 파생 클래스를 통해 UI와 명령을 노출합니다.
3. 알고리즘 클래스에서 'CVTemplateParam' 및 기타 입력 매개변수를 조합합니다.
4. MQTT 이벤트를 통해 서비스 측에 요청을 보냅니다.
5. 결과 프로세서 또는 흐름 노드는 적절하게 계속 소비합니다.

이것이 바로 이러한 모듈을 단순히 "알고리즘 라이브러리"로 작성하는 것이 왜곡되는 이유입니다. 현재 구현에서는 UI, 템플릿 및 명령 호스트가 통합되어 있기 때문입니다.

## 필요에 따라 지금 소스코드를 읽고 싶다면

### 영역 선택 또는 자르기를 보고 싶습니다.

먼저 [ROI](./roi.md)를 읽어보세요.

### 포인트 세트 템플릿, 포인트 세트 구성 또는 POI 재사용을 보고 싶습니다.

먼저 [POI](./poi.md) 및 [POI 템플릿](../templates/poi-template.md)을 읽어보세요.

### 이미지 템플릿 일치를 보고 싶습니다.

`Engine/ColorVision.Engine/Templates/Matching/AlgorithmMatching.cs` 및 `TemplateMatch.cs`를 먼저 읽어보세요.

### 이러한 구성요소가 프로세스에 어떻게 배열되어 있는지 알고 싶습니다.

먼저 [프로세스 엔진](../templates/flow-engine.md) 및 `Templates/Flow/NodeConfigurator`를 읽어보세요.

## 현재 가장 흔히 저지르는 실수 중 일부는 다음과 같습니다.

### Universal은 독립 프레임워크와 동일하지 않습니다.

이러한 공유 모듈은 별도로 출시된 공개 SDK를 구성하지 않지만 'ColorVision.Engine/Templates' 아래에 배포되고 기본 프로그램 호스트에 의해 균일하게 호스팅됩니다.

### 공유 모듈은 순수하지 않습니다.

일반적으로 템플릿, UI, MQTT 메시지 및 결과 표시가 혼합되어 있습니다. 엄격한 3계층 아키텍처에 따라 계속해서 글을 쓴다면 현 상태와 어긋나기 쉽습니다.

### POI, ROI, 매칭 간에는 상호 참조가 있습니다.

예를 들어 Matching은 'TemplatePoi'를 계속 열 수 있으며 ARVR의 'SFR_FindROI'는 POI 템플릿을 요청합니다. 이러한 모듈은 서로 완전히 분리된 섬이 아닙니다.
