#ColorVision.FileIO

이 페이지에서는 현재 웨어하우스에서 실제로 사용할 수 있는 `ColorVision.FileIO` 모듈만 설명하며 "범용 파일 프레임워크 + JSON/YAML/일괄 처리 플랫폼" 이전 초안을 더 이상 유지하지 않습니다.

## 먼저 이 모듈이 무엇인지 살펴보겠습니다.

현재 소스 코드 상태에 따르면 `ColorVision.FileIO`는 일반화된 파일 처리 프레임워크가 아니라 ColorVision의 사용자 정의 이미지 파일 형식을 중심으로 구성된 전용 I/O 모듈입니다. 현재 가장 명확한 착륙 지점은 다음과 같습니다.

- `CVCIE` / `CVRAW` / `CVSRC` 클래스 파일을 인식하고 읽습니다.
- 파일 헤더, 버전, 노출, 크기, 채널 및 원시 데이터 영역을 구문 분석합니다.
- 채널 또는 파일 유형별로 로컬 ColorVision 이미지 파일을 엽니다.
- 'CVCIEFile'을 핵심 데이터 매체로 사용합니다.

따라서 이전 문서에 설명된 "일반 JSON/XML/YAML/이미지 일괄 처리 지원"보다 훨씬 더 좁고 현재의 실제 코드에 더 가깝습니다.

## 현재 가장 중요한 파일

- `엔진/ColorVision.FileIO/CVFileUtil.cs`
- `엔진/ColorVision.FileIO/CVCIEFile.cs`

적어도 현재 모듈의 판독 가능한 구현으로 판단하면 이 두 곳에서는 핵심 형식 식별, 헤더 구문 분석 및 데이터 캐리어 정의를 다루었습니다.

## 현재 처리 중인 형식은 무엇인가요?

현재 `CVCIEFile.cs`에 정의된 `CVType`에는 다음이 포함됩니다.

- `원시`
- `Src`
- `CIE`
- `교정`
- `티프`
- `데이트`

그러나 현재 'CVFileUtil'의 가장 집약적인 구현으로 볼 때 실제로 초점을 맞추고 있는 것은 일반 사무실이나 구성 파일 형식보다는 'CVCIE' / 'CVRAW' ColorVision 사용자 정의 이미지 파일 세트입니다.

## 현재 읽기 체인의 작동 방식

현재 `CVFileUtil`의 핵심 프로세스는 대략 다음과 같습니다.

1. 파일 헤더를 사용하여 `CVCIE` 시스템 파일인지 확인합니다.
2. 헤더와 버전을 읽어보세요.
3. 다양한 버전에 따라 파일 이름, 게인, 채널 번호, 노출, 크기 등과 같은 메타데이터를 구문 분석합니다.
4. 그런 다음 데이터 영역 오프셋에 따라 원래 바이트 블록을 읽습니다.
5. 이 정보를 'CVCIEFile'로 수렴합니다.

현재는 다음 두 가지를 모두 지원합니다.

- 파일 경로에서 읽기
- 바이트 배열에서 읽기

그리고 많은 세부 사항은 "범위를 벗어난, 예외, OOM 및 유효하지 않은 헤더 방지"와 관련되어 있습니다. 이는 일반적인 파일 서비스 계층이 아닌 일반적인 형식 구문 분석 코드입니다.

## `CVCIEFile`은 현재 어떤 역할을 하고 있나요?

`CVCIEFile`은 이제 모듈의 핵심 데이터 구조로 다음을 수행합니다.

- 파일 버전
- 파일 형식
- 순위 및 비트 깊이
- 채널 수
- 이득 및 노출 배열
- 소스 파일 이름
- 원시 데이터 바이트
- 파일 경로

또한 출시될 때 데이터 배열의 큰 블록을 적극적으로 지우는 'IDisposable' 자체를 구현합니다. 이는 현재 모듈이 가벼운 텍스트 구성보다는 대용량 이미지 데이터를 주로 다루고 있음을 다시 한 번 보여줍니다.

## 현재 어떤 실용적인 입학이 가능합니까?

현재 `CVFileUtil` 구현으로 판단하면 더 중요한 항목은 다음과 같습니다.

-`IsCIE파일(...)`
- `IsCVCIE파일(...)`
- `CIE파일 헤더 읽기(...)`
- `CIE파일 데이터 읽기(...)`
- `읽기(...)`
- `OpenLocalFileChannel(...)`
- `OpenLocalCV파일(...)`

이러한 입구는 기본적으로 두 가지 목표에 중점을 둡니다.

- 파일이 ColorVision 독점 형식인지 확인
- 독점 형식을 메모리 개체로 안전하게 구문 분석합니다.

## 현재 가장 흔히 저지르는 실수 중 일부는 다음과 같습니다.

### 일반화된 파일 시스템 플랫폼이 아닙니다.

현재 소스 코드는 이전 문서에서 공통된 `FileIOManager`, JSON 프로세서, YAML 프로세서, 배치 실행기 등의 전체 시스템을 구현하지 않습니다. 그러한 글쓰기 스타일을 계속 사용하면 존재하지 않는 추상화 계층이 사실로 바뀔 것입니다.

### 핵심 객체는 텍스트 구성 모델이 아닌 바이너리 이미지 캐리어입니다.

'CVCIEFile'은 현재 주로 이미지 메타데이터와 대규모 원시 바이트 배열을 전달합니다. 이는 구성, 압축 및 직렬화 서비스에 중점을 둔 이전 초안의 설명과 완전히 다릅니다.

### 버전 분기는 구현의 초점입니다

`ReadCIEFileHeader(...)` 다양한 버전의 분기 확인은 현재 복잡성의 실제 원인 중 하나입니다. 문서에 "사용자 정의 형식 파일 읽기"만 표시되어 있으면 이 세부 정보 레이어가 지워집니다.

### 보안은 주로 방어적 독서에 반영됩니다.

현재 코드는 소위 "핫 업데이트 비동기 파일 프레임워크"를 구현하는 대신 파일 길이, 오프셋, 배열 할당 및 예외를 집중적으로 확인합니다. 이를 이해해야만 코드가 헤더/데이터 구문 분석에 초점을 맞춘 이유를 이해할 수 있습니다.

## 추천읽기순서

1. `엔진/ColorVision.FileIO/CVCIEFile.cs`
2. `엔진/ColorVision.FileIO/CVFileUtil.cs`
