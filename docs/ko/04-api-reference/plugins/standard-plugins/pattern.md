# 패턴/그림카드 생성 기능

이 페이지에서는 더 이상 패턴을 "현재 웨어하우스에 완전히 존재하며 직접 개발할 수 있는 표준 플러그인"으로 설명하지 않습니다. 현재 소스 코드 상태에 따르면 이전 문서에 설명된 독립적인 '플러그인/패턴/' 플러그인 구현은 더 이상 웨어하우스에서 일치할 수 없습니다.

## 현재 결론

현재 소스 트리를 보면:

- `Conscope/`, `EventVWR/`, `Spectrum/`, `SystemMonitor/`, `WindowsServicePlugin/`과 같은 프로젝트는 실제로 `Plugins/` 디렉토리에 존재합니다.
- `Plugins/Pattern/` 디렉토리가 현재 웨어하우스에 존재하지 않으며 해당 `.csproj`, 창, `manifest.json` 또는 독립 플러그인 항목 구현이 없습니다.
- 이전 문서에 언급된 `IPattern`, `IPatternBase<T>`, `PatternManager`, 배치 생성 인터페이스 및 기타 유형은 현재 웨어하우스 소스 코드에 해당하는 실제 정의가 없습니다.

따라서 이 페이지는 이제 "창고 내 그림 카드 관련 기능의 나머지 배치 지점에 대한 설명"으로만 사용할 수 있으며 독립적인 플러그인 API 참조로 계속 사용할 수 없습니다.

## 실제로 창고에서 매칭이 가능한 관련 코드

독립 패턴 플러그인 프로젝트가 누락되었지만 PG/그림 카드 전환과 관련된 두 코드 라인은 여전히 저장소에 유지됩니다.

### `엔진/cvColorVision/PG.cs`

이 부분은 `cvCamera.dll`에 있는 PG 관련 기본 인터페이스의 P/Invoke 캡슐화입니다. 현재 명확하게 볼 수 있는 기능은 다음과 같습니다.

-`CM_InitPG`
-`CM_ConnectToPG`
-`CM_StartPG`
-`CM_StopPG`
-`CM_ReSetPG`
-`CM_SwitchUpPG`
-`CM_SwitchDownPG`
- `CM_SwitchFramePG`

이는 현재 창고에 있는 더 안정적인 "카드 관련 기능"이 자체 패턴 편집기를 갖춘 고급 플러그인 프로젝트보다는 PG 장치 제어임을 보여줍니다.

### `엔진/FlowEngineLib/PG/PGLoopNode.cs`

FlowEngine 측에는 루프 노드의 PG 매개변수를 명령 목록으로 변환한 다음 프로세스 실행 체인을 통해 실행하는 `PGLoopNode`도 있습니다.

- `시작` -> `CM_StartPG`
- `중지` -> `CM_StopPG`
- `재설정` -> `CM_ReSetPG`
- `위` / `아래` -> 그림 카드 전환
- `지정` -> `CM_SwitchFramePG`

이는 이전 문서에서처럼 로컬에서 11가지 패턴을 생성하고 PNG/JPEG/BMP를 내보내는 완전한 플러그인이라기보다는 "PG 장치를 제어하여 프로세스에서 그래픽 카드를 전환하는" 기능에 더 가깝습니다.

## 이전 문서를 더 이상 지금처럼 사용할 수 없는 이유

이전 페이지에는 소스 코드로 증명할 수 없는 여러 유형의 콘텐츠가 있습니다.

- 독립적인 '패턴' 플러그인 항목과 메뉴 항목이 존재한다고 주장합니다.
- `IPattern` / `IPatternBase<T>`와 같은 확장 인터페이스의 존재를 주장합니다.
- 11개 패턴, 로컬 템플릿 관리, 일괄 내보내기, 미리보기 최적화 등과 같은 전체 고급 기능 세트를 지원한다고 주장합니다.
- 현재 Repository에 없는 다수의 샘플 API와 확장코드를 제공합니다.

이러한 콘텐츠를 계속 유지하면 독자는 해당 구현을 현재 웨어하우스에서 직접 찾을 수 있다고 잘못 생각하게 될 것입니다.

## 현재의 보다 합리적인 이해 방법

지금 이 창고에서 "그림 카드" 능력을 쫓고 있다면 먼저 다음과 같이 이해해야 합니다.

1. 먼저 독립형 WPF 플러그인이 아닌 PG 장치 제어 체인의 일부로 생각하십시오.
2. 먼저 'cvColorVision/PG.cs'를 살펴보고 하위 계층에서 어떤 PG 명령을 보낼 수 있는지 확인하세요.
3. 'FlowEngineLib/PG/PGLoopNode.cs'를 다시 보면 프로세스에서 일괄 또는 주기로 PG를 전환하는 방법을 이해할 수 있습니다.
4. 그래도 상위 차트 생성 UI를 확인해야 하는 경우 해당 소스 코드가 창고 밖으로 이동되었는지, 다른 프로젝트에 위치했는지, 아니면 단지 문서 잔재인지 먼저 확인하시기 바랍니다.

## 현재 이러한 항목을 이해하는 방법

- 루트 디렉토리의 [Plugins/README.md](../../../../Plugins/README.md)는 이제 "현재 소스 코드에 실제로 존재하는 플러그인 디렉토리"와 과거 잔여 이름을 명확하게 구분합니다.
- API 레퍼런스의 이 페이지는 현재 웨어하우스의 패턴 관련 기능의 나머지 배치 지점에 대한 설명만 유지하며 더 이상 기존 표준 플러그인으로 작성하지 않습니다.
- 두 항목 간의 표현에 차이가 있는 경우 소스 코드 디렉터리와 런타임 로딩 동작이 우선되어야 합니다.
