## 简介

本文档详细描述 ColorVision 插件系统的完整生命周期，包括插件发现、装载、初始化、运行时管理、以及卸载过程。还涵盖了失败隔离机制和版本兼容性策略。

## 目录
# 插件生命周期

本页描述当前仓库里可以直接从代码确认的插件运行路径，不再使用旧版“独立插件宿主 + 异步生命周期接口”的叙述。

## 从启动到可用的大致过程

```mermaid
flowchart TD
    A[主程序启动] --> B[扫描 Plugins 目录]
    B --> C[读取 manifest.json]
    C --> D[整理插件清单缓存]
    D --> E[检查 .deps.json 和 ColorVision 依赖版本]
    E --> F[装载插件 DLL]
    F --> G[刷新已加载程序集]
    G --> H[平台扫描扩展类型]
    H --> I[菜单/设置/工具入口可见]
```

## 1. 发现插件

`PluginLoader.LoadPlugins()` 会扫描运行目录下的 `Plugins/`。每个子目录都被视为一个候选插件目录。

平台优先寻找：

- `manifest.json`
- 插件主 DLL
- 可选的 `.deps.json`

如果目录里存在 `manifest.json`，平台会读取插件 ID、名称、描述、DLL 路径等信息，并把这些信息同步到内部配置缓存。

如果目录里没有清单，平台仍会尝试按“目录名同名 DLL”方式装载，但这只是兼容行为，不建议作为正式交付方式。

## 2. 清理和同步插件列表

在扫描开始时，平台会先把配置里已经记录、但磁盘上已经不存在的插件 ID 从缓存中移除。也就是说，插件目录的存在状态会反向影响平台的已知插件列表。

这也是为什么删掉插件目录后，下一次启动时插件会从管理列表中消失。

## 3. 校验依赖

如果插件目录中带有一个 `.deps.json`，平台会读取依赖关系，并重点检查 `ColorVision.*` 相关程序集：

- 目标 DLL 是否存在于主程序目录
- 实际版本是否满足插件声明的最低版本

如果版本不满足，插件会停止装载，并给出日志或提示信息。

## 4. 装载程序集

当清单和依赖检查通过后，平台会：

1. 计算插件主 DLL 的实际路径。
2. 使用 `Assembly.LoadFrom(...)` 装载程序集。
3. 记录程序集名称、版本、路径、构建时间等信息。
4. 在所有插件装载完成后刷新程序集列表。

当前代码路径里，插件装载是“把程序集加入主进程并参与后续类型扫描”，而不是为每个插件建立独立宿主或可回收加载上下文。

## 5. 扩展点生效

DLL 被装载后，平台会在已加载程序集上继续扫描扩展类型。常见结果包括：

- 菜单项提供者被发现
- 设置页或配置项提供者被发现
- 状态栏、工具窗口或其他扩展入口被发现

因此，一个插件是否“看起来可用”，往往不取决于是否单纯被装载，而取决于程序集里有没有实现平台期望的 provider 接口。

## 6. 更新与管理

插件信息被记录后，平台可以基于缓存中的插件信息做管理和更新提示。更新逻辑和插件市场集成位于 UI 层插件相关模块中，但它们建立在前面这套扫描和装载结果之上。

## 发生问题时先查什么

### 插件目录存在，但完全没有被识别

- 检查 `manifest.json` 是否存在且可解析
- 检查 `dllpath` 是否正确
- 检查 DLL 是否真的复制到了插件目录中

### 插件被识别，但装载失败

- 检查 `.deps.json` 中对 `ColorVision.*` 的版本要求
- 检查主程序目录里是否存在需要的依赖 DLL
- 查看日志中的依赖版本不足或 DLL 缺失提示

### 插件已装载，但菜单或功能没出现

- 检查是否实现了对应的 provider 接口
- 检查入口类型是否满足非抽象、非开放泛型、公开无参构造等基本要求
