# Socket 通信模块优化路线

本文档面向 `UI/ColorVision.SocketProtocol`，用于把当前可用的 Socket 管理能力继续推进成更稳定、更容易排查、更适合长期运行的通信模块。

## 当前结构判断

`ColorVision.SocketProtocol` 当前已经形成完整闭环：

- `SocketInitializer` 按配置启动或停止服务
- `SocketManager` 承担 TCP 监听、客户端管理和请求分发
- `SocketJsonDispatcher` / `SocketTextDispatcher` 负责 JSON 与文本模式
- `SocketMessageManager` 将消息写入 SQLite
- `SocketManagerWindow` 和 `SocketStatusBarProvider` 提供桌面侧运维入口

这说明它已经不是“协议示例代码”，而是带 UI、持久化和运行时状态的通信子系统。优化时应优先保护现有行为，再逐步补齐协议边界和长期运行能力。

## 已完成的体验优化

管理窗口已经补齐一批低风险能力：

- 顶部服务状态：启用状态、监听地址、协议模式、连接状态、客户端数量
- 打开失败提示：最后一次启动错误会显示在窗口顶部和服务诊断页
- 消息过滤：按客户端、事件名、MsgID、响应码、内容过滤
- 方向过滤：全部、接收、发送
- 列表虚拟化：降低历史消息浏览时的 UI 压力
- 详情格式化：JSON 内容可格式化查看，也保留原文复制
- 自动滚动：新消息到达时可自动定位，也可关闭
- 重发兜底：优先匹配原客户端，匹配不到时使用当前选中的客户端
- 布局收敛：右侧用“消息详情 / 客户端 / 服务诊断”标签页承载辅助信息
- 快捷键：`Ctrl+F`、`Esc`、`F5`、`Ctrl+C`、`Delete`

## P0 稳定性路线

目标：先把服务生命周期和 TCP 收发边界收紧，减少现场“偶现卡住、端口占用、消息截断”的问题。

| 项目 | 当前风险 | 建议做法 | 验收标准 |
| --- | --- | --- | --- |
| 服务重复启动 | 多次调用 `StartServer()` 会创建新的后台任务 | 增加 `_serverTask`、`CancellationTokenSource` 和启动状态保护 | 连续点击启停不会产生多个监听循环 |
| 停止服务 | `AcceptTcpClient()` 依赖 `TcpListener.Stop()` 抛异常退出 | 显式取消、捕获取消路径，区分正常停止和错误停止 | 停止服务日志不再表现为异常错误 |
| TCP 边界 | 当前按一次 `Read` 当作一条消息 | 引入长度前缀、分隔符或可配置帧解析器 | 大消息和连续小消息不会粘包/半包 |
| 客户端集合 | `ObservableCollection<TcpClient>` 与后台线程交互需要保持 UI 线程约束 | 封装 Add/Remove，所有集合变更走 Dispatcher | 长时间连接/断开不抛跨线程异常 |
| 客户端清理 | 断开、停止、异常路径分散 | 增加统一 `CloseClient` 方法 | 客户端断开后列表、资源和日志一致 |

## P1 可观测性路线

目标：让窗口和日志能回答“现在谁连着、刚刚发生了什么、失败集中在哪”。

| 项目 | 建议做法 | 验收标准 |
| --- | --- | --- |
| 连接生命周期 | 增加连接建立、断开、异常原因记录 | 管理窗口可看到连接历史或日志可定位 |
| 消息导出 | 支持导出当前过滤结果为 JSON/CSV | 现场可以直接带走问题样本 |
| 错误统计 | 统计 JSON 解析失败、Handler 未找到、发送失败 | 状态栏或窗口能显示失败计数 |
| 处理耗时 | 记录 Handler 执行耗时 | 可以定位慢事件 |
| 数据库保留策略 | 按数量或时间清理历史 | 长期运行不会无限增长 |

## P2 协议化路线

目标：把外部设备对接从“约定俗成”推进到“可校验、可兼容、可迁移”。

| 项目 | 建议做法 | 验收标准 |
| --- | --- | --- |
| 消息模型 | 明确 `Version`、`MsgID`、`EventName`、`SerialNumber`、`Params` 的必填性 | 文档和运行时校验一致 |
| 错误码 | 定义统一错误码范围 | 客户端能根据 Code 做稳定处理 |
| Handler 元数据 | 为处理器增加描述、版本、示例参数 | 管理窗口或文档可枚举支持事件 |
| Schema | 为重点事件提供 JSON Schema | 对接方可提前校验 |
| 兼容策略 | 约定新老版本字段兼容方式 | 升级不破坏已有设备 |

## P3 性能与容量路线

目标：支持更长时间运行、更高消息频率和更大的消息历史。

| 项目 | 建议做法 | 验收标准 |
| --- | --- | --- |
| UI 分页 | 管理窗口按分页或增量加载历史 | 打开窗口不因历史库变大而卡顿 |
| 批量写库 | 高频消息时批量写入或后台队列写入 | 消息高峰下 UI 不被数据库写入拖慢 |
| 索引 | 为 `MessageTime`、`ClientEndPoint`、`EventName` 增加索引 | 查询和过滤历史数据可控 |
| 背压 | 对极端消息频率设置队列上限和降级策略 | 过载时可提示而不是拖垮进程 |
| 压测 | 增加本地 TCP 压测脚本 | 可复现容量上限和回归结果 |

## 建议实施顺序

1. 先做 P0 的服务生命周期保护和帧解析，避免协议层隐患继续放大。
2. 再补 P1 的导出、统计和保留策略，提升现场问题处理效率。
3. 然后推进 P2 的协议文档、错误码和 Handler 元数据。
4. 最后根据真实消息量决定 P3 的分页、批量写库和压测投入。
